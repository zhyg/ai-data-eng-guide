# 导学 05：凌晨两点出故障，谁来、怎么修、修完怎么确认？

## 一句话理解

应急手册要写成"照着命令一行行执行就能修好"的脚本，团队规范要定成"不用问人就知道该找谁"的规则。

## 用生活打比方

继续我们的物流仓库。前面几节课建好了货物编号、分区货架、补发流程、追溯系统，这节课解决最后一个问题：凌晨两点，分拣系统突然宕机，当天的货全卡在仓库出不去，怎么办？

差的仓库只有一句话："系统坏了联系运维。"好的仓库有一份操作手册，贴在墙上，新员工看了就能执行：第一步切换到备用分拣通道，第二步检查积压货物数量并通知下游分仓延迟到货，第三步联系运维恢复主系统，第四步系统恢复后核对积压货物是否全部发出。

同时，仓库还有一套日常管理规范：每个货架有挂牌标明负责人、每批货的编号格式统一、任何改动走签审流程。这些规则不是为了好看，而是为了出事时不互相推诿，平时不互相踩脚。

应急手册 + 协作规范，就是一座数据工厂从"能跑起来"到"稳稳地运行"的最后一步。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **Runbook（应急手册）** | 写给值班人看的操作指南，每一步都是可以复制粘贴的具体命令，不是笼统的"检查系统" |
| **Game Day（演练日）** | 定期模拟故障，让团队照 Runbook 跑一遍。跑不通的地方就是 Runbook 的 bug |
| **Postmortem（事故复盘）** | 事故结束后写的复盘报告，重点不是追责，而是找到根因并产出改进项 |
| **Branch Deployment（分支部署）** | 每个 PR 自动起一个隔离的数据沙盒，两个团队同时改同一张表也不会冲突 |
| **slim CI** | 只跑 PR 实际影响到的部分，不跑全量。CI 时间从 30 分钟降到 3 分钟 |
| **Ownership（负责制）** | 每份数据资产都挂一个负责团队，出问题告警直接路由到具体的人，不是群里 @everyone |
| **criticality（重要等级）** | 资产分 P1/P2/P3 三档，不同等级对应不同的响应速度和新鲜度要求 |

## 避坑提醒

- **以为 Runbook 是文档**：写着"检查上游是否正常"的 Runbook 等于没有。合格的 Runbook 每一步都是具体命令加期望输出，凌晨两点脑子不清醒也能照做。
- **以为写完 Runbook 就够了**：没演练过的 Runbook 在真正出事故时大概率跑不通，因为命令可能过时、环境可能变了。每个季度至少做一次 Game Day。
- **以为事故复盘是追责会**：好的复盘只对事不对人，目的是找到设计和流程上的问题，每一条改进项要有责任人和截止日期。
- **以为两个人不会同时改一张表**：只要团队超过三个人，变更冲突就会出现。用分支部署隔离变更，合到主分支才生效，是避免"你改完我才发现"的可靠方式。

## 一句话总结

> **资产流稳不稳，不看平时跑得多顺，看出事时能不能照手册在可接受的时间内修好。**
