# 导学 02：为什么补数不能"全部重来"？

## 一句话理解

分区让你能精确指着某一格说"就重跑这一份"，而不是把整个仓库翻一遍。

## 用生活打比方

继续我们的物流仓库。上节课建立了"每批货都有编号"的管理方式，这节课解决一个实际问题：某天发出的一批货出了质量问题，怎么办？

如果你的仓库没有分区管理，所有货物混在一起存放，出问题就得把整个仓库翻一遍，挨个检查、全部重新打包。但如果仓库按日期分了货架，每天的货物放在对应的格子里，你只需要走到"5 月 15 号"那个货架，把那一格拉出来处理，其他格子不用动。

进一步，如果你同时按日期和客户分格，那就能精确到"5 月 15 号发给 A 客户的那一批"，其他客户的完全不受影响。这就是复合分区的价值。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **分区（Partition）** | 把一份大数据按某个维度切成独立的小格子，每格独立管理、独立重跑 |
| **时间分区（Time Partition）** | 按日期或小时切格子，90% 的场景用这种就够了，比如每天一格 |
| **维度分区（Dimension Partition）** | 按业务维度切，比如按租户、按地区。适合需要隔离不同客户数据的场景 |
| **复合分区（Composite Partition）** | 时间和维度的组合，比如"日期 x 租户"，能精确到"5 月 15 号的 A 客户"这一格 |
| **event_time（事件时间）** | 数据本身记录的时间，比如工单创建时间。分区要按 event_time 切，不要按机器处理时间切 |
| **水位线（Watermark）** | 告诉系统"迟到多久的数据还能归入原来的分区"。超过水位线的迟到数据进隔离区 |
| **分区元数据** | 每个格子上贴的标签：物化时间、行数、上游快照 ID、状态。这是事故排查的关键线索 |

## 避坑提醒

- **以为分区只是为了查询快**：查询性能只是副产品，分区的核心作用是让你能按格子精确补数、按格子独立回滚、按格子单独验证。
- **以为数据量小就不用分区**：分区解决的是"出事故时能不能精确处理"的问题，和数据量大小没有直接关系。等出了事故再加分区，历史数据的迁移成本很高。
- **以为分区粒度越细越好**：按分钟分区会导致分区数爆炸，元数据查询变慢。高频数据用流处理，分区保持在小时或天级别。
- **以为用机器处理时间当分区键就行**：机器时间（wall-clock）和数据实际发生时间（event_time）可能差几天。用 wall-clock 分区，迟到数据会落到错误的格子里。

## 一句话总结

> **分区就是给数据仓库装上带编号的格子，出问题时只拉出那一格处理。**
