# 导学 03：补数为什么不能"再跑一次"？

## 一句话理解

补数的本质是回到出问题的那个时刻，用当时的条件重新生产，而不是用今天的代码跑昨天的数据。

## 用生活打比方

继续我们的物流仓库。上节课给仓库装好了分区货架，这节课要处理一个常见情况：某一批货出了问题，需要补发。

补发听起来简单，直接重新打包一批就行？没那么容易。5 月 15 号那批货用的是当时的包装规格和配货清单，但现在仓库的包装规格已经更新了，配货清单也调整过了。如果你用今天的标准重新打包，客户收到的东西和原来的不一样，可能引发更多问题。

正确的补发流程是：找到当时的包装规格和配货清单，按当时的标准重新打包，打包完检查质量，然后通知所有相关方"这批货已经补发了"。

还有一个容易忽略的问题：补发的时候，系统自动给客户发了一封"您的包裹已寄出"的通知邮件。如果补发失败要重试，重试一次就多发一封邮件，客户收到 5 封一模一样的通知。这就是"副作用没有做幂等"的后果。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **回填 / 补数（Backfill）** | 对历史某段时间的数据重新生产。要用当时的条件重建，不是简单地"再跑一次" |
| **幂等（Idempotency）** | 同一个操作跑一次和跑十次，结果完全一样。补数和重试的安全基础 |
| **写入层幂等** | 数据写入用 MERGE 或 UPSERT，同一条数据写多次不会出现重复行 |
| **副作用层幂等** | 发邮件、调 API 这些外部操作要带去重标记，避免补数时重复触发 |
| **下游层幂等** | 补完数要通知下游"上游数据变了"，下游才知道要重新消费 |
| **确定性合成主键（Deterministic Hash）** | 用分区键 + 业务键 + 版本号算出一个固定的 ID，跑多少次 ID 都一样，天然幂等 |
| **水位线延迟（Watermark Lateness）** | 设定一个时间窗口，比如 7 天，允许迟到的数据归入原来的分区。超过的进隔离区 |

## 避坑提醒

- **以为补数就是重新运行任务**：今天的代码和两周前的不一样，上游数据也变了。不锁定当时的条件就补，会用今天的逻辑改写历史事实。
- **以为写入不重复就叫幂等**：写入层幂等只是第一层。补数时如果多发了通知邮件、多调了外部 API，照样出事故。大多数补数事故不是出在数据上，而是出在副作用上。
- **以为补数前不需要评估**：每次补数应该先回答范围、资源、影响、策略、回滚这五个问题。跳过评估直接补，很容易补漏了或者补出新问题。
- **以为 UUID 就能做主键**：UUID v4 每次生成都不同，重跑一次就多一批新 ID，下游根本没法去重。生产场景应该用确定性哈希。

## 一句话总结

> **补数要锁定当时的代码和上游条件重新生产，写入、副作用、下游通知三层都要做到幂等。**
