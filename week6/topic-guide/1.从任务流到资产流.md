# 导学 01：为什么"跑完了"不等于"交付了"？

## 一句话理解

数据团队的核心交付物不是"跑成功的任务"，而是"可追溯、可回退、有人负责的数据产品"。

## 用生活打比方

这一周我们用"物流仓库管理发货"来串起所有概念。

假设你经营一个仓库，客户问"我订的那批货到了吗"：

- 旧做法：你只记录了"今天派了 5 趟车"，但说不清每趟车装了什么、送到哪、谁签收了。客户投诉少发了一箱，你翻遍记录也找不到那箱货的去向。
- 新做法：每批货都有编号，记着批次号、打包时间、送达地址、签收人。客户问"第 42 批货怎么回事"，30 秒就能查到。

前一种是"车跑了就行"，后一种是"每批货都可追踪"。数据工程里的"任务流到资产流"，就是从记录"车跑了几趟"转向管理"每批货的状态"。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **任务流（Task-based orchestration）** | 编排系统只管"这个任务跑没跑完"，类似只记发车次数，不管每批货的状态 |
| **资产流（Asset-based orchestration）** | 编排系统管的是"这份数据现在什么状态"，每份数据都有编号、版本、负责人 |
| **可寻址（Addressability）** | 每份数据都有稳定的 ID，可以像快递单号一样精确定位，比如 `support_ops.silver.ticket_fact@snap_42` |
| **可版本（Versioning）** | 每次生产数据都留一个不可改的快照，回退就是把指针切回去，不用重新跑 |
| **可责任（Ownership）** | 每份数据挂明确的负责团队和响应等级，出问题告警直接找到人 |
| **可观测（Observability）** | 数据的生产历史、血缘关系、质量指标全部可查，像产品仪表盘一样透明 |
| **Software-Defined Asset** | Dagster 里的核心概念，用代码定义"我要交付什么数据"，引擎自动处理依赖和执行 |

## 避坑提醒

- **以为换个调度工具就是资产化**：资产化的关键是换思维，从"我跑了什么"转向"我交付了什么"。用 Dagster 写的代码如果只关心任务是否跑完，本质上还是任务流。
- **以为 Airflow 必须马上扔掉**：已有的 Airflow 可以和 Dagster 共存，新项目用资产化的方式起步就行。一次迁移 200 个 DAG 几乎必然失败。
- **以为资产化只对大公司有用**：小团队第一次出事故补数时，就会发现缺少版本、缺少负责人、缺少物化记录的代价有多高。
- **以为加了元数据就叫可观测**：可观测要求物化历史、血缘、质量指标、变更日志全部可查。只记了 row_count 不叫可观测。

## 一句话总结

> **管好"每批货"比数"跑了几趟车"重要得多，数据工程也是一样。**
