# 导学 04：改了一个字段，怎么知道谁会受影响？

## 一句话理解

血缘是系统在运行时自动生成的依赖关系图，它能告诉你"改了这里，哪些下游会跟着变"。

## 用生活打比方

继续我们的物流仓库。前几节课解决了货物编号、分区管理、补发流程，这节课处理一个管理层面的问题：某批货物的包装标签印错了，你怎么知道这批货发到了哪些客户手里？

如果没有追溯系统，你只能挨个问各个发货组"你们用了这批标签吗"，再问下游分仓"你们把货转发给了谁"。开三小时会，还可能有遗漏。

如果有一套追溯系统，每次打包和发货时自动记录"这批货用了哪批标签、发给了谁、转到了哪个分仓"，你输入批次号，30 秒就能看到完整的影响链：哪些包裹、哪些分仓、哪些客户。

关键在于，这套追溯记录不是人工填表填出来的。人工填表半年后一定过期。它是每次打包和发货时系统自动记录的，所以永远跟实际操作一致。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **血缘（Lineage）** | 数据之间的依赖关系图。A 表生成了 B 表，B 表喂给了 AI 索引，这条链就是血缘 |
| **表级血缘（Table-level）** | 只记录"A 表到 B 表"的关系，粒度最粗，工具支持最广，但改一个字段说不清影响范围 |
| **字段级血缘（Column-level）** | 记录"A 表的 X 列影响了 B 表的 Y 列"，精细度高十倍，但复杂 SQL 和自定义函数容易断链 |
| **行级血缘（Row-level）** | 追溯到具体某一行数据的来源，成本很高，一般只在关键 AI 输出链路上做 |
| **OpenLineage** | 一个开放标准，定义了血缘事件的格式。各种引擎跑完任务后按这个格式发事件，汇总起来就是完整血缘图 |
| **Facet（扩展信息）** | 血缘事件上可以附加的额外信息，比如 schema 版本、行数统计、字段级映射关系 |
| **影响分析（Impact Analysis）** | 改一个字段之前，先用血缘图看下游哪些报表、AI 索引会受影响 |

## 避坑提醒

- **以为血缘是文档**：人工维护的血缘文档半年后就会和实际情况脱节。现代做法是让引擎在运行时自动发血缘事件，代码改了血缘自动更新。
- **以为表级血缘就够了**：表级血缘只能告诉你"B 表依赖 A 表"，但改了 A 表的某个字段，B 表到底受不受影响，表级血缘回答不了。核心域要做字段级。
- **以为字段级血缘能 100% 自动**：遇到窗口函数、自定义函数、JSON 嵌套字段，SQL 解析器经常断链。实际操作中，90% 自动加 10% 关键域手工补是当前的最佳实践。
- **以为血缘只管到数据表就行**：AI 时代的血缘要管到向量库、提示模板、模型版本。AI 回答错了，需要沿着血缘追回去看当时用的是哪批数据、哪个版本的提示词。

## 一句话总结

> **核心域做字段级血缘，AI 链路的血缘要延伸到向量库和提示模板，表级血缘在生产环境远远不够。**
