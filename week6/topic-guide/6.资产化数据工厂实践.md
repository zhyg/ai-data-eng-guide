# 导学 06：一座数据工厂从头到尾怎么串起来？

## 一句话理解

这节课把前五节的概念用 Dagster 串成一条完整的流水线，重点在理解编排思路，不在逐行读代码。

## 用生活打比方

继续我们的物流仓库。前五节课分别解决了货物编号、分区货架、补发流程、追溯系统、应急手册，这节课把整个仓库从进货到出货完整走一遍。

把所有环节串起来就是：先登记进货清单（manifest），检查清单是否合规（质量门禁），把原料入库（raw 层），加工成成品（silver 层），汇总出报表（KPI 层），每一步都留下记录（证据链），最后生成一份发货总结（delivery summary）。

这条主线之外，还有两个辅助岗位：一个人负责巡查仓库现状（lakehouse_state），另一个人负责排查哪些格子需要补货（backfill_plan）。

Dagster 的核心理念是"一切皆资产"。不管是数据采集、加工、质量检查还是证据报告，都被当作资产来管理。你只需要定义"我要交付什么"，引擎自动根据依赖关系决定执行顺序。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **definitions.py（注册层）** | 把所有资产、检查、任务、资源统一登记到 Dagster，是引擎的入口 |
| **assets.py（资产编排）** | 用代码定义每份资产和它们之间的依赖关系。引擎根据依赖自动生成执行顺序，不用手动编排 |
| **质量门禁（checks）** | 在数据流转的关键节点做检查，比如 manifest 是否合法、ticket ID 是否唯一、必填字段是否为空 |
| **证据链（evidence）** | 把每次运行的审计信息按固定格式写入 JSON 文件，先校验再落盘，保证记录本身不会出错 |
| **回填计划（backfill_plan）** | 默认用 dry-run 模式生成报告，告诉你哪些分区有缺口、建议怎么补，不实际执行 |
| **声明式编排** | 你只定义"我要什么资产"，Dagster 自动推断依赖、调度执行。和手动编排 DAG 的命令式方式相反 |
| **物化（Materialization）** | 在 Dagster UI 里触发资产的生产过程，能看到每个资产的执行状态、分区、血缘关系 |

## 避坑提醒

- **以为要先读懂每一行代码**：这节课的重点是编排思路和脉络。代码可以课后借助工具慢慢读，先搞清楚"为什么按这个顺序串"比"这行代码干了什么"更重要。
- **以为 Dagster 和 Airflow 只是换了个名字**：Airflow 是"你告诉引擎先跑 A 再跑 B"（命令式），Dagster 是"你定义 B 依赖 A 的输出，引擎自己算顺序"（声明式）。思维方式完全不同。
- **以为实验环境只能跑示例数据**：dry-run 模式下可以用正式数据测试，去掉固定分区参数就能尝试。但建议先在示例数据上跑通全链路。
- **以为质量检查可以后面再加**：门禁检查应该和资产一起定义。等上线后再补检查，中间已经有脏数据流到下游了。

## 一句话总结

> **数据工厂的核心是把采集、加工、检查、证据、补救全部当作资产来管理，引擎自动串联执行。**

本周建立的资产化范式是后续处理 PDF、音频、视频等非结构化数据的稳定上游。
