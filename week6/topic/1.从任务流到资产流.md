## KEY THESIS：资产化不是工具升级，是范式换挡

资产化不是工具升级，是数据团队的范式换挡——从「我跑了什么」转向「我交付了什么」。

Airflow 这套东西最开始是任务调度器，它只关心一件事：这个 DAG 跑没跑。但 AI 时代下游 RAG / Agent 真正在问的问题完全是另一组：

- 我现在拿到的 customer_dim，是哪一个版本？
- 最近一次更新是什么时候？
- 这张表谁负责？出问题该找谁？
- 要回退到上一个版本，我得花多久？

前面是脚本视角，后面是产品视角。资产化就一句话——**把数据当产品管理：可寻址、可版本、可责任、可观测。**

> 老司机说：Airflow 没错。错的是用 Task 这个抽象去服务一个需要回答「这是哪一版」的下游。

## 三种数据编排范式

| 维度 | 脚本式 ETL | 任务流（Airflow） | 资产流（Dagster/dbt） |
| :--- | :--- | :--- | :--- |
| 核心单位 | Python 脚本 | Task / Operator | Software-Defined Asset |
| 首要关心 | 脚本跑没跑 | DAG 跑通没跑通 | 资产现在是什么状态 |
| 故障定位 | 看日志，人肉排查 | 看 DAG 重试记录 | 看物化历史 + 血缘 |
| 影响分析 | 人肉评估，容易漏 | 看上下游 Task，粗 | Asset Graph 一键渲染 |
| 补数 | 写专用脚本 | 配 Backfill 但要人肉判断 | 声明式 Partition 自动补 |
| 给 AI 的承诺 | 没有 | 弱（只到 Task 层） | 强（每个分区有 snapshot） |

不是 Airflow 不好，是它的核心抽象错位了——「Task」服务不了「我要拿到第 42 版的 silver」这种诉求。

## 任务流的 3 个隐藏代价

上线半年才会爆：

| 代价 | 表现 | 根因 / 解药 |
| :--- | :--- | :--- |
| **补数（COST）** | 一次补数 ≈ 半周协调：改一个上游字段 → 人肉评估影响 → 写专用 backfill 脚本 → 通知 N 个下游各自补 → 彼此校对。某厂 2025 年一季度做了 11 次补数，平均每次占 1 个工程师 3 天 | 根因：无声明式分区 |
| **协作（COLLAB）** | 一次冲突 ≈ 一天扯皮：两个团队改同一张表，没有变更隔离，PR 合到主分支才发现冲突 | 根因：dev / prod 共用一套连接；解药：Branch Deployments + slim CI |
| **信任（TRUST）** | 一次问询 ≈ 半月信任流失：AI 团队问「这表是新的吗？」，没人给确定答案 → AI 团队开始自己抓数据 → 数据团队被绕过 → 治理崩塌 | 必须从根本上修 |

这 3 个代价在脚本式 / 任务流时代是「可以忍」，在 AI 数据时代是「忍不住」——AI 系统对「数据这一刻是什么样」的精度要求，和 BI 不在一个量级。

## 资产化的 4 条军规

少一条都不算「真资产化」：

| 军规 | 含义 | 工程体现 |
| :--- | :--- | :--- |
| **01 Addressability · 可寻址** | 每份资产都有稳定 ID（namespace + key + version），可以被像 URL 一样引用 | 不是「那个表」，是 `support_ops.silver.ticket_fact@snap_2026_05_18_042` |
| **02 Versioning · 可版本** | 每次物化产生一个不可变 snapshot。下游消费方可以锁到某个 snapshot_id | 回退就是切个指针，秒级完成——Iceberg 的 `rollback_to_snapshot` 是天然底座 |
| **03 Ownership · 可责任** | 每份资产挂明确的 owner / SLO / criticality | 告警不是「群里 @everyone」，是按 routing 自动找到主负责人。MTTR 从小时压到分钟 |
| **04 Observability · 可观测** | 物化历史 / 血缘 / 质量指标 / 变更日志全部可查 | 资产页面像一个产品仪表盘，而不是黑盒 ETL——「数据产品」的产品两个字，就指这个 |

## OmniSupport 项目里的资产定义

```python
# pipelines/assets/ticket_silver.py
from dagster import (
    asset, AssetIn, FreshnessPolicy,
    DailyPartitionsDefinition,
)

@asset(
    key_prefix=["support_ops", "silver"],
    name="ticket_fact",
    group_name="silver_support",
    partitions_def=DailyPartitionsDefinition(
        start_date="2026-01-01"),
    ins={"raw_ticket": AssetIn(
        key_prefix=["support_ops", "bronze"])},
    owners=["team:support-platform"],
    freshness_policy=FreshnessPolicy(
        maximum_lag_minutes=30),
    tags={
        "criticality": "P1",
        "contract": "tic.northstar.support_ticket.v3",
        "pii": "true",
    },
    metadata={
        "description": "工单事实表",
        "downstream_consumers": ["rag-faq", "support-kpi"],
    },
)
def ticket_fact(context, raw_ticket):
    df = clean_and_normalize(
        raw_ticket,
        partition_key=context.partition_key,
    )
    context.add_output_metadata({
        "row_count": len(df),
        "freshness_minutes": measure_lag(df),
        "contract_version": "v3",
        "snapshot_id": current_snapshot_id(),
    })
    return df
```

> 老司机说：这段代码不长，但每一行都对得上前面的 4 条军规。`key_prefix + name` → 稳定 ID（Addressability）；`partitions_def` → 每天独立物化，版本天然按分区切（Versioning 的根）；`owners + freshness_policy + tags` → Ownership 的元数据；`add_output_metadata` → Observability 的入口。一个原则：asset 函数内部别再藏「我跑哪个分区」「我用哪个版本」这种动态判断——通通让框架来推。

## 团队自检 5 题

超过 3 个答「不」，你团队还在脚本时代：

1. **Q1**：能不能 30 秒内回答「customer_dim 当前是哪一版？上一次更新几点？」——答「需要登服务器看日志」= 没有 Observability。
2. **Q2**：改一个核心字段前，你能不能先看到「下游哪 23 张表 + 哪 4 个 RAG 索引会受影响」？——答「开会评估」= 没有自动血缘。
3. **Q3**：上游昨天补数了，你能不能让所有用过昨天那批数据的下游「立刻知道」？——答「群里发个通知」= 没有 Asset 级事件传播。
4. **Q4**：凌晨 3 点告警，值班同事能不能照 Runbook 把 P1 故障在 30 分钟内压下来？——答「先打电话叫人」= 没有 Runbook 或 Runbook 只是文档摆设。
5. **Q5**：过去 90 天，你能不能给出「每一次补数花了多少机器钱、改了几个分区、被谁审过」的清单？——答「查不到」= 没有补数 Audit Trail。

## Airflow → Dagster 迁移路径

2026 老司机的渐进式答案：

| 阶段 | 策略 | 做法 |
| :--- | :--- | :--- |
| **Stage 1 · 共存** | 不动旧的 | Airflow 留着，Dagster 起一个新仓库，只接 1-2 个新的「数据产品」域。两套系统并行 4-8 周，谁也不破坏 |
| **Stage 2 · 反向桥接** | Dagster 触发 Airflow | Dagster 用 `ExternalAssetDefinitions` 把 Airflow DAG 当成「外部资产」抽进图里。血缘开始统一，数据图能合一 |
| **Stage 3 · 收编核心** | 关键域先搬家 | 把核心域的 silver / gold 层从 Airflow 搬到 Dagster，原 DAG 退化为「数据源 sensor」。配审批 + 灰度 |
| **Stage 4 · 退役旧的** | 只留 Edge 任务 | 只把那些「跑得稳、对资产化没诉求」的 Edge 任务留在 Airflow（报表推送、外部 webhook）。核心数据栈完全资产化 |

> 踩过的坑：不要「一次性把 200 个 DAG 迁过来」——95% 的团队会失败。一次只迁一个域，用 30-60-90 天节奏。

## 行业方向 2026

| 状态 | 代表 | 现状 |
| :--- | :--- | :--- |
| **已落地** | Dagster · dbt | Dagster Cloud Asset Graph 已是行业事实图谱；dbt 1.9+ Materialized Lake / Cumulative Materialization；OpenLineage 原生支持；已被 Anthropic / Stripe / Block 大规模使用。→ 新项目从这里起步 |
| **在追** | Snowflake · Databricks | Snowflake Dynamic Tables（声明式 + 自动重算）；Databricks Lakeflow（2025 重写的资产化引擎）；都在向「数据资产 = 一等公民」靠拢；平台耦合度高。→ 深度押注某一朵云直接用 |
| **在退** | Airflow · 自研脚本 | Airflow 3.0 引入 Asset（Dataset）感知，但 Task 抽象在核心 API 仍然是一等公民；自研脚本 + cron + on-call 几乎不再有新项目。→ 已上线的不必拆，但不要再起新 Airflow 仓库 |

2024 之后，新项目几乎没人再从 Airflow 起步——不是 Airflow 不能用，是它的核心抽象错位了。
