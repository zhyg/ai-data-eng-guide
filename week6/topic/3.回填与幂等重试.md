# 3. 回填与幂等重试

## 核心目标与背景

- **定调**：补数不是「再跑一次」，是「在历史时刻精确重建」。把「补数」从「写个脚本」升级成「工程化能力」。
- **主要任务**：掌握 4 阶段补数流程 + 3 层幂等 + 幂等 ID 策略 + 迟到数据处理 + 真实事故复盘 + 成本核算。
- **核心警示**：补数最危险的一句话——「我直接 rerun 一下就好。」没问场景、没看快照、没通知下游，事故 80% 是这样酿出来的。

## KEY THESIS：补数的本质是时间旅行

补数的本质是「时间旅行」——精确还原系统到某个历史时刻，从那里重新走。

很多人理解的补数是「再跑一次脚本」。这是个非常危险的想法。因为「再跑一次」用的是今天的代码 + 今天的上游 + 今天的依赖，而你要补的可能是 2 周前的数据。2 周前的代码不一样，上游数据状态不一样，甚至外部 API 返回都不一样——硬补一次，你会用「今天的逻辑」改写「2 周前的事实」，问题更大。

**正确的补数 = 精确回到那个历史时刻，用那时的代码版本 + 那时的上游 snapshot + 重新物化。** 这就叫——可重建（reproducibility）。

## 4 种典型补数场景

每种处理逻辑都不一样：

| 场景 | 触发原因 | 关键判断 | 回退策略 |
| :--- | :--- | :--- | :--- |
| **事故补数** | 上游故障 / 数据错乱 | 原数据是否还可信？ | 隔离原区 + 全量重跑 |
| **字段新增补数** | 新增计算列 | 是否影响历史含义？ | 只增量补新字段 |
| **业务变更补数** | KPI 口径调整 | 需不需要追溯回历史？ | 协调下游 + 双版本并行 |
| **上游修复补数** | 上游补完通知 | 当时下游用的是错的吗？ | 精确补 + 通知所有消费方 |

**补数前必须先问：这是哪种场景？回答错了，补数就会制造更大的问题。**

## 幂等性的 3 个层次

少一层都会在补数时炸：

### WRITE · 写入层——数据写入要幂等

- INSERT 改 MERGE / UPSERT
- partition_key + business_key
- 同分区多次写入结果一致
- 现代解药：
  - Iceberg snapshot 天然幂等
  - Delta MERGE + WHEN MATCHED
  - dbt incremental + unique_key
- **没有这一层，补数 = 制造重复**

### SIDE-EFFECT · 副作用层——外部操作要幂等

- 发邮件 / 推送通知
- 调外部 API
- 更新缓存 / 触发下游任务
- 关键武器：
  - idempotency_key 头
  - at-least-once + 去重表
- **最容易漏：补数时把同一封「账单已到账」邮件发 7 次**

### DOWNSTREAM · 下游层——消费方要感知

- 下游已经消费过旧数据
- 是否需要让下游一起重跑
- 消费方锁 snapshot_id
- 机制：
  - Watermark Event 通知
  - Asset Reconciliation Sensor
  - Dagster Auto-Materialize

**3 层只要漏一层，补数都会出事——99% 的补数事故，不是写入层的错，是副作用层或下游层没考虑。**（Week 14 治理周会展开）

## 幂等 ID 的 3 种策略

选错了，重试会制造重复：

| 策略 | 做法 | 优点 | 缺点 | 适合 |
| :--- | :--- | :--- | :--- | :--- |
| **S1 自然主键（Business Key）** | 用业务字段当主键，如 ticket_id / order_no | 天然幂等 | 上游改键就崩、跨域容易冲突 | 源头明确、业务键稳定的场景 |
| **S2 合成主键（UUID）** | 用 UUID v4 / ULID 现场生成 | 永不冲突 | 每次重跑 ID 都不一样，完全不幂等 | append-only 事件流（只入不改） |
| **S3 确定性合成主键（Deterministic Hash）** | 用 hash(partition_key + business_key + version) | 幂等 + 防冲突 | 实现稍复杂、改 schema 时 hash 要做版本化 | **大多数生产场景的标配选择** |

> 看到一个表 ID 是 uuid_v4()，先警惕——除非这是纯事件流，否则补数一定会重复；问设计者：你们怎么去重？如果答不上来，这就是隐患。

## 迟到数据：90% 的补数事故根源

| 环节 | 内容 |
| :--- | :--- |
| **PROBLEM · 现象** | 上游 API 5 月 18 日的数据，5 月 22 日才到达。如果你 5 月 18 日已经 materialize 了那一天的分区，这批数据要算今天的还是昨天的？ |
| **SIGNAL · 水位线** | 声明 watermark_lateness = 7d：「event_time 在 7 天内的迟到数据，归入它原本所属的分区」。超过 7 天 → 进 quarantine |
| **STRATEGY · 落分区** | 迟到数据到达 → 触发那一天的分区重物化（不是新建，是 overwrite）。Dagster Auto-Materialize 或 dbt incremental + filter 都能实现 |
| **GUARD · 阈值** | 迟到数据占当日总量 > 5% → 报警。说明上游有系统性延迟，要从根上修，不只是补数据 |

**水位线不是「写一行 watermark = 7d」就完了——下游所有 SLO 公式都要把 7 天延迟算进去，产品和业务那边必须沟通到。**

## 一次合格的补数：4 个阶段

每个阶段都有产物（Artifact）：

| 阶段 | 内容 | 产物 |
| :--- | :--- | :--- |
| **ASSESS · 评估** | 回答 5 个问题：范围、资源、影响、策略、回滚 | 补数评估单 PR |
| **APPROVE · 审批** | 主负责人 + 受影响下游主负责人共同点头 | PR 上的 approve + 通知群周知 |
| **EXECUTE · 执行** | 声明式 backfill，自动按分区并行，失败隔离 + 重试 | backfill_run_id + 物化历史 |
| **VERIFY · 验证** | 资产质量检查 + 下游消费方对账 | 验证报告 + Watermark Event |

**工程团队最容易跳过的是「评估」和「验证」——但这两步出问题，代价远大于补数本身。**

## Dagster Backfill 实战

```bash
# 1) CLI 方式：点选范围，声明原因和工单号
dagster job backfill \
  --job ticket_silver_job \
  --partition "2026-05-01:2026-05-07" \
  --tenant northstar_workspace,northstar_edge \
  --tags "reason=upstream_fix,\
          ticket=INC-2026-0512,\
          approver=zengdan@,\
          risk_level=P1"
```

```python
# 2) Python 方式：可编程触发（适合自动化补数）
from dagster import materialize, MultiPartitionKey

result = materialize(
    assets=[ticket_fact_by_tenant],
    partition_key=MultiPartitionKey({
        "date":   "2026-05-15",
        "tenant": "northstar_edge",
    }),
    tags={
        "backfill_run_id": "bf-20260518-001",
        "approver": "team:support-platform",
        "reason":   "upstream_fix",
        "source_snapshot_at": "2026-05-22T03:12:00Z",
    },
)

# Dagster 自动做的事：
# · 展开分区清单（笛卡尔积、去重已成功）
# · 按 max_concurrent 控制并发（别压垮上游）
# · 失败隔离 + 重试（只重跑失败 cell）
# · 上下游传播（下游自动感知，触发 reconcile）
# · 物化历史：materialized_at / snapshot_id / row_count
```

> 老司机说：`--partition` 用区间，Dagster 自动展开，你不用写 for-loop。`--tags` 必填 reason / ticket / approver，审计能追到人——少一个 tag PR 不该过。`source_snapshot_at` 是「时间旅行」的关键：告诉系统按那个时刻的上游 snapshot 跑。编程触发用于自动化（比如检测到上游补数完成，自动起下游 backfill）——这是从「人工」走向「半自动 Runbook」的桥。

## 补数评估单 5 问

这一页直接用作 PR 模板：

| 评估项 | 具体问题 | 不评估的后果 |
| :--- | :--- | :--- |
| **范围** | 哪些分区？哪些 tenant？多大批？ | 补漏了 / 补多了 / 影响别的租户 |
| **资源** | 需要多少 CPU / 多少 GPU / 多少时间？ | 挤占在线服务资源，引发二次事故 |
| **影响** | 哪些下游已经消费过这批旧数据？ | 下游继续用错数据生产新错 |
| **策略** | reject + replay 还是增量修补？ | 选错策略 = 修不好，反复返工 |
| **回滚** | 失败了怎么回到补数前的状态？ | 补数失败 = 数据进半成品状态 |

**这张评估单不是流程文档，是 PR 模板——每次发起 backfill PR 必须填，reviewer 看不到这张表，直接 close。**

## WAR ROOM：一次真实事故

补 7 天数据，3 个小时和 3 个深坑：

| 时间 | 事件 | 详情 |
| :--- | :--- | :--- |
| T+0 | 上游 API 返回 schema 漂移 | 2026-05-22 09:15 · contract check 报警，17 个分区进 quarantine |
| T+30m | 评估 → 影响 7 天 × 3 tenant | 21 个 cell，下游 14 个消费方，5 个已发出报表 |
| T+1h | 审批 → 主负责人 + 下游 owners 同意 | PR + 群周知 + Slack 工作流通知 7 个 RAG 索引 |
| T+1h30 | 执行 → Dagster Backfill 起 9 路并发 | reject + replay，iceberg.rollback_to_snapshot 兜底 |
| T+2h45 | 验证 → 21 个 cell 全部 SUCCESS | 对账 query 通过，下游 KPI 偏差 < 0.3% |
| T+3h | 通知下游 + 关 incident | 发 Watermark Event，RAG 索引自动重建 |

**事后复盘：3 个差点搞砸的深坑**

| 坑 | 问题 | FIX |
| :--- | :--- | :--- |
| **坑 1 副作用没幂等** | "账单已修复"邮件给同一个客户发了 5 次 | 补数前关掉所有外发副作用（toggle） |
| **坑 2 RAG 索引滞后** | 索引还在用旧 chunk，客服回答仍然错 | 发 Watermark Event，RAG 自动 reindex |
| **坑 3 下游报表对齐** | 5 月业绩报表已经出，补数后差了 0.3% | 出补丁报表 + 通知接收方更新 |

## 补数的成本账

别让一次"小补"变成季度账单的大坑（对照案例 · 3 tenant × 7 天）：

| 项 | 估算 | 说明 |
| :--- | :--- | :--- |
| Compute 直接消耗 | ~ ¥2,400 | 21 个 cell × 平均 8 分钟 × 中型 Spark cluster |
| 上游 API 重拉 | ~ ¥800 | 21 次重新拉取原始数据，部分供应商按调用计费 |
| 存储/快照膨胀 | ~ ¥300 | 新 snapshot + 老 snapshot 同时保留 30 天 |
| 下游 RAG 重索引 | ~ ¥1,500 | 14 个索引重建，embedding API 按 token 计费 |
| 工程人时 | ~ ¥3,200 | 2 个工程师 × 3 小时 + 1 位 reviewer |
| **总计** | **~ ¥8,200** | **一次「3 小时小补」实际成本 ≈ 一个工程师一周薪资** |

**老司机怎么把账压一半：**

- 评估单上必须写「能不能不补」——大量「补数」其实是「重新声明 contract + 加 reject 规则」就够
- 用 reject + replay 而不是 truncate + reload——只动错的格，不动好的
- 下游索引重建走「增量 reindex」而不是「全量 reindex」，RAG 成本能砍 70%
- 高频 backfill 集中到夜间低价时段（部分云有低峰价），账单能再降 20-30%
