# 2. 分区是资产化的原子单位

## 核心目标与背景

- **定调**：分区不是「为了查询快」，是让资产「在时间和维度上可寻址」——没有分区，补数没法精准、血缘只能停在表级、回放永远是全量。
- **主要任务**：理解三种分区类型、常见坑、粒度权衡、复合分区实战、Lakehouse 时代的分区现状、分区演进策略以及分区元数据。
- **核心问题**：2026-05-15 那一天的 customer_dim 出问题了，你能不能只重跑那一天？答不出来的团队，资产化都只做了一半。

## KEY THESIS：分区是时间维度上的可寻址性

分区不是性能优化——是「时间维度上的可寻址性」。

很多人把分区当查询优化，这只对一半。在资产化数据工厂里，分区真正的作用是：

- 让你能精确指着说「就重跑 2026-05-15 这一格」
- 就补 tenant_A 在 5 月的数据，别动 tenant_B
- 下游问「你用的哪一版」，你能给出 (date, tenant, snapshot_id)

没有分区，资产化是一片连续的湖；有了分区，资产化是一张精细的网——每一格独立物化、独立回滚、独立验证。

> 老司机说：一个常见误区：小公司想「我们数据少，不用分区」。等到第一次出事故，你会怀念那个早分好区的版本。

## 三种分区类型

每种解决不同的工程问题：

### TIME · 时间分区（90% 默认选择）

- DailyPartitions / HourlyPartitions
- 最常见、最稳定
- 解决：补数 / 回放 / 增量
- **关键**：用 event_time，不要用 wall-clock（挂钟时间）。event_time 是数据自己的时刻，wall-clock 是机器的时刻，迟到数据会让两者错位。

### DIMENSION · 维度分区（多租户 / 灰度）

- StaticPartitions(a / b / c)
- 按业务维度切
- 解决：多租户隔离 / 灰度
- **关键**：维度值必须稳定，不能频繁加新值。加新值意味着所有下游都要重建分区元数据。

### COMPOSITE · 复合分区（时间 × 维度）

- MultiPartitionsDefinition
- 时间 × 维度笛卡尔积
- 解决：精细化运营 + 定向补数
- **关键**：组合数别爆炸。5 tenant × 365 天 = 1825，OK；5 tenant × 24 时 × 365 = 43800，元数据查询变慢，要权衡。

**90% 的业务场景用时间分区就够。** 维度分区是「业务隔离需求」的产物，复合分区是「精细化运营」的产物，先有需求再上。

## 时间分区里最常踩的 4 个坑

每个都见过真实事故：

| 坑 | 工程师为什么会犯 | 后果 | 正确做法 |
| :--- | :--- | :--- | :--- |
| 用 wall-clock 当分区键 | 本能、改起来快 | 时区错乱 / 时钟漂移 / 数据漏掉 | 永远用 event_time + watermark |
| 粒度太粗（按月） | 想少建几个分区 | 一次补数重跑整月，资源浪费 | 默认按天，必要时按小时 |
| 粒度太细（按 1 分钟） | 想做实时 | 分区数爆炸、查询元数据慢 | 高频用流处理，分区按小时 |
| 没设 grace period | 不知道上游会迟到 | 迟到数据漏掉，下游算错 | 声明 watermark_lateness 缓冲 |

看着最简单的「时间分区」其实最容易翻车——选错粒度、忘了水位线，2 周后业务才会发现数据少了一截。

## 不同分区粒度的工程权衡

别问「应该选哪个」，先问「我下游要什么」：

| 粒度 | 适用场景 | 补数代价 | 元数据 | 推荐 SLO |
| :--- | :--- | :--- | :--- | :--- |
| 小时分区 | 高频更新的 ticket / event | 小（1 小时） | 高（8760/年） | P1 · 30 分新鲜度 |
| 天分区 | 90% 默认 | 中（1 天） | 中（365/年） | P1 · 4-12 小时新鲜度 |
| 周分区 | 聚合宽表 / 周报 | 高（7 天） | 低 | P2 · 24-48 小时新鲜度 |
| 月分区 | 财务月结 / 历史归档 | 极高（1 个月） | 极低 | P3 · 周度新鲜度 |

**默认从「天分区」起步**——只在两类情况偏离：高频需求往小走（小时），归档需求往大走（月）。先按天做，再调，这是 90% 团队的最优路径。

## 复合分区实战

让「时间 × tenant」的补数代价从 O(N) 降到 O(1)：

```python
# pipelines/assets/ticket_silver.py
from dagster import (
    asset, MultiPartitionsDefinition,
    DailyPartitionsDefinition,
    StaticPartitionsDefinition,
)

TENANTS = [
    "northstar_workspace",
    "northstar_edge",
    "northstar_studio",
]

PARTITIONS = MultiPartitionsDefinition({
    "date":   DailyPartitionsDefinition(
        start_date="2026-01-01"),
    "tenant": StaticPartitionsDefinition(TENANTS),
})

@asset(
    partitions_def=PARTITIONS,
    group_name="silver_support",
    tags={"criticality": "P1"},
)
def ticket_fact_by_tenant(context):
    keys = context.partition_key.keys_by_dimension
    date    = keys["date"]
    tenant  = keys["tenant"]
    df = load_raw_ticket(date=date, tenant=tenant)
    df = clean_and_normalize(df)
    context.add_output_metadata({
        "partition": f"{date}/{tenant}",
        "row_count": len(df),
        "snapshot_id": write_iceberg(df, date, tenant),
    })
    return df
```

> 老司机说：这段代码的工程价值，看 3 点就够了：一次补数可以精确到「2026-05-15 × northstar_edge」一个 cell——别的 tenant 不动，机器钱省一半；某个租户上游出问题，只重跑那一个 cell——爆炸半径从「全表」收到「一格」；snapshot_id 写在 metadata 里——下游可以锁定「我用的是哪一次物化」，这就是 versioning 的底。注意分区数：3 tenant × 365 天 ≈ 1100，问题不大；如果再加 hour 维度，就要做分层或者放弃 hour。

## Lakehouse 时代的分区现状（2026）

| 引擎 | 分区机制 | 亮点 | 坑 |
| :--- | :--- | :--- | :--- |
| **Iceberg V3** | Hidden Partitioning + Row Lineage | 分区列不暴露 / Schema 演进零成本 / 行级血缘原生 | 社区版自管 metadata 文件，小文件治理要自己来 |
| **Delta Lake 4.0** | Liquid Clustering | 动态调整，不用 partition by 写死 / VACUUM 自动 | 只在 Databricks 生态体验最好，跨厂支持弱 |
| **Apache Hudi 1.x** | COW / MOR + Clustering | 实时 upsert 友好 / Indexing 多 | 运维复杂度最高，小团队劝退 |
| **DuckLake** | SQL Catalog + Parquet | 极简、本地起服、对中小团队和授课场景友好 | 2026 仍偏新，生产用要看团队工具链 |

**选型口诀**：云原生且预算够 → Databricks + Delta；开源 + 多云 → Iceberg V3；教学 / PoC → DuckLake；Hudi 留给已经在用 Hudi 的团队。

## 分区演进

业务变了，分区怎么改才不伤筋动骨：

| 策略 | 场景 | 做法 |
| :--- | :--- | :--- |
| **S1 只加分区维度，不改老分区** | 业务从单租户变多租户 | 新建 MultiPartitionsDefinition 并把老数据全归到 default tenant，老分区一行不动 |
| **S2 粒度变细：先双轨，再废老** | 从天分区变小时分区 | 新表 ticket_fact_hourly 双写 4 周，确认新表稳定后下游切换，老表保留只读 |
| **S3 粒度变粗：用聚合，不重建** | 从小时变天 | 不要 truncate，起一个新的「daily summary」资产去聚合 hourly 数据，老 hourly 当 raw 留着 |
| **A1 硬规则** | 分区键一旦定下 | 永远不要原地改。Iceberg V3 的 partition transform 演进能减痛，但「改键」永远要走「新表 + 历史回填 + 切换」三步，没有更便宜的路 |

演进的成本一半在工程，一半在协作——下游有几十个消费方时，先做好沟通（谁切、何时切、回退条件），代码反而是简单的部分。

## 每个分区都应该带的元数据

这是资产化的「黑匣子」：

| 字段 | 作用 | 不带的后果 |
| :--- | :--- | :--- |
| partition_key | 唯一标识这一格 | 无法定位补数对象 |
| materialized_at | 本次物化时间戳 | 不知道用的是哪次物化结果 |
| source_snapshot_id | 上游所用的 snapshot | 上游变了下游不知道，出 bad case 复现不了 |
| row_count / size_bytes | 数据规模 | 补数前评估资源不准 |
| status | success / failed / quarantine | 下游不知道这一格能不能消费 |
| contract_ref | 所用契约版本 | 语义漂移无法回溯到契约源头 |

这套元数据 = Week 4 治理那一周「版本回滚」的全部依赖。少一个字段，事故复盘就少一条线索。
