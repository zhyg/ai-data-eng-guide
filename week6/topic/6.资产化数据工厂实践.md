# 6. 资产化数据工厂实践

## 核心目标与背景

- **定调**：本节是 Week06 的实验课与代码课，将前 5 周的结构化数据处理通过 Dagster 编排成一座完整的资产化数据工厂。重点不在逐行代码，而在理解整体编排思维和脉络。
- **主要任务**：理解 Dagster 的注册层、资产编排、质量门禁、回填计划、证据链等核心组件如何串联；通过实验验证端到端流程的正确性。
- **学习建议**：代码不是重点理解对象，重在掌握思维和脉络。课后可借助代码阅读器理顺代码结构，并自行尝试正式数据的 dry run。

## Dagster 与 Airflow 的核心区别

| 维度 | Dagster | Airflow |
| :--- | :--- | :--- |
| 核心抽象 | 基于资产（Asset） | 基于任务流（Task） |
| 定义方式 | 声明式——定义资产，引擎自动处理依赖和参数 | 命令式——手动分解 pipeline 并建立 DAG |
| DAG 构建 | 通过资产间的函数依赖关系自动形成 | 手动声明任务依赖 |
| 触发方式 | 支持人工命令式 + 事件流触发，本质是事件流系统 | 以调度为主 |

声明式理念的核心优势：你只需专注于定义资产，底层引擎会自动检测依赖、推断参数并执行——不需要你手动编排 DAG。

## 数据工厂整体架构

数据工厂是一个综合系统，通过 Dagster 底层组件将采集、湖仓、检查、契约、语义模型等串联起来。

### 输入依赖（来自前 5 周）

- 环境变量配置
- manifest 采集声明
- 契约定义（JSON Schema）
- 种子数据（seed）

### 核心编排流程

整个流程由 assets.py 串联，分为主线流程和辅助流程：

**主线流程**（资产间存在血缘依赖）：

1. seed_manifests —— 读取采集声明
2. manifest_gate —— 质量准入控制（确认有合法输入声明）
3. raw_ticket_events —— 使用 Ingest 脚本将数据采集到 raw 层
4. ticket_fact —— 从原始表转换为可交互的 silver 层
5. support_kpi_mart —— 语义聚合层
6. run_evidence_report —— 生成可审计的 JSON 证据文件
7. data_factory_delivery_summary —— 收口总结

**辅助流程**（观察型 / 补救型）：

- lakehouse_state —— 湖仓状态观察（不嵌入主流程）
- backfill_plan —— 回填计划（dry run 模式，提供补救措施）

### "一切皆资产"的统一抽象

Dagster 将所有内容都理解为资产——无论是数据采集、湖仓建模、质量检查还是证据报告，都以 asset 的形式定义和管理。这种抽象的价值在于：

- 能够串联全链路血缘关系
- 便于对整体进行统一控制
- 后续 RAG 中的检索/分块/嵌入、Agent 层面的提示模板，都可以纳入同一框架管理

## Dagster 核心组件说明

### definitions.py —— 注册层

- 将所有资产、检查、Jobs、资源统一注册
- 依赖 resources、partitions 等底层组件
- 是 Dagster 底层引擎编排的入口

### config.py —— 配置文件

- 包含环境测试、回填计划等多方面设置
- 分区设置尤为重要：默认分区配置用于处理特定日期的数据（如 2026-04-17）
- 运行时需要固定分区参数，用户可根据需求自行调整

### partitions.py —— 分区定义

- 定义分区的依赖项，包括起始时间等
- 默认起始时间会在 UI 界面中显示

### resources.py —— 资源配置

- 包含 MinIO 端点、访问 key 等配置
- 用于构建运行所需的外部资源连接

### assets.py —— 资产编排

- 通过 asset 注解和函数定义所有资产
- 资产间通过引用关系形成依赖，自动构建 DAG
- 资产命名规则：命名空间 + 层级关系 + 实际名称（如 `week06.silver.ticket_fact`）

### checks —— 质量门禁

包含两层检查机制：

| 检查项 | 验证内容 |
| :--- | :--- |
| manifest 一致性 | 至少存在一个结构化 manifest，且包含资产声明 |
| log count | 采集的统计总数大于 0 |
| ticket ID 唯一性 | 保证幂等性和写入无冲突 |
| 非空字段 | 必要字段不为空 |
| 分区有数据 | 指定分区必须有数据，避免无数据运行 |

这些检查既可以通过 pytest 运行，也可以在 Dagster UI 内部执行。

### evidence.py —— 证据链

- 将 raw 数据的审计信息契约化，变成可长期追踪的证据
- 基于 JSON 模板和契约定义结构化信息（使用 data class 实现）
- 流程：先通过 JSON Schema 校验 payload → 校验通过才写入 JSON 文件（防止坏证据落盘）

### backfill_plan —— 回填计划

- 默认模式为 dry-run，不执行实际重写操作
- 生成 report JSON 文件，可检查是否有错误发生
- 报告内容包括：输入 count、输出 count、gap 原因、action 建议

## 实验操作要点

### 环境准备

- 底层依赖已加载到 Docker 文件中，形成集成环境
- 启动 postgres 和 MinIO 作为基础服务
- Dagster UI 端口为 3000

### 验证步骤（由浅入深）

| 阶段 | 内容 | 目的 |
| :--- | :--- | :--- |
| 契约测试 | 运行 contract week06 测试用例 | 验证契约定义的正确性 |
| 注册验证 | 验证 Dagster 资源定义加载 | 确认所有资产已成功注册 |
| 回填验证 | 以模块化方式运行 backfill（dry-run 模式） | 验证回填计划的输出报告 |
| 单元测试 | 运行 checks 和 evidence 的测试 | 确认 5 个 check 通过 + evidence 格式合规 |
| 冒烟测试 | 对资产图 DAG 进行端到端串联测试 | 确保整体流程正确 |
| 回归测试 | 整合所有内容进行集成测试 | 验证定义 + 注册 + 回填 + 检查 + 证据链全部通过 |
| UI 物化 | 通过 Dagster UI 运行整个资产的物化过程 | 可视化验证血缘关系和执行顺序 |

### 回填报告解读

- input count 表示输入数据量
- output count 为 0 时，gap 原因可能是 output_lag
- 如果 expected_input_count = 0，说明分区选择错误或原始 seed 文件不存在

### Dagster UI 中可观察的内容

- 资产 Catalog：所有定义的资产列表
- 血缘图：资产间的依赖关系可视化
- Jobs：资产编排成的静态任务执行流程
- 物化记录：每次运行的分区、状态、时间

## 资产依赖关系总结

整个第六周的资产依赖关系框架：

1. 先编写 manifest 清单
2. 进行数据采集
3. 通过质量门禁
4. 再次采集进入 raw 层
5. 转换为 silver 层数据
6. 生成 evidence 证据
7. 形成 delivery summary

lakehouse_state 和 KPI 是观察型依赖（不嵌入主流程），backfill_plan 提供补救措施。
