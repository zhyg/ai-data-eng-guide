# 4. 全链路血缘

## 核心目标与背景

- **定调**：血缘不是好看的图，是数据系统的 X 光机——能不能用、用得好不好，决定了你的团队能跑多快。
- **主要任务**：理解血缘的 3 个粒度、5 大场景、OpenLineage 核心模型、AI-Native 血缘扩展、字段级血缘的现实坑、以及 Dagster + OpenLineage 集成与血缘后端选型。
- **核心场景**：产品经理跑过来：「我们改了客户分级字段，影响哪些报表？哪些 AI 的回答？」没有血缘的团队：开会 3 小时、问 5 个组、还可能漏掉。有血缘的团队：渲染一张 Asset Graph，30 秒给出完整影响范围。

## KEY THESIS：血缘是运行时自动生成的拓扑

血缘不是「文档」，是「运行时自动生成的拓扑」——文档会撒谎，血缘不会。

过去 20 年企业都有「数据字典」「血缘文档」——90% 的团队都失败了，因为人工维护的血缘永远赶不上代码变化。

现代数据栈对血缘的根本性突破：血缘不再是「人写的文档」，而是「编排引擎跑的时候自动产生的元数据」。代码改了，血缘自动更新——文档不会过期，因为它本身就不是文档。

**一个心智转换**——不要再问「我们的血缘文档是谁负责维护的？」正确的问题是「我们的血缘事件流是从哪个 emitter 发的？数据存到哪个 backend？」

> 老司机说：看到「血缘文档」这四个字就要警惕——再认真的人，半年后这份文档都会变成"看着像那么回事"的废纸。

## 血缘的 3 个粒度

每一层解决不同问题：

| 粒度 | 定位 | 内容 | 工具 | 局限 |
| :--- | :--- | :--- | :--- | :--- |
| **TABLE · 表级** | 入门档 | A.table → B.table，最容易做，工具支持最广。解决：粗粒度影响分析 | dbt-docs / DataHub | 改一个字段说不清哪些下游受影响；BI 时代够用，AI 时代不够 |
| **COLUMN · 字段级** | 生产档 | A.col1 → B.col2，工程难度大 10 倍。解决：精细影响分析 | OpenLineage SQL Facet / sqllineage / Marquez | 复杂 SQL（window / UDF）解析仍是难题 |
| **ROW · 行级** | 硬核档 | 某一行的源头，性能成本极高。解决：bad case 追溯 | Iceberg V3 Row Lineage / OpenLineage + audit | 全量行级开销过大，通常只在抽样 + 关键域上做 |

**2026 的及格线**：表级 100% + 字段级覆盖核心域 + 行级在关键 AI 输出链路上做。Iceberg V3 的 Row Lineage 让行级第一次有了规模化可能。

## 血缘的 5 大场景

每一个都直接决定 MTTR：

| 场景 | 说明 |
| :--- | :--- |
| **U1 影响分析（Impact Analysis）** | 上游改 schema 前，先渲染影响范围：哪些 asset / dashboard / RAG 索引会受影响。原来开会评估，现在按一下按钮 |
| **U2 故障定位（Root Cause）** | 下游 KPI 异常，沿血缘逆向追溯，找到第一个出问题的资产分区。原来 3 小时，现在 5 分钟 |
| **U3 合规审计（Compliance）** | 审计员问：「这个客户字段在哪 7 个系统里使用过？」血缘直接出报告。GDPR / CCPA + AI Act 都靠它 |
| **U4 查询优化（Query Optimization）** | 看哪些下游都在消费一张「巨表」——精准物化中间层，避免下游重复 join。这是给 FinOps 的礼物 |
| **U5 Bad Case 复现（Reproducibility）** | AI 回答错了——血缘 + snapshot ID 让你回到那一刻，看那时的数据、复现整个推理过程。这是 AI 治理的根 |

## OpenLineage 的 4 个核心对象

理解这 4 个，就理解了现代血缘：

| 对象 | 含义 | 主要字段 | 什么时候产生 |
| :--- | :--- | :--- | :--- |
| **Job** | 一段可重复执行的代码 | name / namespace / source | 代码定义时声明 |
| **Run** | 一次具体执行 | run_id / start / end / state | 每次运行时生成 |
| **Dataset** | 一份数据集（表/文件/流） | name / namespace / schema | 输入或输出时声明 |
| **Facet** | 附加元数据扩展 | 任意结构化扩展信息 | 按需附加到上面 3 类对象 |

这 4 个对象一组合，就能描述「某个时刻、某段代码用了哪些数据、产生了哪些数据」——这就是血缘的全部。

你不需要写它，Dagster 跑完自动发：

  "run": {
    "runId": "bf-20260522-001",
    "facets": {
      "parent": { "job": "backfill_ticket_silver" },
      "nominalTime": {
        "nominalStartTime": "2026-05-15T00:00:00Z"
      }
    }

  "job": {
    "namespace": "omnisupport.support_ops",

    "name": "raw_ticket",
    "facets": {
      "schema": { "fields": ["..."] },
      "version": { "datasetVersion": "snap_42" }

  }],
  "outputs": [{
    "namespace": "omnisupport.support_ops.silver",
    "name": "ticket_fact",

> 老司机说：inputs / outputs 自动带「上游用的哪个 snapshot」+「输出了什么」——这就是血缘。facets 是扩展点：schema、版本、统计、columnLineage 都挂这里——它会随生态长大。所有事件汇集到血缘后端（Marquez / DataHub / OpenMetadata）后，形成实时资产图。关键：这是机器产生的，不是人写的——所以它不会过期。「血缘文档」时代结束了，「血缘事件流」时代开始了。

## AI-Native 血缘

AI 时代，血缘要管到哪里？边界已经被推远了。

过去：血缘 = 「表 → 表 → 表」。AI 时代：血缘要纳入 vector store / prompt / model / eval set / agent decision——只有这样，bad case 复现才走得通，审计才答得上来。

完整链路：

```
数据源 Bronze Source（业务库/日志/API）
  → 结构化层 Silver / Gold（fact / dim 表 + 契约）
    → Doc Chunks（切分块 + chunk_id + 来源）
      → Vector Store（向量 + version + index）
        → Prompt Template（版本化 + variables）
          → AI Answer（推理记录 + 决策依据）
```

**AI-Native 血缘的 4 个工程动作：**

1. 每个 chunk 带 source_asset_id + source_snapshot_id —— bad case 复现的根
2. Vector store 的 collection 视为「资产」纳入 Asset Graph，每次重建都发 OpenLineage 事件
3. Prompt template 走 Git + version_tag，推理时记录到 Trace，血缘里能看到「这次回答用的是 prompt v17」
4. Eval set 也是资产—— 在血缘里能看到「v17 prompt 在 eval_set_v3 上跑分多少」

## 字段级血缘的现实坑

SQL Parser 撑不住的地方，你要自己来：

| 坑 | 问题 | Workaround |
| :--- | :--- | :--- |
| **T1 Window function 解析盲区** | sqllineage 这类静态解析器在 ROW_NUMBER() OVER(...) / LAG / LEAD 这种里几乎都会丢字段 | 核心域改用 dbt model 显式声明 columns，把字段血缘下沉到 model 元数据 |
| **T2 UDF 直接断链** | 一个 Python UDF / Spark Function 内部跑了什么，SQL parser 看不到 | UDF 显式声明 inputs / outputs 元数据，在 OpenLineage 事件里用 ExtractionFacet 补上 |
| **T3 JSON / 半结构化字段** | json_extract(payload, "$.user.id") 这种，parser 经常只能解到 payload，看不到内部路径 | 把 JSON 字段先拍平（structify）到独立列，血缘自然恢复 |
| **T4 跨引擎（SQL → Spark → ML）** | 一条管线跨 3 个引擎，事件标准不一致 → 血缘断链 | 统一 OpenLineage emitter，所有引擎都发同一种事件——这是 2026 年新项目的硬要求 |

**接受现实：字段级血缘不可能 100% 自动，90% 自动 + 10% 关键域手工补，是 2026 年的最佳实践。**

## Dagster + OpenLineage 集成

```python
# pipelines/lineage/setup.py
from dagster import Definitions, EnvVar
from dagster_openlineage import (
    openlineage_resource,
)

defs = Definitions(
    assets=[
        ticket_fact_by_tenant,
        ticket_kpi_mart,
        kb_serving_chunks,
        kb_vector_index,
    ],
    resources={
        "openlineage": openlineage_resource.configured({
            "transport": {
                "type": "http",
                "url":  EnvVar("OPENLINEAGE_URL"),
                "endpoint": "/api/v1/lineage",
            },
            "namespace": "omnisupport",
            "extractors": ["sql", "schema", "stats",
                           "column", "vector_store"],
        }),
    },
)

# 之后：所有 asset 物化时自动发血缘事件
# asset 函数本身不需要写任何额外代码
# DataHub UI 上能看到：
#   ticket_fact_by_tenant
#   ├── upstream: raw_ticket@snap_42
#   └── downstream:
#       support_kpi_mart, kb_vector_index
```

## 血缘后端选型 2026

| 产品 | 定位 | 适合 |
| :--- | :--- | :--- |
| **Marquez** | 开源最小可用 | 小团队 / PoC |
| **DataHub** | 血缘 + 目录 + 治理 | 中大型组织 |
| **OpenMetadata** | 统一元数据 + 治理 | 需要治理协作 |
| **Atlan** | 体验最现代 | 预算充足的商业团队 |
| **Unity Catalog** | Databricks 原生 | 深度押 Databricks |

**默认路径**：新项目 OpenLineage + DataHub；如果在 Databricks 生态 → Unity Catalog 优先。
