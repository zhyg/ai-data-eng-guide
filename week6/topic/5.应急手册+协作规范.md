# 5. 应急手册+协作规范

## 核心目标与背景

- **定调**：前 4 节建好了数据工厂的核心能力（Asset / Partition / Backfill / Lineage），这一节做两件事：把这些能力组装成一份能在凌晨两点照搬执行的 Runbook；把团队协作的规则定下来。
- **主要任务**：掌握 Runbook 的 5 个必备要素、6 大反模式、真实模板；建立命名 / Ownership / SLO 团队约定、变更隔离机制（Branch Deployments + slim CI）、Game Day 演练、Postmortem 模板。
- **目标状态**：不是「我会用 Dagster」，而是「我和我的团队能稳稳地运行一座数据工厂」。

## KEY THESIS：Runbook 是事故压力下也能稳定执行的脚本

Runbook 不是文档，是「事故压力下也能稳定执行的脚本」。

我见过最差的 Runbook 长这样——「检查上游数据是否正常。」什么叫正常？查哪些字段？阈值多少？凌晨两点没人看得懂。

好的 Runbook 长这样——「执行 `dagster job status ticket_silver_job --partition 2026-05-15`；如果 status 是 FAILED，跳到第 3 步。」

**Runbook 的核心不是「描述」，是「指令」——一个能在凌晨两点照搬执行的脚本。**

> 老司机说：判断一个团队是否真有运维能力，看 Runbook——看不到具体命令的 Runbook，等于没有。

## Runbook 失败的 6 大反模式

你的 Runbook 中招几个？

| 反模式 | 具体表现 | 后果 | 正确做法 |
| :--- | :--- | :--- | :--- |
| 抽象描述 | "检查上游是否正常" | 凌晨两点没人看得懂 | 给具体命令 + 期望输出 |
| 口口相传 | "问老 X 就行" | 老 X 离职就完蛋 | 强制每个故障写 Runbook |
| 只在事故后写 | 复盘时补 Runbook | 凭记忆不准，有遗漏 | 事前演练 + 事后修订 |
| 没人测试 | 写完没人跑过 | 真用时命令早废了 | 每季度 Game Day 演练 |
| 没有验证步骤 | 操作完不知道好没好 | 修了还在出问题 | 每个动作配验证命令 |
| 不连接监控 | 看不到当前状态 | 凭感觉判断 | Runbook 第一步看 Grafana |

**Runbook 不是写出来的，是「演练 + 反复修订」出来的——第一版永远是错的。**

## 一份合格 Runbook 的 5 个必备要素

缺一个就用不起来：

| 要素 | 说明 |
| :--- | :--- |
| **R1 Symptoms · 症状** | 怎么发现这是这类故障？告警是什么？面板长什么样？给出可观察的判定特征——截图 + 阈值，不是用文字描述 |
| **R2 Diagnosis · 排查** | 症状到根因的判断流程图——决策树形态。哪条分支查什么、跑哪个命令、期望输出是什么 |
| **R3 Action · 操作** | 修复操作的具体命令——每一行都能复制粘贴。不要写"重启服务"，要写 `systemctl restart dagster-daemon` |
| **R4 Verification · 验证** | 修完怎么确认好了？查哪些指标、跑哪些验证命令、期望多久内恢复正常。没有验证 = 修了等于没修 |
| **R5 Postmortem · 复盘** | 事故复盘模板：发生了什么 / 影响范围 / 根因 / 改进项 / 是否升级 Runbook。每次事故必须产出，沉到知识库 |

## 真实模板：ticket_silver 物化失败 · schema 漂移

```markdown
# Runbook · ticket_silver schema drift

## SYMPTOMS / 症状判定
- Dagit UI: ticket_fact_by_tenant FAILED
- 错误信息含: "ContractValidationError"
- Slack #data-alert 收到 P1 告警

## DIAGNOSIS / 排查
1. dagster asset-check status ticket_fact_by_tenant
   ↳ contract violation: 跳 ACTION-A
   ↳ schema mismatch:   跳 ACTION-B
   ↳ 其他:               升级 on-call senior

## ACTION-A · 契约违反
1. pytest tests/contract/test_ticket.py -v
2. slack @support-platform-oncall
3. additive 变更 → 升级 contract
4. breaking 变更 → quarantine 流程

## VERIFICATION / 验证
- dagster job execute ticket_silver_job \
    --partition $(date -u +%F)
- 预期: 5 分钟内 SUCCESS
- Grafana: support_ops.silver.lag_min < 30

## POSTMORTEM / 复盘
- 见 postmortems/template.md
- 必填: 时间 / 影响 / 根因 / 改进项
```

### 3 档成熟度

| 档位 | 模式 | 说明 |
| :--- | :--- | :--- |
| **L1 · Manual** | 人工执行 | Markdown / Confluence，每步明确命令。所有团队的起步，不要跳级 |
| **L2 · Semi-Auto** | 半自动 | 关键步骤变 CLI / Dagster Job。人工触发 + 流程化，失败回退人介入 |
| **L3 · Fully-Auto** | 全自动 | Skill Pack / Self-healing。检测 + 决策 + 执行闭环，只看结果 |

## 协作规范：命名、Ownership、SLO 的团队约定

### 资产命名 · 4 条铁律

1. **命名空间**：`{domain}.{layer}.{name}`
   - ✓ `support_ops.silver.ticket_fact`
   - ✗ `silver_ticket` / `ticket_v2_final`
2. **layer 仅四档**：bronze / silver / gold / serving，不要造新词（没有"platinum"）
3. **名词单数 + 蛇形**：不写复数，不写驼峰
   - ✓ `ticket_fact`
   - ✗ `tickets_facts` / `TicketFact`
4. **永远不要把版本写在 name 里**
   - ✓ `ticket_fact`（版本走 snapshot_id）
   - ✗ `ticket_fact_v2` / `ticket_fact_new`

### Ownership + SLO · 4 条不可妥协的硬规则

1. **每个资产 ≥ 1 个 Owner**（team-level，不是个人）：`owners=["team:support-platform"]`
2. **每个 P1 资产必须声明 FreshnessPolicy**：`maximum_lag_minutes` / `cron_schedule`，告警自动 routing 到 Owner team
3. **criticality 强制三档**：
   - P1 → 30 分新鲜度 + 立刻响应
   - P2 → 4-12 小时 + 工作时间响应
   - P3 → 周度 + 尽力而为
4. **任何「无主资产」不准上 prod**：CI 检查——asset 没声明 owner → fail

## 变更隔离：Branch Deployments + slim CI

90% 的「我改完上线发现你也改了」事故，根因是没做变更隔离。解药是把「数据栈」当代码栈管——每个 PR 起一个独立的数据沙盒，合到 main 才生效。

### Dagster · Branch Deployments

每个 PR 自动起一个隔离的 Dagster 部署：

- 开 PR → Dagster Cloud 自动起 branch deployment
- 同一份代码跑在「沙盒 schema」上，不动 prod
- 沙盒里能跑 backfill / 跑 asset check / 看血缘
- merge 到 main → 自动 promote 到 prod
- 关 PR → 沙盒资源自动回收

**工程价值**：两个团队同时改同一张表，各自有沙盒，互不踩；reviewer 可以直接在沙盒里看效果；出问题 retract PR 即可，prod 一行不动。

### dbt · slim CI

只跑 PR 真正影响到的 models，而不是全量：

```bash
# 放进 GitHub Action
dbt build \
  --select state:modified+ \
  --defer \
  --state ./prod-state
```

- `state:modified+` → 只跑被改的 model + 下游
- `--defer` → 上游引用直接指 prod state
- 一个 PR 的 CI 时间从 30 分钟 → 3 分钟
- 配合 `dbt build` → test 内置在同一次 CI

**工程价值**：工程师不用等长 CI → 迭代节奏 5×；资源消耗骤降，FinOps 也开心。

## Game Day 演练

把 Runbook 跑成肌肉记忆。Runbook 写完没有 Game Day，就跟健身办了卡不去一样。一年至少 4 次 Game Day，把 Runbook 在「假事故」里跑一遍——这是 Runbook 唯一可信的检验方式。

| 步骤 | 要求 | 做法 |
| :--- | :--- | :--- |
| **Step 1 · 选场景** | 场景库随机抽 | 从历史事故 + 风险清单里随机抽一个：上游 schema 漂移 / 某 tenant 数据漏 / iceberg metadata corrupt / 主备节点同时挂 |
| **Step 2 · 选演员** | 随机抽人 | on-call 名单里随机抽 1 人当主力 + 1 人当观察员。抽到 senior 也照样跑，不能换人——Game Day 不是让最强的人来跑 |
| **Step 3 · 真实演** | 只用 Runbook | 观察员只能记录，不能提示。主力照 Runbook 命令一行行执行。卡住的地方 → 就是 Runbook 的 bug。每个 bug 立刻记录到 issue |
| **Step 4 · 复盘升级** | 当晚出修订 | 当晚 1 小时复盘：Runbook 哪一步不顺？命令是不是已经废了？监控有没有缺？→ 当晚提 PR，3 天内合并 |

> Google 叫 Wheel of Misfortune，Netflix 叫 Chaos Monkey——本质都是同一件事：让事故在「可控时刻」发生，Runbook 才不会撒谎。

## 事故复盘模板（Postmortem）

一份好的 Postmortem 不是「回顾故障」，是「让同类问题不再发生」——少了任何一节，事故都白出了。

| 章节 | 内容 |
| :--- | :--- |
| **① 摘要（TL;DR）** | 3 句话讲清楚：发生了什么、影响了谁、修了多久。让没参与的人 30 秒读懂 |
| **② 影响范围** | 受影响的 asset / 下游 / 客户数 / SLO 违反时长。要带数字，不要写"少量" |
| **③ 事件时间线** | T+0 → T+N 完整时间线。每一行写「谁做了什么，看到了什么」。这是事后所有改进的素材 |
| **④ 根因分析（5 Why）** | 不要停在「网络抖了」「上游错了」——继续 ask why 直到看见组织/设计问题 |
| **⑤ 做得好的地方** | 不仅记录失败，也记录响应快的人 / 起作用的工具——固化下来 |
| **⑥ 做得不好的地方** | 坦诚——只有诚实复盘的团队才会进步。但只对事不对人（Blameless） |
| **⑦ 改进项（Action Items）** | 每一项 → 责任人 + DDL + 是否完成。这是 Postmortem 唯一的产出 |
| **⑧ Runbook 升级建议** | 本次事故是否升级 Runbook？如果是，新版 PR 链接放这 |

## Runbook 的下一站：Skill Pack

给 Agent 也能执行的「可调用知识」：

| 维度 | Runbook（人执行） | Skill Pack（人 + Agent 都能执行） |
| :--- | :--- | :--- |
| 形式 | 单个 Markdown 文档 | 目录 + SKILL.md + scripts/ + assets/ |
| 触发 | 人收告警手动打开 | description 字段供 Agent 自动匹配 |
| 执行 | 人复制命令逐步执行 | 脚本可被 Agent 参数化调用 |
| 验证 | 人看结果判断 | scripts 内置验证 + 结构化输出 |
| 约束 | 靠人理解上下文 | YAML frontmatter 声明输入 / 输出 / 风险 |
| 版本 | Git 跟 Markdown | 完整目录纳入 release manifest |

Week 9 会专门展开 Anthropic Agent Skills 标准——届时你今天写的 Runbook，会被升级成「Agent 触手可及的能力」。

## 大厂已经在走这条路

| 公司 | 实践 | 要点 |
| :--- | :--- | :--- |
| **Google SRE** | Wheel of Misfortune | 随机抽人 + 随机故障，必须照 Runbook 跑通，跑不通 = Runbook 是 bug。保证 Runbook 长期可用，新人 onboarding 直接接手 |
| **Anthropic** | Agent Skills Spec | 把工程规范封装为 Skill，Claude / Codex 按需发现加载，YAML frontmatter 声明边界。Runbook → 自动化的桥 |
| **Netflix** | Chaos + Runbook Driven | Chaos Monkey 主动制造故障，强制 Runbook 演练，没 Runbook 的服务不准上线。文化：事故是常态，Runbook 是工程一等公民 |

**Runbook 在大厂从「补救文档」升级成「工程一等公民」——这是 SRE 文化的核心成果，也是 AI 数据工程绕不开的一步。**
