# 5. 受控指标查询工具 v1 的契约、权限与审计设计

## 核心目标与背景

- **定调**：你和 LLM 之间，不能是"信任和祈祷"，必须是"契约和闸门"。
- **主要任务**：把自由度压进可测试、可授权、可审计的工具里，不让 Agent 裸写 SQL。
- **本课交付**：
  - 一份 tool contract（JSON Schema strict + additionalProperties=false）
  - 一份 runtime（参数化 SQL + 7 道闸 + 9 种 denial_code）
  - 一份 audit payload（trace_id / release_id / policy_applied）
  - 在 OmniSupport 项目里跑通 4 个正负例 + Tool API endpoint

## WAR STORY：2024 PII 越权事故

某客户 Agent 上线 3 个月后，数据库日志里出现大量 customer_email。

**事故现场**：客服把客户邮箱复制到 Agent 提示词里："帮我查 xxx@example.com 最近的工单"。Agent 为了"更好回答"，把 customer_email 当过滤条件写进了 SQL。

**后果**：

- 数据库 query log 出现大量 `customer_email = ...` 的明文查询
- DBA 备份系统也跟着把这些查询固化下来
- 合规审查发现：PII 跨系统的非授权暴露

**修复花了 11 周**：8 周合规审查 / DPO 评估 / 客户通知；3 周工程修复。总成本是整套 Agent 工具治理预算的 6 倍。

**判断**："Agent 永远不能写 SQL。"不是它能力不够——是它没法对结果负责。

## KEY THESIS：五道闸缺一不可

受控工具不帮 Agent 拼 SQL；它从根上换了接口形状——你直接把 SQL 拿走，给它一个只能传"指标 + 维度 + 时间窗"的入口。

| 闸 | 保护什么 |
| :--- | :--- |
| **① schema validation** | 形状先合法。JSON Schema strict |
| **② registry lookup** | 指标/维度/过滤必须来自白名单 |
| **③ guards** | 角色/窗口/成本/PII/组织范围 |
| **④ parameterized SQL** | 确定性查询。不允许 raw SQL |
| **⑤ audit payload** | 可复现/可追责/可评测。audit_id + release_id |

**schema 限"形状"；registry 限"语义"；guards 限"权限与成本"；参数化 SQL 限"确定性"；audit 限"可复盘"。**

## RISK TABLE：6 类风险对应 6 道 guard

| 风险 | 具体表现 | 工具 guard |
| :--- | :--- | :--- |
| **口径漂移** | 同名指标 SQL 不同；月会三个数 | metric registry + accepted_values |
| **越权访问** | 查到 customer_email / 评论正文 / 未授权维度 | role policy + safe view |
| **注入/非法过滤** | Agent 接受任意 where；prompt 走漏 | filter whitelist + 参数化 SQL |
| **成本失控** | 365 天大窗口 / 全表扫 / limit 失控 | max_window_days + limit cap |
| **审计缺失** | 不知道谁查了什么；出问题无法复盘 | audit payload + trace_id + release_id |
| **不可复现** | SQL 每次生成不同；答案漂 | 参数化 SQL + idempotency_key_fields |

## LLM 工程基石：Function Calling x Structured Outputs

| 概念 | 说明 |
| :--- | :--- |
| **Function Calling** | 告诉模型"我有这些工具，每个输入是这样的 JSON Schema"。模型输出工具调用 + 符合 schema 的 JSON，应用层执行真正的函数 |
| **Structured Outputs (strict)** | 2024.08 OpenAI 推出。`strict: true` 后输出严格符合 JSON Schema，4o-2024-08-06 实测 100%。要求 `additionalProperties: false` + `required` 全列 |
| **Week05 怎么用** | 把 `query_support_kpis_v1.json` 的 input_schema 直接交给 Function Calling，模型生成的 payload 100% 形状合法；剩下交给 runtime guards |

**为什么 strict 还不够**：strict 只保证"形状合法"。它不知道指标真不真实、角色能不能查、31 天是不是太长。这些必须 runtime 兜底。

## TOOL DEFINITION：它"不是"什么很关键

| 它是 | 它不是 |
| :--- | :--- |
| registry-driven：从 metric_registry_v1.yml 读真相 | 不是 NL2SQL；不接受自然语言 |
| parameterized：只生成参数化 SQL | 不是任意选表选字段 |
| structured + audit：返回结构化 rows + audit 字段 | 不是 raw SQL 安全壳；不接受 raw_sql 参数 |
| idempotent：相同输入 → 相同结果 | 不是完整 Week10 治理；HITL/配额还在后面 |

**工具的价值在于"拒绝边界"，不在"灵活性"。**

## INPUT SCHEMA：收紧到极致

`additionalProperties: false` 是 schema 真正的"安全锁"——让 `raw_sql` 这种字段彻底进不来。

```json
{
  "name": "query_support_kpis_v1",
  "version": "v1.1",
  "input_schema": {
    "type": "object",
    "required": ["actor_role", "metrics", "date_from", "date_to"],
    "properties": {
      "actor_role":    { "type": "string" },
      "actor_id":      { "type": "string" },
      "trace_id":      { "type": "string" },
      "purpose":       { "type": "string",
                         "enum": ["classroom_demo","support_ops_analysis",
                                  "incident_review","regression_eval"],
                         "default": "classroom_demo" },
      "actor_org_ids": { "type": "array", "maxItems": 50,
                         "items": { "type": "string",
                                    "pattern": "^[A-Za-z0-9_.:-]+$" } },
      "include_experimental_metrics": { "type": "boolean", "default": false },
      "metrics":       { "type": "array", "minItems": 1, "maxItems": 11,
                         "items": { "type": "string",
                                    "pattern": "^[a-z][a-z0-9_]*$" } },
      "date_from":     { "type": "string", "format": "date" },
      "date_to":       { "type": "string", "format": "date" },
      "dimensions":    { "type": "array", "maxItems": 4 },
      "filters":       { "type": "object" },
      "limit":         { "type": "integer", "minimum": 1, "maximum": 500,
                         "default": 100 }
    },
    "additionalProperties": false
  }
}
```

## RUNTIME：6 步确定性查询

每一步都不可绕过：

```
① load       → load schema + load registry
② validate   → jsonschema.validate strict → SCHEMA_VALIDATION_FAILED
③ policies   → ROLE / METRIC / DIMENSION / FILTER / WINDOW / ORG_SCOPE / EXPERIMENTAL 七道闸
④ build SQL  → 参数化 placeholder，select from analytics.safe_view
⑤ execute    → asyncpg.fetch → DB_UNAVAILABLE 兜底
⑥ assemble   → rows + audit + data_freshness + policy_applied
```

### _build_query 核心逻辑

```python
def _build_query(payload, registry):
    where = ["metric_name = any($1::text[])",
             "metric_date between $2::date and $3::date"]
    params = [payload["metrics"],
              _parse_date(payload["date_from"]),
              _parse_date(payload["date_to"])]

    for field, value in payload.get("filters", {}).items():
        params.append([str(v) for v in (value if isinstance(value, list) else [value])])
        where.append(f"{field} = any(${len(params)}::text[])")

    if payload.get("actor_role") != "admin" and payload.get("actor_org_ids"):
        params.append([str(v) for v in payload["actor_org_ids"]])
        where.append(f"org_id = any(${len(params)}::text[])")  # 行级权限

    return (f"select ... from analytics.{registry.safe_view} "
            f"where {' and '.join(where)} ..."), params
```

## v1.1 ENHANCEMENTS："加字段不破"

v1.1 在 v1 上只加 4 个字段——这是工具版本演进的标准姿势：

| 新增字段 | 作用 |
| :--- | :--- |
| **trace_id** | 端到端追踪 ID，从 Agent → 工具 → DB → 评测的 5 跳，全用同一 trace_id 串起来 |
| **purpose** | 声明用途：classroom_demo / support_ops_analysis / incident_review / regression_eval。用于审计与策略匹配 |
| **actor_org_ids** | 调用方的组织 ID 列表。课堂版客户端传；生产版应由服务端身份解析后注入 |
| **include_experimental_metrics** | 显式确认才允许 experimental_proxy 指标，避免 LLM 在不知情时用代理口径"高自信地错" |

## DENIAL CODES：9 种拒绝码

拒绝码不是失败，是接口能力——它让 LLM"知道怎么改"。

| code | 触发条件 | 对 Agent 的提示 |
| :--- | :--- | :--- |
| SCHEMA_VALIDATION_FAILED | 输入不符合 JSON Schema | 改 payload 形状 |
| ROLE_DENIED | actor_role 不在 allowed_roles | 换角色或申请权限 |
| METRIC_DENIED | 指标未注册或角色不允许 | 换指标/申请白名单 |
| DIMENSION_DENIED | dimension 不在 allowed_dimensions | 改维度选择 |
| FILTER_DENIED | filters 字段不在 allowed_filters | 改过滤字段 |
| WINDOW_TOO_LARGE | date_to - date_from > max_window_days | 缩小窗口 |
| ORG_SCOPE_REQUIRED | support_ops 没传 actor_org_ids | 补 org scope |
| EXPERIMENTAL_METRIC_NOT_ACKNOWLEDGED | experimental 指标未确认 | 设 include_experimental_metrics=true |
| DB_UNAVAILABLE | 底层 DB 不可用 | 稍后重试或降级 |

## AUDIT PAYLOAD：6 个维度的"保命字段"

没有 audit 的工具调用不应进入生产。出事后，这 6 个维度让你能回答："谁查了什么、什么版本、走了哪些策略、得到什么结果"。

| 维度 | 字段 |
| :--- | :--- |
| **Who（调用方）** | actor_id / actor_role / trace_id / purpose |
| **What（请求内容）** | metrics / dimensions / filters / date_from / date_to / actor_org_ids |
| **Registry（语义版本）** | registry_id / registry_version / safe_view |
| **Data（数据快照）** | release_id / data_release_id / generated_at_max |
| **Outcome（结果）** | row_count / denial_code / query_fingerprint（输入指纹） |
| **Policy（走的策略）** | policy_applied 列表：tool_contract / metric_registry / safe_view / parameterized_sql / role_policy / window_policy / org_scope_filter ... |

## RECAP：7 个判断

1. Agent 可以问指标，但不能裸写 SQL——这是 LLM 时代的工程师必答题。
2. tool contract 不是提示词，是工程边界；JSON Schema strict + additionalProperties=false 是关键。
3. registry 是 runtime 的事实来源；safe view 是数据边界；参数化 SQL 是安全底线。
4. structured result 是审计和评测基础；audit payload 是生产复盘入口。
5. v1.1"加字段不破"是工具版本演进的标准姿势——以后做任何工具都这么干。
6. 9 种 denial_code 让 LLM"知道怎么改"——这是把 LLM 当作"可学习客户端"对待。
7. Week05 指标包会被 Week06 / Week08 / Week10+ 继续消费——它不是终点，是地基。
