# 1. 把业务口径变成工程接口

## 核心目标与背景

- **定调**：指标不是 SQL 的结果，是要能被多方一起负责的工程接口。
- **演进**：从 Week01-04 的数据工程（把数据搬进仓）升级到 Week05 的分析工程（把仓里的事实做成业务能消费的口径）。
- **主要任务**：把"指标 = SQL"这个直觉打掉，建立"指标 = 定义 + 来源 + 粒度 + 测试 + 文档 + 权限 + 接口"的工程认知。

## 本课交付物

- 一份"指标接口卡"（先用纸写，5 分钟出草稿）
- 一份 ADR：为什么走"本地 dbt + registry"，而不是 dbt Cloud / Cube
- 两份蓝图：`adr-week5-analytics-path.md` / `metric-interface-principles.md`

## WAR STORY：CFO 一句话，三周复盘

2020 年某电商，CFO 月会前问运营"P1 工单这季度涨了多少"：

- 运营看板：+28%（按 `created_at` 算新建 P1）
- BI 临时报表：+41%（按 `last_status_change_at` 算历史升级）
- 客服 PPT：+12%（按 `priority_changed_at` 加 SLA tier 过滤）

三个 SQL 都跑得通，三个口径都"合理"，没有一个能在公司内部被多方共识。

**根因**：不是 SQL 写错，而是没有统一的 owner、grain、时间字段定义、查询权限和审计。

**解法**：把所有"和 P1 工单"相关的口径重做成一份"指标接口"——有 owner、有 grain、有时间字段定义、有谁能查、有审计。

## KEY THESIS：指标接口的 7 要素

没有 grain / owner / source / filters / tests / audit / docs，它不叫接口。

三方诉求：

| 角色 | 诉求 |
| :--- | :--- |
| **BI** | 稳定显示，不要每张看板复制一份口径 |
| **Agent** | 受控查询，别让它"自由发挥" |
| **治理** | 可复盘可追责：哪天的数、哪一版语义、谁查的 |

## CONCEPT SPLIT：指标的三个工程对象

中文里都翻译成"指标"，但生产里它们是三个不同的对象：

| 对象 | 答什么问题 | 生产例子 | 混了的后果 |
| :--- | :--- | :--- | :--- |
| **metric（指标）** | 业务上要复用、要解释的数字 | `p1_ticket_count` | 同名不同义；月会上吵半天 |
| **business definition（业务口径）** | 怎么算、什么粒度、什么时间窗 | P1 = priority in (p1, p1_critical) 且按 created_date | SQL 跑得通但口径漂；先甩锅 |
| **engineering interface（工程接口）** | 谁能查、怎么查、输出什么、如何审计 | `query_support_kpis_v1` + registry + audit_id | Agent 越界、无法复盘、不可评测 |

**metric 是名字，definition 是含义，interface 让多方对得齐。**

## METRIC CARD：16 字段最低必填集

少一个字段，事故就以那个字段的形状出现。

| 字段 | 说明 | p1_ticket_count 示例 |
| :--- | :--- | :--- |
| metric_name | 稳定机器名 | p1_ticket_count |
| business_label | 人能读的名字 | P1 工单数 |
| business_question | 它在回答的问题 | 最近 P1 是否变多/集中在哪些产品线 |
| business_definition | 业务定义 | 窗口内创建的 P1 或 P1_critical 工单数 |
| source_model | 来源 mart | support_kpi_mart |
| grain | 一行代表什么 | metric_date x product_line x priority x org_id x category |
| measure_expression | 计算逻辑 | sum(p1_ticket_count) |
| time_field | 时间字段 | metric_date |
| allowed_dimensions | 允许维度 | product_line / priority / org_id / category |
| allowed_filters | 允许过滤 | 同上 + data_release_id |
| owner | 口径负责人 | support_ops |
| tests | 必要测试 | not_null / accepted_values / registry validation |
| docs_link | 文档入口 | dbt docs / metric-interface-principles.md |
| allowed_roles | 允许角色 | support_ops / instructor / admin |
| audit_fields | 审计字段 | actor_role / actor_id / release_id / row_count |
| change_policy | 改口径要做什么 | 先写 impact note，再更新 tests / docs / registry |

## CHECKLIST：上线前 10 个问题

答不上来一项 = HOLD。HOLD 不是失败，是工程纪律。

1. **Owner**：谁负责解释和修复？
2. **Grain**：一行到底代表什么？
3. **Time field**：created_at / updated_at / event_time？
4. **Filters**：默认过滤条件是什么？
5. **Dimensions**：开放哪些；不开哪些？
6. **Tests**：哪些断言保护口径？
7. **Docs**：消费者读得懂吗？
8. **Lineage**：变更影响能定位吗？
9. **Roles**：哪些角色能查？
10. **Agent gate**：能通过受控工具调用吗？

## COMMON MISREADS：四个常见误区

| 你大概率信过 | 现实是 | 代价 |
| :--- | :--- | :--- |
| "指标就是 SQL" | 指标是定义 + 来源 + 粒度 + 测试 + 文档 + 权限 + 接口的合体 | 月会三个数；CFO 拍桌 |
| "Dashboard 显示了就够" | Agent / Eval / 治理 / 运营复盘都要同一口径 | AI 答错后没人能复盘 |
| "SQL 改了大家同步一下" | 没有 lineage 和 docs，影响范围根本不可控 | 旧口径继续被消费 6 个月没人发现 |
| "Agent 会自己写对 SQL" | 它会"高自信地"写出能跑、口径错、越权、不可审计的 SQL | 一次 PR 事故 ≈ 三年指标治理预算 |

## BAD CASE：同一个问题的四方答案

问题："最近 7 天 P1 工单是不是涨了？"

| 谁回答 | 怎么算 | 数字差异来自哪 | 暴露的真问题 |
| :--- | :--- | :--- | :--- |
| 运营看板 | 按 created_at 聚合新建 P1 | 时间字段无约定 | 没人写明"P1 用哪个时间" |
| BI 临时报表 | 按 updated_at + 历史升级算 | grain/filters 无约定 | 没有 owner，每个分析师拼一遍 |
| Agent 工具 | 查底表/自己生成 SQL，可能带出 PII | 工具边界和审计无约定 | 没有 registry，也没 safe view |
| 评测复盘 | 复用某张 dashboard 截图 | 数据 release_id 无约定 | 没有可复现的事实快照 |

**四方都"能算"，但没人能"负责"——Week05 修的就是这条裂缝。**

## LANDING：OmniSupport 的 11 个指标

| 指标 | 业务问题 | 粒度 | 最危险误差 | 主要消费者 |
| :--- | :--- | :--- | :--- | :--- |
| p1_ticket_count | P1 是否变多 | daily metric | 时间字段选错 | BI / Ops / Agent |
| sla_breach_count / rate | SLA 是否失守 | daily metric | 状态窗口未约定 | Ops / Eval |
| open_ticket_count | 当前积压规模 | daily metric | closed 口径错 | Ops / Agent |
| escalation_count / rate | 升级是否变多 | daily metric | 重复计数 | Ops / Agent |
| avg_first_response_minutes | 首响是否变慢 | case-level avg | 忽略业务时区 | BI / SLA |
| first_resolution_rate | 一次解决率（代理） | ratio | 分母混乱；reopen 未接入 | Eval / Ops |

`first_resolution_rate` 被故意标为 `experimental_proxy`——示范"诚实地标'代理口径'"比偷偷上线高级一万倍。

## 练习：为 sla_breach_rate 写一张卡

```yaml
metric_name: sla_breach_rate
business_label: SLA 违约率
business_question: 服务承诺是不是在被打破？
business_definition: SLA 违约工单数 / 工单总数
source_model: support_kpi_mart
grain: [metric_date, product_line, priority, org_id, category]
measure_expression: avg(sla_breach_rate)  # 已在 mart 里聚合好
time_field: metric_date
allowed_dimensions: [product_line, priority, org_id, category]
allowed_roles: [support_ops, instructor, admin]
max_window_days: 31
tests: [between_0_and_1, denominator_not_zero_when_reported]
docs_link: docs/blueprints/week05/metric-interface-principles.md
audit_fields: [actor_role, actor_id, release_id, row_count]
change_policy: ratio 改口径必须双写一段时间，并发邮件给 BI/Ops
```

## 自检：6 个问题

1. 我能用一句话解释为什么 KPI 不是 SQL 结果
2. 我能说清 BI / Agent / Eval / 治理为什么必须共享一套口径
3. 我能指出 P1 工单事故里三方数字不同的根因
4. 我能列出 Agent 裸写 SQL 至少三类风险
5. 我能为任意一个指标写出 grain / source / time field / dimensions
6. 我能解释为什么 first_resolution_rate 是 experimental_proxy

## RECAP：7 个判断

1. 指标不是 SQL，是 metric 名字 x business definition x engineering interface 的合体。
2. grain（一行是什么）比 SQL 写法重要十倍——grain 错了，看着都对，全是错。
3. owner / source / filters / dimensions / max_window 五个字段必须可见，缺一就是事故种子。
4. BI、Agent、Eval、治理必须共享同一口径——不是想不想，是 2026 的合规底线。
5. 指标接口卡 16 字段是"最低集合"，不是装饰；少一个，事故就以那个字段的形状出现。
6. 上线前 10 个问题，任一答不出 = HOLD。HOLD 比事故便宜十倍。
7. 下一讲：用 dbt 分层，让指标接口从 source 自然长出来。
