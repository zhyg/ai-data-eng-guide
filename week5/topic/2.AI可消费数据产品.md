# 2. AI 可消费数据产品

## 核心目标与背景

- **定调**：分层不是目录命名，是"上游变化不污染下游"的工程纪律。
- **主要任务**：讲清 dbt 四层（sources / staging / intermediate / marts）各自的消费边界和"不该做的事"。
- **本课交付**：
  - 一份"为什么 4 层"的决策记录（不是默认照抄）
  - 能讲清 staging / intermediate / marts 各自的"不该做的事"
  - 在 OmniSupport 项目里跑通 `dbt build --select tag:week05`

## WAR STORY：14 处同时挂掉

2018 某物流公司，上游同事把 status 从 `"In Transit"`（字符串）改成枚举 `IN_TRANSIT`（大写），他自己 PR 全绿。

下游 14 处同时挂——3 张 BI 看板、4 个 cron 报表 SQL、2 个 stored procedure、2 个 Python 脚本、2 个 Hive 视图、1 个客户对账接口。

**根因**：

1. 没有 sources 声明：上游表谁在用，没人能查
2. 没有依赖图：改动影响范围靠"我大概记得"
3. 没有测试：枚举漂移没人发现
4. 修复全靠 grep

**CTO 事故复盘时的判断**："这事必须靠工具，不靠人记。"

dbt 第一次让 SQL 拥有"代码的属性"：依赖图、测试、文档、可复现命令。

## 数据栈演进：15 年路径

| 时代 | 主流路径 | 埋下的雷 | 代表工具 |
| :--- | :--- | :--- | :--- |
| 2005-2015 ETL | 源系统 → ETL 服务 → 仓（变形在仓外） | 脚本不进版本控制；分析师够不着 | Informatica / Talend / 手写 PySpark |
| 2015-2020 ELT | 源系统 → 仓内 staging → 仓内 transform | 仓内 SQL 散落，没依赖、没测试、没文档 | Fivetran / Airbyte + 仓内手写 SQL |
| 2020+ Analytics Engineering | SQL 工程化：模型 + 依赖 + 测试 + 文档 + DAG 编译 | 新工种诞生：分析工程师 | dbt Core / dbt Cloud / SQLMesh |

**dbt 把"写 SQL"变成了"写代码"——它逼你像写程序一样写指标。**

## dbt 的 6 块基本盘

1. **模型 = SQL 文件**：每个 `.sql` = 一个表/视图。dbt 编译成 CREATE TABLE/VIEW 落仓。dbt 不存数据，它只生成 SQL。
2. **Jinja 模板**：SQL 里写 `{{ ref(...) }}` / `{{ source(...) }}` / `{{ var(...) }}`。SQL 第一次拥有变量、循环、宏。
3. **依赖 DAG**：只要写 `{{ ref("stg_tickets") }}`，dbt 就知道依赖，拓扑排序自动算运行顺序。
4. **Tests**：内置 not_null / unique / accepted_values / relationships，再加自定义 SQL test。每次 build 自动跑。
5. **Docs + Artifacts**：`dbt docs generate` → manifest / catalog / run_results。血缘网站 + 审计证据，两件事一次到位。
6. **Materialization**：view / table / incremental / ephemeral。这是"成本 x 新鲜度"决策点。

## dbt IS / IS NOT

| 你可能以为 | 其实不是 | 它真正负责的 |
| :--- | :--- | :--- |
| dbt 是数据库 | 它不存数据；数据在仓里 | SQL transform 的工程框架 |
| dbt 是 BI | 它不画图、不交互 | 把模型和口径变成可复现对象 |
| dbt 是采集工具 | 它不负责 EL；上游要先有数据 | 承接已加载的数据，再加工 |
| dbt 是 SQL 文件夹 | 只堆 SQL 不叫 dbt | 依赖、测试、文档、DAG、artifacts |
| dbt = 语义层 | dbt 本身不暴露指标 API | 语义层定义的事实来源 |

## KEY THESIS：分层的核心价值

**分层 = 污染隔离 + 消费边界 + 影响可定位**，不是给目录起好听名字。

- **污染隔离**：上游漂了控制在 staging；业务规则变了控制在 intermediate；消费方变了只动 marts 不动底层。
- **消费边界**：每一层有"该做的"和"绝对不该做的"。mart 字段不是"塞越多越好"，是"刚好够下游用"。
- **影响可定位**：依赖图清晰，"改了 X 谁会受影响"一分钟看得到。

**判断一个团队的 dbt 工程好不好，不看模型数量，看"目录之间该做的事是不是泾渭分明"。**

## LAYER 1：SOURCES

承认真实上游——这一层越简单越稳。

```yaml
# analytics/models/sources.yml
sources:
  - name: omni_postgres
    description: "Week03/Week04 已经加载到 PostgreSQL 的结构化工单数据"
    database: "{{ env_var('POSTGRES_DB', 'omnisupport') }}"
    schema: public
    tables:
      - name: ticket_fact
        columns:
          - name: ticket_id
            tests: [not_null, unique]
          - name: created_at
            tests: [not_null]
      - name: customer_dim
      - name: ticket_comment_fact
      - name: knowledge_doc
```

| 可以做 | 不要做 |
| :--- | :--- |
| 声明物理上游表 | 不做 cast、不做 rename（rename 是 staging） |
| 挂关键字段约束测试 | 不写 case-when |
| 配 freshness | 不让 dashboard 直接引用 |

**sources 是上游和下游之间的契约。越简单越稳；一旦复杂，上游一变这层就崩。**

## LAYER 2：STAGING

整形输入，绝不塞 KPI。

```sql
-- analytics/models/staging/stg_tickets.sql
with source as (
    select * from {{ source('omni_postgres', 'ticket_fact') }}
),
renamed as (
    select
        ticket_id, customer_id, org_id,
        lower(status::text)   as status,
        lower(priority::text) as priority,
        cast(created_at as date) as created_date,
        coalesce(updated_at, created_at) as updated_at,
        case when lower(status::text) in ('open','pending_customer','in_progress','escalated')
             then true else false end as is_open,
        case when lower(priority::text) in ('p1_critical','p1')
             then true else false end as is_p1,
        case when sla_due_at is null then false
             when coalesce(resolved_at, current_timestamp) > sla_due_at then true
             else false end as sla_breached
    from source
)
select * from renamed
```

| 可以做 | 坚决不要 |
| :--- | :--- |
| rename / cast / 枚举标准化 | 不做 group by 聚合 |
| 派生纯类型布尔（is_open / is_p1） | 不写 dashboard 专用字段 |
| PII 标记（不删数据加标志） | 不做跨表 JOIN 业务逻辑 |

**"派生布尔 OK，聚合 KPI 不 OK"——这是 staging 与 intermediate/marts 的最重要分界。**

## LAYER 3：INTERMEDIATE

业务组合的"集中入口"，不直接面向消费者。

```sql
-- analytics/models/intermediate/int_support_cases.sql
with tickets as ( select * from {{ ref('stg_tickets') }} ),
     customers as ( select * from {{ ref('stg_customers') }} ),
     first_comments as (
         select ticket_id, min(created_at) as first_comment_at
         from {{ ref('stg_ticket_comments') }}
         group by ticket_id
     ),
joined as (
    select
        t.ticket_id, t.org_id, c.org_name,
        t.status, t.priority, t.created_date, t.resolved_at,
        fc.first_comment_at,
        case when fc.first_comment_at is not null
             then extract(epoch from (fc.first_comment_at - t.created_at)) / 60.0
             else null end as first_response_minutes,
        case when t.resolved_at is not null
             then extract(epoch from (t.resolved_at - t.created_at)) / 86400.0
             else extract(epoch from (current_timestamp - t.created_at)) / 86400.0
             end as backlog_age_days,
        case when t.resolved_at is not null and not t.is_escalated then true
             else false end as is_first_resolution_proxy
    from tickets t
    left join customers c on t.customer_id = c.customer_id
    left join first_comments fc on t.ticket_id = fc.ticket_id
)
select * from joined
```

| 这一层做的事 | 边界 |
| :--- | :--- |
| JOIN ticket + customer + comment | 不直接给 BI / Agent 消费——不稳定 |
| 派生"业务事实"：first_response_minutes 等 | 不开放给外部团队引用——是内部细节 |
| 一行还是 case 粒度 | 不做最终聚合（那是 mart） |

## LAYER 4：MARTS

marts 不是"最后一个文件夹"，是"对外契约"。

**三个核心 mart：**

| Mart | Grain | 作用 | 物化 |
| :--- | :--- | :--- | :--- |
| **support_case_mart** | 一个 support case | 供分析师/BI 做 case-level 切片 | table |
| **support_kpi_mart** | 指标 x 日期 x 维度 | 11 个 metric 全走它，Week05 最核心 mart | table |
| **agent_tool_input_view** | 同 kpi_mart | Agent 唯一能读的对象，只暴露白名单字段 | view |

**一句话判断**：打开 schema.yml，说不出"一行是什么 + 谁消费 + 不能暴露什么"——这个 mart 还不该被消费。

## GRAIN：工程师之间最便宜的共识

| grain | 一行是什么 | 错了会怎样 |
| :--- | :--- | :--- |
| case grain | 一个 support case | 复开率分母混乱；reopen 被重复算 |
| daily activity grain | 某天 x 某类活动 x 某维度组合 | 历史升级被重复算；漏掉时区 |
| metric row grain | 某指标 x 某日期 x 某维度组合 | 不同指标被混到同一行；列错位 |
| query result grain | 工具返回的结构化结果 | Agent 汇总错维度；下游图表说谎 |
| snapshot grain | 某时刻的当前状态快照 | 把"已关闭"算成"当前打开" |

**评审 mart 的第一句话永远是："这一行代表什么？"——答不上来，PR 不批。**

## support_case_mart 解剖

重点不在 SQL 本身，在于：什么字段被选了 / 什么字段被故意没选。

```sql
select
    ticket_id, org_id, org_name, status, priority,
    category, product_line, product_version, assignee_id,
    sla_tier, sla_due_at,
    created_at, created_date, updated_at, resolved_at, first_comment_at,
    first_response_minutes,
    handle_time_minutes, backlog_age_days,
    is_open, is_resolved, is_escalated, is_p1, sla_breached,
    is_first_resolution_proxy,
    pii_level, pii_redacted,
    data_release_id, ingest_batch_id, schema_version
from {{ ref('int_support_cases') }}
-- 故意没选的：customer_email / phone / subject 全文 / comment body
```

| 它保护的指标 | 严禁暴露的 |
| :--- | :--- |
| reopen_rate / active_ticket_count / sla_breach_count | customer_email、phone |
| backlog_age 分布 | 工单 subject 全文 / comment body 原文 / IP / 设备指纹 |

## support_kpi_mart 解剖：宽表 → 长表翻转

核心技巧：PostgreSQL 的 `CROSS JOIN LATERAL VALUES` 把"宽表"翻转成 `metric_name + metric_value` 的长表。

```sql
with daily as (
    select activity_date as metric_date, product_line, priority, org_id, category,
           ticket_count, open_ticket_count, p1_ticket_count, ...
    from {{ ref('int_ticket_activity_daily') }}
),
metric_rows as (
    select daily.metric_date, metrics.metric_name,
           daily.product_line, daily.priority, daily.org_id, daily.category,
           metrics.metric_value
    from daily
    cross join lateral (
        values
            ('ticket_count',      daily.ticket_count::numeric),
            ('p1_ticket_count',   daily.p1_ticket_count::numeric),
            ('sla_breach_count',  daily.sla_breach_count::numeric),
            -- ... 11 个指标全部一次性翻转
    ) as metrics(metric_name, metric_value)
    where metrics.metric_value is not null
)
select metric_date, metric_name, product_line, priority, org_id, category,
       metric_value,
       '{{ var("week05_data_release_id") }}' as data_release_id,
       current_timestamp as generated_at
from metric_rows
```

**做客服 / 销售 / 风控指标包，长表骨架都是这套。**

## 安全视图与物化策略

```sql
{{ config(materialized='view') }}
select metric_date, metric_name, product_line, priority, org_id, category,
       metric_value, data_release_id, generated_at
from {{ ref('support_kpi_mart') }}
where metric_name in (
    'ticket_count','open_ticket_count','p1_ticket_count', ...
)
```

| 物化 | 什么时候用 | 本项目选哪个 |
| :--- | :--- | :--- |
| **view** | 中小数据；要"最新" | staging / intermediate / safe view |
| **table** | 稳定列、稳定快照 | support_case_mart / support_kpi_mart |
| **incremental** | 大表，按日期/批次增量 | 本周不用；事件流大表才需要 |
| **ephemeral** | 只被 1-2 个模型用的中间步骤 | 本周不用；保留作为优化项 |

**safe view 为什么必须是 view**：要实时跟上 mart 最新构建；做成 table 会有"两份数据"的隐患。

**物化口诀**：staging/intermediate 默认 view（便宜）；marts 默认 table（稳定）；大表才考虑 incremental。

## 自检：6 个问题

1. 我能讲清四层各自的"不该做的事"
2. 我能用一句话说出 case_mart / kpi_mart / safe_view 的 grain
3. 我没把 KPI 聚合写进 staging
4. 我能解释 LATERAL VALUES 长表翻转的工程意义
5. 我知道 safe view 为什么必须是 view
6. 我知道大表才考虑 incremental，不是越大越用

## RECAP：7 个判断

1. dbt 不是数据库 / BI / 采集——它是 SQL 工程化框架。
2. 分层 = 污染隔离 + 消费边界 + 影响可定位；不是目录命名。
3. sources 越简单越稳；staging 整形不塞 KPI；intermediate 是业务组合的集中入口。
4. marts 不是"最后文件夹"，是对外契约；说不出 grain 就不该被消费。
5. metric-row 长表（CROSS JOIN LATERAL VALUES）是 BI/Agent/Eval 共享口径的工程关键。
6. safe view 必须是 view；物化按"新鲜度 x 成本 x 下游"挑，不是越大越 incremental。
7. 下一讲：把"能跑的 SQL"升级成"能负责的口径"——证据链。
