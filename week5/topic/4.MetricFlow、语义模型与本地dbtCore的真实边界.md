# 4. MetricFlow、语义模型与本地 dbt Core 的真实边界

## 核心目标与背景

- **定调**：先把内核立住，再谈平台外延。这是给真实架构师的判断，不卖任何 SaaS。
- **主要任务**：给你"要不要上、什么时候上、怎么过渡"的判断。
- **本课交付**：
  - 能讲清 metric / measure / dimension / entity / semantic graph 五个对象
  - 一份决策记录：本周为什么走 local registry，而不是 dbt SL / Cube / Looker
  - 在 OmniSupport 项目里跑通 `metric_registry_v1.yml` 的 v1.1 实验扩展

## WAR STORY：60 万美元/年的真实判断

2025 某客户花 60 万美元/年订阅 dbt Cloud Semantic Layer，定义了 300 个指标，准备给 BI、CRM、Agent 共享。

一年后架构盘点：BI 实际用了 78 个，Agent 用了 31 个。剩下 191 个"指标"——有定义、有 owner、没人用，且半数说不清 grain。

**真正坏点不在平台**，而在"组织还没准备好"——metric_name 还没统一、grain 还没人能讲清。

**判断**：先把本地 registry 做扎实，验证 6 个月，再决定要不要上托管。

**现实主义**：不是反平台——是反"跳过准备就买票"。语义层的核心从来不是"用哪家"，是"你公司里同名指标能不能不变意思"。这个内核立不住，再贵的平台也救不了。

## 2025-2026 语义层市场全景

| 类型 | 代表 | 本质 | 与 Week05 的关系 |
| :--- | :--- | :--- | :--- |
| 仓内 SQL 工程化 | dbt Core / SQLMesh | 把 SQL 编译成可负责的模型 | Week05 底座，必选 |
| dbt SL (managed) | dbt Cloud Semantic Layer | 托管 metric API：GraphQL / SQL / Python | 不强依赖；可作未来迁移目标 |
| Open MetricFlow | Apache 2.0 开源 | dbt SL 的查询引擎 | 本周作为参考；不要求生产化 |
| 通用语义层平台 | Cube / AtScale / GoodData | 独立部署服务；含 cache / ACL / API | 已有可平迁；不强求 |
| BI 内置语义 | LookML / Power BI 模型 | 绑定 BI 工具；Gemini 进入中 | 与 Agent 共享口径困难 |
| OSI v1.0 | 2026.01 Apache 2.0 | 跨厂商语义层交换标准 | 让 registry 可跨平台迁移 |

## BOUNDARY：核心 vs 外延

把核心立住，迁到任何平台都是"换前端"；核心没立，平台救不了你。

```
Semantic Core          → Local Registry           → Tool Contract
metric_name               metric_registry_v1.yml      query_support_kpis_v1
grain                      版本号 + owner              JSON Schema strict
dimensions                 allowed_dimensions          输入收紧
filters                    sensitivity
roles + window

                       → Semantic Model Draft      → Managed SL / APIs
                          entity / measure             dbt SL / Cube / OSI
                          准 MetricFlow 形态            GraphQL / MCP
                          future mapping               optional extension
```

**Week05 只做前两件**：Semantic Core + Local Registry。Tool Contract 下一节做完。后两件（Semantic Model Draft + Managed SL）是迁移目标，不是本周硬要求。

**本地可跑、可解释、可审计、可迁移——这条路径对所有公司都通用。**

## KEY THESIS：同一个词不要变意思

语义层的核心不是平台名，是"事实来源"。

| 要素 | 要求 |
| :--- | :--- |
| **metric name** | 名称唯一，不复用歧义词。`ticket_count` 有歧义；宁可叫 `ticket_created_count` |
| **grain** | 一行 = 哪个粒度，必须可见。标准：metric x date x dimensions |
| **dimensions / filters** | 可切维度与过滤白名单。不允许"任意 where" |
| **role / time window** | 谁能查，能查多久（max_window_days）。本质是访问控制 + 成本控制 |

## 语义层的 5 个工程对象

| 术语 | 它是什么 | 它不是什么 | OmniSupport 例子 |
| :--- | :--- | :--- | :--- |
| **semantic model** | 实体 + 维度 + 度量的语义结构 | 不是 dashboard，也不是单条 SQL | support_case / ticket_activity |
| **entity** | 业务里"能被连接的对象"的主键 | 不是所有字段 | case_id / customer_id / org_id |
| **dimension** | 用来切分/过滤的属性 | 不是任意 where 条件 | product_line / priority / category |
| **measure** | 可聚合的基础值 | 不是最终业务指标 | ticket_count / sla_breach_count |
| **metric** | 面向业务消费的最终指标 | 不是 SQL 片段 | p1_ticket_count / sla_breach_rate |
| **semantic graph** | 语义对象之间的关系 | 不是漂亮图 | case → customer → activity |

## 四方案对比

挑方案的标准不是"先进"，是五维都过得了。

| 方案 | 本地可跑 | 语义一致性 | 工具边界 | 迁移性 | 运维成本 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| dashboard SQL only | 高 | 低 | 低 | 低 | 低 |
| dbt marts only | 高 | 中 | 中 | 中 | 中 |
| **local metric registry（本周）** | **高** | **高** | **高** | **高** | **中** |
| Managed SL / MetricFlow | 看部署 | 高 | 高 | 中（看 OSI） | 高 |

**本周选 local registry**：先把"可迁移的语义内核"立住；将来要上 dbt SL / Cube，再做 mapping。

## metric_registry_v1.yml：runtime contract

Tool runtime 会按这份 YAML 校验每一次请求——它就是 Agent 与口径之间的"合法清单"。

```yaml
# analytics/metric_registry_v1.yml （节选）
version: 1
registry_id: week05_support_metrics_v1
registry_version: 1.1.0
owner: analytics_engineering
source_model: support_kpi_mart
safe_view: agent_tool_input_view
time_dimension: metric_date
measure_column: metric_value
max_window_days: 31
allowed_dimensions: [product_line, priority, org_id, category]
allowed_filters: [product_line, priority, org_id, category, data_release_id]
allowed_roles: [support_ops, instructor, admin]

metrics:
  - name: p1_ticket_count
    label: "P1 Ticket Count"
    business_name_zh: "P1 工单量"
    business_definition_zh: "priority 属于 p1 或 p1_critical 的工单数量。"
    owner: support_ops
    metric_type: count
    formula: "sum(case when is_p1 then 1 else 0 end)"
    aggregation: sum
    unit: tickets
    sensitivity: low
    definition_status: production
    version: 1.1.0
    allowed_roles: [support_ops, instructor, admin]
    quality_tests: [not_null, non_negative]
```

## Registry 字段说明

每个字段都是 runtime 决策点，少一个字段 runtime 就缺一份判断：

| 字段 | 回答的问题 | 缺失后果 |
| :--- | :--- | :--- |
| metric_name | 请求哪个指标 | unknown metric 漂移 |
| owner | 出问题谁负责 | 事故无人认领 |
| source_model / safe_view | 从哪个 mart / 哪个安全视图 | 下游猜表 / 越权读 PII |
| grain / time_dimension | 一行是什么 / 时间字段 | 时间错；重复/漏计 |
| allowed_dimensions / filters | 能按什么切 / 过滤 | 维度越权；任意 where |
| allowed_roles | 哪些角色能查 | 越权访问 |
| max_window_days | 能查多久 | 成本爆 / 滥用 |
| sensitivity / definition_status | 敏感级 / 是不是实验代理 | 高敏感指标无显式确认 |
| quality_tests | 哪些测试在保护它 | 不可验收 |

## v1.1 EXPERIMENTAL EXTENSION

从"可查询"推进到"可解释、可审计、可测试"——加字段不破，旧调用全兼容。

**新增指标（5 个）**：

- avg_first_response_minutes
- avg_handle_time_minutes
- first_resolution_rate（experimental）
- escalation_rate
- sla_breach_rate

**registry 新增字段**：owner / unit / metric_type / formula / sensitivity / definition_status / version / quality_tests

**tool contract 新增输入**：trace_id / purpose / actor_org_ids / include_experimental_metrics

**runtime 新增策略**：audit_id / policy_applied / data_freshness / org_scope_filter / experimental_metric_ack

### experimental_proxy：工程师的"诚实"

`first_resolution_rate` 当前用"已解决且未升级 / 已解决"作为代理；真实生产需接入 reopen 事件后重定义。registry 标 `experimental_proxy`，runtime 强制 `include_experimental_metrics=true` 才放行。

## HOLD IS NORMAL：不开放也是工程能力

| 原因 | 说明 |
| :--- | :--- |
| **未通过关键 tests** | ratio 范围 / not_null / unit test 没过。开放它 = 高自信地错 |
| **维度未授权** | dimension 可能暴露敏感边界（如 assignee_id 涉及员工绩效），legal/security 没签字前不开 |
| **窗口太大** | max_window_days 还没定，怕扫表。先用 31 天试运行 |
| **PII / raw body 风险** | safe view 没守住或 no_pii test 没加，直接 hold 工具入口 |
| **grain 不稳定** | 一行还说不清；upstream 还有漂移，等 mart 稳定后再开 |
| **owner 不明确** | 没有人能解释这个指标，找到 owner 再上线 |

## 五步迁移路径

不是反平台，是把"可迁移的语义内核"先立住——避免被任何 SaaS 绑死。

1. **local registry**：本周的事实来源，本地 YAML + validator
2. **semantic model draft**：写成 MetricFlow YAML，entity / measure / metric
3. **MetricFlow validate**：本地 CLI 校验语义，回归测试
4. **managed SL / OSI**：dbt Cloud / Cube 或按 OSI 标准导出
5. **BI/Agent/Eval 共享**：下游统一消费，同一口径

**迁移三原则**：

1. 任何阶段都不要扔掉五个核心字段：metric_name / grain / allowed_dimensions / allowed_filters / allowed_roles + max_window_days
2. 过渡期保持 registry 与 semantic model 双写一段时间——避免一刀切失败
3. 监控三件事：迁移后口径数字一致性（指标级 diff）、下游延迟变化、SaaS 成本曲线

## OSI v1.0：跨厂商语义交换标准

- **是什么**：Open Semantic Interchange，2026.01 v1.0 / Apache 2.0
- **谁签了**：dbt Labs、Cube、Atlan、Snowflake 等
- **定义 4 件事**：
  1. 指标的形状（name / formula / unit）
  2. 维度与实体关系
  3. 访问策略（allowed_roles / allowed_filters）
  4. 与 AI 工具的交互（含 MCP 兼容）

你的 registry 字段已经 90% 与 OSI 对齐——今天的投入未来跨平台几乎无痛。
