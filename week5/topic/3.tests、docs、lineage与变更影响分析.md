# 3. tests、docs、lineage 与变更影响分析

## 核心目标与背景

- **定调**：能跑 ≠ 能交付。SQL 改对了，系统也能答错——这是 80% 团队的真实剖面。
- **主要任务**：让指标能"被人和 Agent 共同信任"，要的不是更炫的 SQL，是证据链。
- **本课交付**：
  - 一份 `schema.yml`（不是占位描述，是写给消费者的"接口说明"）
  - 一份自定义 PII 测试（`tests/no_pii_columns_*.sql`）
  - 一份 `dbt_build_evidence.md`（可被 reviewer 在 3 分钟内复核）
  - 一份 `lineage-impact-notes.md`（变更影响清单）

## WAR STORY："reopen 算不算"的半年公案

某物流公司，客户支持团队的"工单数"指标，运营说 +18%，产品说 +6%。复盘发现：差异全在"已关闭后重新打开（reopen）算不算一次新工单"。

三个问题没人能答：

1. 这个口径的 owner 是谁？
2. docs 写在哪？
3. 变更过哪几次？

SQL 是 2017 年的某前同事写的，文档没有，人离职了。

**真正坏点**：不是 SQL 写错——SQL 一直在跑。是没人能解释、没人能追责、没人能复盘。这就是"有指标无证据链"的代价：每次出现争议，整个组织停摆一周。

**判断**：能跑的 mart 不该被消费——除非它带着证据进系统。

## KEY THESIS：负责的口径必须带证据

四件凑齐才叫交付，少一件事故就以那件的形状出现：

| 证据 | 证明什么 | 具体内容 |
| :--- | :--- | :--- |
| **tests** | "明显破坏的事情没发生" | not_null / unique / accepted_values + 自定义 SQL + unit test |
| **docs** | "未来的消费者读得懂" | grain + 字段含义 + 时间语义 + 消费者清单 |
| **lineage** | "变更影响可定位" | manifest.json + DAG = 谁依赖谁 |
| **artifacts** | "交付可验收" | manifest / catalog / run_results / sources 四份 |

## TESTING：dbt 的四层测试

| 类别 | 怎么写 | 保护什么 | Week05 示例 |
| :--- | :--- | :--- | :--- |
| **Generic（内置通用）** | schema.yml 字段下挂 not_null / unique / accepted_values / relationships | 主键、关键字段、枚举漂移、外键完整 | ticket_id 唯一 / metric_name 在白名单 |
| **Singular（自定义 SQL）** | tests/ 写 SELECT，返回任何一行 = 失败 | 复杂跨字段断言、安全边界 | no_pii_columns / ratio_between_0_and_1 |
| **Unit（dbt 1.8+）** | unit_tests 给输入打桩，验证 SQL 逻辑 | 复杂 case-when / 升级判定 / reopen 规则 | "reopen 算不算"这种逻辑回归 |
| **Source Freshness** | sources.yml 配 loaded_at_field | "上游断流"工程层发现 | ticket_fact 24h 内必须有新数据 |

**真正保护口径的是"业务断言 + 自定义 + unit test"，不是 not_null 的数量。**

## Week05 落地的三个业务断言

### PII 安全测试

```sql
-- analytics/tests/no_pii_columns_in_agent_tool_input_view.sql
{{ config(tags=['week05']) }}
select column_name
from information_schema.columns
where table_schema = '{{ target.schema }}'
  and table_name = 'agent_tool_input_view'
  and column_name in (
      'contact_email','subject','body','body_preview','customer_id'
  )
-- 任何一行 = 安全视图泄露了 PII 字段 → 红灯
```

### 比率边界测试

```sql
-- analytics/tests/ratio_metrics_between_0_and_1.sql
{{ config(tags=['week05']) }}
select *
from {{ ref('support_kpi_mart') }}
where metric_name in (
    'first_resolution_rate','escalation_rate','sla_breach_rate'
)
and (metric_value < 0 or metric_value > 1)
-- 越界几乎肯定是分子分母搞反
```

### 非负值测试

`metric_values_non_negative.sql` —— count 类指标不能出现负值，这是口径完整性的底线。

## DOCS：写给"未来的消费者"

docs 不是负担，是消费者接口。

```yaml
# analytics/models/marts/schema.yml （节选）
models:
  - name: support_kpi_mart
    description: "Controlled metric rows by date and safe business dimensions."
    columns:
      - name: metric_date
        description: "It comes from created_date in stg_tickets, not from updated_at..."
        tests: [not_null]
      - name: metric_name
        description: "Whitelisted metric identifier (see metric_registry_v1.yml)."
        tests:
          - not_null
          - accepted_values:
              values: ['ticket_count','p1_ticket_count','sla_breach_count', ...]
      - name: metric_value
        description: "Non-negative for count; 0..1 for ratio."
        tests: [not_null]
      - name: data_release_id
        description: "Required for reproducibility and audit."
        tests: [not_null]
```

**必写 4 件事**：

1. model description（一行 grain）
2. 字段含义 + 时间语义
3. 边界（不能怎么用）
4. 消费者清单

**第一性问题**："消费者算错了，他能不能从文档里看出来？" 答不出"能"就还要补。

## LINEAGE：变更影响入口

看 DAG 永远要追问：改了它，谁会因为我而变？

**依赖链全景**：

```
上游 source          → staging / intermediate           → marts                    → 下游消费方
ticket_fact            stg_tickets                        support_case_mart          BI 看板
customer_dim           int_support_cases                  support_kpi_mart           Agent tool runtime
comment_fact           int_ticket_activity_daily           agent_tool_input_view      Eval Week08
                                                                                     Governance
```

**变更影响对照表**：

| 你改了什么 | 下游可能受谁影响 | 必须同步更新的工件 |
| :--- | :--- | :--- |
| stg_tickets.is_p1 定义 | 所有用 priority / p1_ticket_count 的 mart 与 dashboard | schema.yml / docs / registry / tool examples |
| int_support_cases.first_response_minutes | avg_first_response_minutes / SLA 看板 / Agent 工具 | tests / docs / registry / impact note |
| support_kpi_mart 加新指标 | agent_tool_input_view / registry / tool contract | safe view + accepted_values test + registry validator |

## ARTIFACTS：dbt 的 4 份"事实证据"

| artifact | 位置 | 记录了什么 | Week05 怎么用 |
| :--- | :--- | :--- | :--- |
| **manifest.json** | analytics/target/ | 所有节点、依赖、tests、资源元数据 | 抽 lineage 做影响分析；CI 比对前后差异 |
| **catalog.json** | analytics/target/ | 仓库里真实的 schema、列、类型 | 回填到 dbt docs 网站；比对模型与仓库 |
| **run_results.json** | analytics/target/ | 每次 run/build/test 的状态、耗时、错误 | dbt_build_evidence.md 的事实来源 |
| **sources.json** | analytics/target/ | source freshness 检查结果 | 上游断流的工程预警 |

**工程纪律：交付物里不允许只有"build 通过"截图——要带这 4 个 JSON 的关键摘录。**

## METRIC_READY GATE：7 道闸

任一缺失 = HOLD。HOLD 是工程纪律，不是失败。

| 闸口 | 为什么必须有 | 没过该做什么 |
| :--- | :--- | :--- |
| **build pass** | 模型可以复现 | 先修模型；不要 patch 测试 |
| **tests pass** | 关键断言成立 | 修业务逻辑；不要 disable test |
| **docs complete** | 消费者能读懂 | HOLD，补字段 description |
| **lineage clear** | 能定位变更影响范围 | HOLD，补 impact note |
| **registry entry** | runtime 有事实来源 | 补 metric_registry_v1.yml 条目 |
| **safe view available** | 工具拿到的字段是安全的 | 修 safe view + 加 no_pii test |
| **audit fields ready** | 出错能复盘 | 补输出契约 + 审计字段 |

## ANTI-PATTERNS：6 类直接拒签

| 反模式 | 表现 | 为什么拒 |
| :--- | :--- | :--- |
| **build screenshot only** | 只贴 dbt build 成功截图 | 没有命令、没有选中模型、没有 test 数量 |
| **empty docs** | schema.yml 只有字段名 | 没有 description、没有时间语义说明 |
| **DAG screenshot only** | 只贴漂亮 lineage 图 | 没有"改了 X 谁会受影响"的影响清单 |
| **tests 全是 not_null** | 堆 not_null 凑数 | 没有 ratio 边界、PII 测试、业务断言 |
| **registry stale** | mart 改了，registry 没同步 | Agent 用旧白名单"成功"拿到错数据 |
| **safe view 无 PII test** | view 没配 PII 边界测试 | 下次新人忘了配置就泄露了 |

## CI/CD：把证据链放进流水线

```
① compile    dbt parse + dbt compile           → 语法 / Jinja / 依赖完整
② build      dbt build --select state:modified+ → 只跑改动 + 下游
③ docs       dbt docs generate                  → 上传 manifest/catalog/run_results
④ registry   validate_metric_registry.py        → pytest tests/contract tests/integration
```

**证据链如果只"这次做了"，下次还会漂；放进 CI 才是真正工程化。**

## RECAP：7 个判断

1. build pass ≠ metric ready；要带证据链 tests / docs / lineage / artifacts。
2. tests 是口径断言：generic / singular / unit / freshness 四层都不能少。
3. docs 是写给未来消费者的接口；grain + 字段含义 + 时间语义 + 消费者。
4. lineage 不是装饰图，是"改了 X 谁会受影响"的工程入口。
5. manifest / catalog / run_results / sources 是 4 份验收证据，不要只截图。
6. metric_ready 7 道闸：任一缺失 = HOLD；HOLD 比事故便宜十倍。
7. 下一讲：把"能负责的口径"立成"语义契约"——本地 registry。
