# 导学 01：为什么"数字对不上"不是 SQL 的错？

## 一句话理解

指标不只是一条查询语句的结果——它是一份"谁算、怎么算、谁能看、改了怎么办"的完整协议。

## 用生活打比方

这一周我们用"中央厨房给连锁餐厅供餐"来串起所有概念。

想象你经营三家连锁餐厅，老板问"上周卖了多少份招牌菜"：

- A 店按"下单时间"算，说 120 份
- B 店按"出餐时间"算，说 98 份
- C 店把退餐也算进去了，说 135 份

三个数都"对"，但老板只想要一个答案。问题不在各店的收银机坏了，而在于没人事先约定"用哪个时间算、退餐算不算、谁说了算、数据记在哪"。

本节课要做的，就是给这家连锁写一份**统一菜谱**——不是教厨师怎么炒菜，而是先把"一份招牌菜到底怎么定义"写清楚：用料标准、负责人、谁能看配方、改了要通知谁。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **指标（metric）** | 一个业务上常用的数字，比如"本周紧急工单数" |
| **业务口径（definition）** | 这个数字到底怎么算的——用哪个时间字段、包不包含某类数据 |
| **工程接口（interface）** | 把口径变成一份"合同"：谁负责维护、谁能查、查出来的格式是什么、改了要通知谁 |
| **grain（粒度）** | 数据表里"一行代表什么"——比如一行 = 一天的一个产品线的一个指标 |
| **owner（负责人）** | 这个指标出了问题，找谁解释和修复 |
| **指标接口卡** | 一张 16 个字段的清单，把上面所有信息写在一起，少填一项就是隐患 |

## 避坑提醒

- **以为指标就是 SQL**：SQL 只是计算方式的一种表达，缺了负责人、粒度、权限、审计这些信息，它就不是一个合格的指标。
- **以为看板上显示了就够了**：看板、AI 工具、评测系统、审计团队都需要同一份口径，只在看板里对齐远远不够。
- **以为 AI 会自己查对**：AI 擅长写出"能运行"的查询，但它分不清该用哪个时间字段、该不该包含历史数据——给它自由就是给它制造事故的机会。
- **以为改了 SQL 通知一下就行**：没有依赖图和文档，改动的影响范围全靠猜，旧的错误可能被用半年都没人发现。

## 一句话总结

> **不是"查出一个数"就叫指标，而是"多方都认同这个数怎么来的"才叫指标。**
