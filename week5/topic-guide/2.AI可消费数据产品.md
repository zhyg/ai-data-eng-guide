# 导学 02：为什么数据要"分层"放？

## 一句话理解

把数据加工过程分成四层，不是为了好看，而是为了"上游改了不至于全盘崩"。

## 用生活打比方

继续我们的连锁餐厅。上节课统一了菜谱，这节课要把**中央厨房的分区**搭起来：

- **收货区（Sources）**：只负责签收食材，不洗不切。原样记录"今天从哪家供应商收了什么"。
- **粗加工间（Staging）**：洗菜、切丝、去皮——把食材整理成统一规格，但绝不在这里炒菜。
- **配菜间（Intermediate）**：把不同食材组合在一起（肉 + 菜 + 调料），为出锅做准备，但这里的半成品不直接端上桌。
- **出餐口（Marts）**：端给客人的成品——每道菜的配方固定、分量明确、不该放的食材绝不混进去。

如果收货区直接往锅里倒，供应商换了个品种，三家餐厅同时出事。分区就是在每个环节画清楚线，把变化控制在最小范围内。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **dbt** | 一个让你"像写程序一样写数据加工"的工具。它不存数据，只生成加工指令 |
| **Sources（源）** | 声明"我的数据从哪来"。越简单越好，不做任何加工 |
| **Staging（整形层）** | 把原始数据"洗干净"：统一大小写、转换日期格式、标记是/否。但绝不做汇总计算 |
| **Intermediate（组合层）** | 把多张表拼在一起，算出一些中间事实（如"首次响应时长"）。这层是内部厨房，不直接对外 |
| **Marts（交付层）** | 最终产品：给看板用的、给 AI 用的、给评测用的。每张表必须说清"一行是什么、谁能用、不能暴露什么" |
| **安全视图（safe view）** | 专门给 AI 查指标用的一张"白名单"视图——只暴露允许查询的指标，AI 碰不到其他数据 |
| **物化（materialization）** | 数据以什么形式存：view（每次查询都是最新的，成本低）、table（稳定快照，适合频繁读取）。选哪个要看场景 |

## 避坑提醒

- **以为分层就是建几个文件夹**：文件夹叫什么名字不重要，重要的是每一层只做该做的事——整形层里写了汇总计算？那就是把"事故种子"埋在了最深处。
- **以为交付层字段越多越好**：恰恰相反，好的交付层是"刚好够用"。多放一个敏感字段，下游所有消费者都可能拿到不该看的数据。
- **以为数据量大就要用增量更新**：小数据用最简单的方式就够了，增量更新是留给真正大表的优化手段。
- **以为 dbt 是数据库**：dbt 不存任何数据，它只负责告诉数据库"怎么加工"。数据还是存在你原来的数据库里。

## 一句话总结

> **分层不是为了整齐，而是为了"上游变了，下游不慌"。**
