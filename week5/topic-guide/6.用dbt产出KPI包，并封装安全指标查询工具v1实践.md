## 用生活打比方

继续我们的连锁餐厅。前五节课分别搞定了统一菜谱、厨房分区、质检验收、配方手册和点菜系统。这一节是**开业试运营**，把所有环节串起来跑一遍，确认从收货到出餐再到点菜的整条链路都通。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **docker compose** | 用一条命令启动整个实验环境（数据库、存储等），不用手动一个个装 |
| **dbt debug** | 检查 dbt 和数据库之间的连接是否正常，相当于上路前检查车辆 |
| **dbt build** | 一键运行所有数据模型并执行测试，相当于把整条流水线跑一遍。同时自动生成构建证据文件 |
| **dbt docs generate** | 自动生成文档网站，供团队浏览数据结构和依赖关系 |
| **registry validator** | 校验指标注册表是否完整、格式是否正确 |
| **正负例测试** | 正例：发一个合法请求，预期返回正确数据；负例：发一个非法请求（比如不存在的指标），预期被明确拒绝 |
| **验收 Checklist** | 8 项检查全部通过才算交付完成 |

## 实践地图

整个实践分三大步，对应前五节的所有产出：

```
第一步：环境就绪
  启动数据库 → 检查连接

第二步：数据构建
  dbt build（源 → 整形 → 组合 → 交付） → 生成文档和证据

第三步：工具验证
  注册表校验 → 正例查询 → 负例拒绝 → API 端点 → 回归测试
```

## 避坑提醒

- **跳过连接检查直接 build**：如果数据库连不上，后面全白跑。先看到"连接成功"再往下。
- **build 报错就慌**：先看是哪个模型、哪个测试挂了。大多数问题出在配置或上游数据不完整，不是你写的代码有 bug。
- **只跑正例不跑负例**：工具的价值一半在"能正确返回"，另一半在"能正确拒绝"。负例测试通过才说明安全边界生效。
- **觉得跑通一次就够了**：真实项目里，这些检查会放进自动化流水线，每次改动自动跑。手动跑通是为了理解过程，最终目标是自动化。
