# 导学 04：为什么不急着买平台，先自己建清单？

## 一句话理解

语义层要解决的问题是"同一个指标名字在公司里不会变意思"，这件事用一份本地清单就能先做起来，不用急着上平台。

## 用生活打比方

继续我们的连锁餐厅。菜品质检都过了，但三家门店的"招牌菜"味道还是不一样，因为每家店的厨师对"少许盐""适量糖"的理解不同。你需要一份**配方手册**。

有两种做法：

- **方案 A**：先花大钱买一套智能配料系统（平台），但每道菜连"一勺是多少克""鸡汤用什么牌子"都还没统一。买了系统也是白搭。
- **方案 B**：先写一份《配方手册》，把每道菜的原料名称、克数、负责人、谁能看配方全部写清楚。三家店照着做，味道先一致了。将来要上智能系统，直接把手册导入就行。

本节课选的就是方案 B，先写手册（本地注册表），等手册验证 OK 了，再考虑要不要上平台。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **语义层** | 一套"统一翻译"机制，让看板、AI、评测系统查同一个指标时，背后的计算方式完全一致 |
| **registry（指标注册表）** | 一份清单文件，记录了每个指标的名字、计算方式、谁能查、能查多久等信息。相当于连锁餐厅的"配方手册" |
| **metric（指标）** | 面向业务的最终数字，比如"P1 工单数" |
| **measure（度量）** | 可以被汇总的基础数值，比如"工单计数"。多个度量组合成一个指标 |
| **dimension（维度）** | 用来切分数据的角度，比如"按产品线看""按优先级看" |
| **entity（实体）** | 业务里的核心对象，比如"一个工单""一个客户"，它们之间可以关联 |
| **experimental_proxy** | 标记为"实验性"的指标，承认它目前的计算方式是近似的，还不够精确，需要使用者明确确认后才能查 |

## 避坑提醒

- **以为买了平台就有了语义层**：平台只是工具，如果公司里"同一个名字不同含义"的问题还没解决，平台救不了你。
- **以为所有能算的数字都要注册**：如果你说不清一个指标的粒度、负责人和消费者，先别注册。"暂不开放"比偷偷上线稳妥得多。
- **以为"实验性"就是"不靠谱"**：把还不够精确的指标诚实地标为"实验性"，比偷偷上线强一万倍。使用者知道它有局限，才不会拿它做关键决策。
- **以为本地清单以后要全部推翻**：好的本地清单（registry）里的核心字段，和行业标准（OSI v1.0）已经 90% 对齐。将来迁移到任何平台，几乎不用重写。

## 一句话总结

> **先把"每个指标名字在公司里只有一个意思"这件事做到位，平台的事后面再说。**
