# 导学 03：怎么证明你的数据"靠得住"？

## 一句话理解

数据跑得通不等于交付得了，你还需要测试、文档、依赖图和构建证据，少了哪个都不踏实。

## 用生活打比方

继续我们的连锁餐厅。中央厨房分好区了，菜也炒出来了，但在端给客人之前，你还需要过一道**质检验收**：

- **试吃检查（测试）**：盐度达标吗？过敏原有没有混入？分量够不够？
- **出品说明（文档）**：这道菜用了什么原料、保质期多久、不适合哪些客人，写给服务员和下一个接班厨师看的。
- **原料溯源图（依赖图）**：如果供应商换了酱油品牌，顺着这张图就能找到所有受影响的菜品。
- **生产记录（构建证据）**：今天谁做的、几点出锅、用的哪批原料，出了食品安全问题，有据可查。

"炒得出来"只是第一步，四样证据齐了才敢端上桌。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **tests（测试）** | 自动检查数据有没有明显问题：必填字段有没有空值、比率有没有超出 0~1 的范围、隐私字段有没有混进安全视图 |
| **docs（文档）** | 写给"下一个用你数据的人"看的说明：这张表一行代表什么、时间字段是哪个、不能怎么用 |
| **lineage（依赖图）** | 一张"谁依赖谁"的地图。改了上游某个字段，顺着这张图就能找到所有受影响的下游 |
| **artifacts（构建证据）** | 每次构建自动生成的 4 份记录文件，相当于"施工日志"，用于验收和事后复盘 |
| **metric_ready 7 道闸** | 指标上线前必须通过的 7 项检查：构建通过、测试通过、文档齐全、依赖清晰、注册完成、安全视图就绪、审计字段齐全 |

## 避坑提醒

- **以为构建成功就是交付成功**：构建通过只说明"没报错"，不说明"数字对"。真正的交付要过 7 道闸。
- **以为堆一堆"非空检查"就叫测试**：最有价值的测试是业务层面的，比如比率不能超过 1、隐私字段不能出现在安全视图里、数量不能是负数。
- **以为贴一张依赖图就叫 lineage**：依赖图的价值在于能回答"我改了 X，谁会受影响"。光有图不写影响清单，跟没有差不多。
- **以为文档是额外负担**：好的文档能让新同事 3 分钟看懂一张表的用法。省掉文档的代价是：每次有人用错你的数据，你要花半天解释。
- **以为只截一张"build 通过"的图就行**：截图不能被验证、不能被追溯。要交的是构建记录文件本身。

## 一句话总结

> **"能跑"只是及格线，带着证据交付才算工程标准。**
