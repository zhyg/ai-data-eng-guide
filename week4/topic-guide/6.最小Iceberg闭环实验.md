# 导学 06：把前 5 节学的全串起来，做一次完整演练

## 一句话理解

把前面学的全部串起来，做一次别人能照着复现的实验。

## 你要达成的 7 件事

1. 起一套本地数据环境
2. 建好账本骨架（命名空间 + 4 张表，先不写数据）
3. 把第 3 周存好的源表物化进 Iceberg
4. 看完整的状态证据链（快照、历史、文件、元数据日志）
5. 跑一次时间旅行，按快照编号读到旧版本
6. 跑一次字段演进，给青铜表加一列，前后对照 schema
7. 输出一份基线报告

## 七个步骤逐个说

### 第 1 步：起环境
把数据库（PostgreSQL）、对象存储（MinIO）一起拉起来。
**怎么算通过**：能打开 MinIO 控制台，能连上数据库，桶（bucket）已经建好。
**最容易踩**：PostgreSQL 和 MinIO 都在 DevBox 容器里跑，不在你本机的 localhost；连接地址要写 Docker Compose 里的服务名，不是 127.0.0.1。

### 第 2 步：建骨架
先告诉账本要建哪 4 张表，把命名空间（bronze / silver）和 4 张表的形状建出来，里面还没数据。
**怎么算通过**：列出表名能看到 4 张（青铜 2 张 + 白银 2 张）；MinIO 的 warehouse 目录下能看到 bronze.db / silver.db；PostgreSQL 里能查到 namespace 和当前的 metadata 指针（账本只记指针，数据和元数据本体都在 MinIO 里）。

### 第 3 步：物化数据
把第 3 周存在 PostgreSQL 里的工单和文档读出来，转成 Arrow，再以 overwrite 写进对应的 Iceberg 表。
先 dry-run 一次，确认每张表能读出多少行，再真的写。
**怎么算通过**：真实写入的回执里至少包含 release_id、batch_id、source_rows、write_mode、action；同一批可重复跑，行数对得上。

### 第 4 步：看状态证据链
按这个顺序看：

```
快照 → 历史 → 文件 → 元数据 → 基线
```

每一层告诉你不同的事：

| 视图 | 告诉你什么 |
| :--- | :--- |
| snapshots | 一共有几个快照、每个是什么操作、父快照是谁 |
| history | 谁在什么时候成为当前版本 |
| files | 现在用了哪些文件、多大、怎么分布 |
| metadata_log | 说明书是怎么一步步演进的（通常比数据本身演进得快） |
| baseline | 把这些结果整理成可复现的报告 |

再写一小批进去，快照数从 N 变成 N+1，序列号和提交时间跟着前进。

### 第 5 步：时间旅行
挑一张白银表（比如 ticket_fact），按快照编号读历史版本。
**怎么算通过**：能输出 snapshot_count 和 selected_snapshot_id；当前行数和指定快照的行数对得上。

### 第 6 步：加一列
挑一张青铜表，加一列新字段，前后对照 schema：
- 新数据带新字段
- 旧快照仍然能按旧结构读出来

**怎么算通过**：加列前后两份 schema 都留下来，旧数据不用回填。

### 第 7 步：写基线报告
按固定模板写，至少包含：
- 环境说明
- 4 张表的状态数字（行数、快照数、文件数）
- 异常现象的解释
- 时间旅行和加列演示的记录

## 验收清单（每项都要打勾）

## 常见踩坑

| 现象 | 先看哪里 |
| :--- | :--- |
| 账本加载失败 | 配置、环境变量，URI 是不是 PostgreSQL，warehouse 是不是 s3 开头 |
| MinIO 连不上 | 是不是把容器内服务名写成了 localhost |
| 桶不存在 | 路径和权限 |
| 字段不匹配 | 回去对照 schema 定义文件 |
| 重跑数据重复 | overwrite 是不是真的执行了，不是停在 dry-run |
| 看不到快照 | 写入是否真的执行了（不是 dry-run） |
| 旧状态读不到 | 是不是太早清掉了 |

跑完这条链路，数据底座的状态留得住、回得去、说得清。下一周进入数据加工和语义层。
