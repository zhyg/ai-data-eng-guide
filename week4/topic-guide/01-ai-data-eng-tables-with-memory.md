# 导学 01：为什么数据要"记得住"？

## 一句话理解

光能查到"现在的数据"还不够，我们还要能找回"那天的数据"。

## 这一周和上一周的关系

- 上一周（Week03）解决的是：**数据进得对不对**（接入正确性）。
- 这一周（Week04）解决的是：**数据状态记不记得住**（状态可复现）。


## 用生活打比方

想象你在管理一个图书馆：

- **能查**：读者问《三体》在哪，你能马上找到 → 这是普通数据库做的事。
- **能回看**：读者问"上周二我借的那本书是第几版？" → 这就需要一本"账本"，记录每一次变动。

数据工程里的"账本"，就是这节课要讲的主角。

## 真实痛点

同一个问题，AI 助手周一回答 A，周三却回答 B。团队想查原因，却发现：

- 不知道原始文档有没有被改过
- 不知道数据是哪一批进来的
- 不知道 AI 用的是哪一版资料

最后只能"猜"。这就是没有"记忆"的代价。

## 四种数据角色（记住它们的分工）

```diagram
╭─────────────╮   ╭─────────────╮   ╭─────────────╮   ╭─────────────╮
│  文件柜     │   │  业务视图   │   │   索引      │   │  数据账本   │
│ (原始文件) │   │ (当前事实) │   │ (快速查找) │   │ (历史记录) │
├─────────────┤   ├─────────────┤   ├─────────────┤   ├─────────────┤
│ 保存原件    │   │ 看现在      │   │ 帮你找得快  │   │ 记得每一次  │
│ 不管查询快慢│   │ 不管历史    │   │ 不管是哪版  │   │ 改动和状态  │
╰─────────────╯   ╰─────────────╯   ╰─────────────╯   ╰─────────────╯
```

前三个角色都做不到"回看历史"，所以我们需要第四个——**数据账本**（课程里叫 Iceberg）。

## 复盘出问题时，要会问这 8 个问题

1. 用的是哪一版原始文档？
2. 是哪一批数据进来的？
3. 当时数据处于什么状态？
4. AI 查的索引是哪一版？
5. 当时给 AI 的提示语变了吗？
6. 分数下降是从什么时候开始的？
7. 字段或权限有没有调整？
8. 能不能回到之前那个好用的状态？

记不住这些细节，就没法定位问题。

## 几个容易踩的坑

- ❌ 以为"这事跟 AI 没关系"——AI 的回答质量恰恰依赖数据状态。
- ❌ 以为"把文件丢进云存储就行了"——没有账本就没有记忆。
- ❌ 以为"有了向量库就能复现答案"——向量库只管召回，不记得"是哪一版"。
- ❌ 以为"账本能回到过去就是把表复制了一份"——它只是翻回旧的那一页，不是抄一遍。
- ❌ 以为"建完账本，治理就做完了"——权限、发布、回滚这些后面几周才讲。

## 账本能做什么 / 不能做什么

容易把账本想得太神奇，先划清边界：

| 账本**能**做 | 账本**不能**做 |
| :--- | :--- |
| 记住每一版数据长什么样 | 自动把脏数据洗干净 |
| 在保留范围内回看旧状态 | 自动让 AI 回答变准 |
| 给评测、回滚提供锚点 | 替代准入检查（上周做的事） |
| 让"是哪一版"有据可查 | 自动决定旧版本什么时候清 |

一句话：**账本只负责"记得清"，不负责"修得好"。**

## 一句话总结

> **"用了新工具"不算成功，"能找到当时那一版数据"才算成功。**

下一节课，我们会拆开这本"账本"，看看它内部是怎么工作的。
