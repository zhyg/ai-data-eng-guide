# 导学 02：拆开"数据账本"，看里面装了什么

## 一句话理解

"回到过去"不是把旧表复制一份，而是翻到账本的某一页，看那时候的样子。

## 用生活打比方

把 Iceberg 想象成你写过的日记本：

- 每天写一页 → 每次写入数据，账本就多记一条
- 想看上个月某天的状态 → 只要那页还没被清掉，翻回去就行，不用把旧日记重抄一遍

这就叫"时间旅行"。

## 借用 Git 来理解

如果你用过 Git 提交代码，下面这个对照表会很顺：

| Git 里的概念 | 账本里的对应 | 它在做什么 |
| :--- | :--- | :--- |
| 每次 commit | 一个 snapshot（快照） | 记下"这一刻表长什么样" |
| 提交历史 | history | 这些快照是按什么顺序来的 |
| 文件清单 | manifest（清单） | 这一版到底用了哪些文件 |
| 工作区文件 | data files（数据文件） | 真正存数据的文件 |
| 切回旧 commit | 切回旧 snapshot | 看以前的状态，不是改回去 |

## 账本是分层记录的

```diagram
╭──────────────────────────╮
│ metadata（说明书）       │  ← 这张表现在长什么样
╰──────────┬───────────────╯
           ▼
╭──────────────────────────╮
│ snapshot（快照）         │  ← 每次改动都拍一张"全景照"
╰──────────┬───────────────╯
           ▼
╭──────────────────────────╮
│ manifest list（清单总目录）│  ← 这一版的全部清单都在哪
╰──────────┬───────────────╯
           ▼
╭──────────────────────────╮
│ manifest（清单）         │  ← 每张清单管一批文件 + 统计
╰──────────┬───────────────╯
           ▼
╭──────────────────────────╮
│ data file（数据文件）    │  ← 真正的数据
╰──────────────────────────╯
```

记住：**数据文件本身不等于"表"**，要靠上面几层把它们组织起来才算。

另外还有两本"索引册"也很重要：

- **history（历史）**：记录哪个快照在什么时候变成了当前版本。
- **metadata log（说明书日志）**：记录说明书是怎么一步步变化的。

复盘问题时，不能只看数据文件，也要看这些记录。

## 为什么读写都很稳？

- **读不会读到一半的数据**：每个人都指向一个完整的快照
- **写失败也不会污染表**：要么整体成功，要么当作没发生
- **改字段不用重写文件**：比如加一列，只动说明书
- **多人同时写也有规矩**：账本会检查冲突，失败的一方要按规则重试

## 出问题时，按这个顺序看

之后调试和写基线报告，都按这个顺序排查，记住这个口诀：

```
快照 → 历史 → 文件 → 元数据 → 基线
```

- **快照**：一共提交了几次、每次是什么操作
- **历史**：哪个快照在什么时候变成了"当前版本"
- **文件**：这一版到底引用了哪些文件、多大
- **元数据**：说明书是怎么一步步演进的
- **基线**：把上面这些汇总成一份成绩单

## 5 个最容易搞错的地方

1. 快照不是一个文件，而是"这一版用了哪些文件"的记录。
2. 清单不只是目录列表，还带统计信息（多少行、多大）。
3. 时间旅行不等于复制旧表，是翻回旧的那一页。
4. 字段不能随便改，本周只演示"加一列"这种安全操作。
5. 旧快照不要急着清，清早了就没法回看，要先定规矩再清。

## 一句话总结

> **账本之所以是账本，是因为它记得清"什么时候是什么样"，而不是"现在长什么样"。**
