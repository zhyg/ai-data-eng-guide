# 导学 05：先量体重，再谈减肥

## 一句话理解

没有"基线记录"，所有的优化都是凭感觉。先把现在的样子记下来，才谈得上后面变好了多少。

## 用生活打比方

减肥前你得先称一下体重、量一下三围。
如果连起点都没记录，后面瘦了 5 斤还是胖了 3 斤，根本说不清。

数据工程也是这样：先建**基线**，再谈**优化**。

## 三个常被混淆的词

| 词 | 它在做什么 | 本周做吗 |
| :--- | :--- | :--- |
| **基线（Baseline）** | 记录现在表是什么样子 | ✅ 主任务 |
| **基准测试（Benchmark）** | 跑分，看性能上限 | ❌ 本周不做 |
| **调优（Tuning）** | 动手优化（合并文件、改分区等） | ❌ 本周不做 |

**本周只做第一件事**：把现状如实记下来。

## 基线里要记录的 7 个数字

| 指标 | 它说明了什么 | 不正常的信号 |
| :--- | :--- | :--- |
| 总行数 | 表里有多少条数据 | 和数据源对不上 |
| 快照次数 | 一共提交过几次 | 太多或太少都要解释 |
| 文件数量 | 现在引用了多少个文件 | 特别碎说明小文件多 |
| 文件大小 | 平均/最小/最大 | 大小分布严重不均 |
| 分区分布 | 数据是怎么分布的 | 一边堆很多，一边没东西 |
| 最新提交时间 | 最后一次写入是什么时候 | 跟实际写入时间对不上 |
| 元数据记录数 | 说明书演进了多少次 | 说明保留策略可能要调 |

记录这些数字时，要能回答三件事：这次写入产生了哪个快照？现在引用了哪些文件？数量和源头数据对不对得上？

## "先记，不修"原则

看到不对劲的现象，**先写下来，不要急着动手修**：

| 看到什么 | 千万别 |
| :--- | :--- |
| 快照特别多 | ❌ 立刻清理（可能丢失重要历史） |
| 文件特别小 | ❌ 立刻合并（写入习惯不改，合了还会碎） |
| 行数对不上 | ✅ 这种要查，是逻辑问题 |
| 分区不均 | ❌ 立刻改分区（先记录，看趋势） |

**为什么？** 因为没记录现状就动手，等于一边瘦一边偷吃，到底有没有效果说不清。

## 几个"维护动作"，本周都做到哪一步？

经常听到的几个名词，提前划清边界：

| 动作 | 它在做什么 | 本周做到哪 |
| :--- | :--- | :--- |
| **清旧快照** | 删掉太老的版本节省空间 | ❌ 不做，先定保留多久再说 |
| **合并小文件** | 把碎文件拼成大文件 | ❌ 不做，先记现状 |
| **清理孤儿文件** | 没人引用的遗留文件清掉 | ⚠️ 只记录风险，不动手 |
| **元数据清理** | 说明书太多时整理一下 | ⚠️ 只在影响性能时才记录 |

**本周只看不动。** 动手优化是后面几周的事。

## 一份合格的基线报告，长这样

要有四块内容：

1. **环境说明**：是哪个 commit、什么时候跑的、用的什么配置
2. **表清单**：覆盖到 4 张最小表
3. **状态摘要**：上面那 7 个数字的表格
4. **异常解释**：看到的奇怪现象，写一段说明

### 两条红线（不达标就重写）

- 🚫 **只贴截图不写数字**：截图过两周自己都看不懂，必须有可比对的表格。
- 🚫 **别人在同一份代码上跑不出同样结果**：报告必须能被别人复现，否则就只是"个人感受"。


## 本周最终要交的 6 份东西

1. 为什么要有这一周（Week04 存在意义）
2. 字段映射表（源 → 表）
3. 4 张表的设计方案
4. 配置计划（账本、仓库、位置）
5. 运行手册（怎么跑、怎么排错）
6. 基线报告（最终成绩单）

## 一句话总结

> **基线不是炫技，是给未来的自己留一个"原点"。没原点，就没有进步可言。**
