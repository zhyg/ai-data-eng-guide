# 01 从「能采上来」到「可重复采集」

> 采到一次不是能力，能稳定复现才是能力。
> 这节课把 Week02 的 contract / manifest / gate 推进到 Week03 的运行时现实：state、run evidence 与恢复路径。

2. **Batch**：幂等写入、重跑与完整性校验

## 一、这节课先解决什么

### 为什么它重要(WHY IT MATTERS)

- Week02 已经定义哪些输入允许进入。
- 但「允许进入」不等于「它能稳定进入 lake」。
- 重复、缺口、错位、静默丢失会先污染下游。
- 没有 state 与 run evidence，失败后只能靠人肉回忆。

- 写出一份最小 smoke report。

### Week03 一句话

## 二、先立生产级判断

- 系统不是「采不到」才危险，而是「看似能采，却无法解释和复现」。
- 输入运行不是「脚本跑完」，而是 run 边界、state 和恢复路径都成立。
- 核心要素：**批次、状态、证据**。

### KEY THESIS

> 生产里最危险的不是采不到，而是采到了却复现不了。

- **RUN（执行可重复）**：同一批次失败后能安全再执行，而不是越跑越脏。
- **STATE（状态可持久化）**：run_id、checkpoint、cursor、watermark 不能只在日志里。
- **EVIDENCE（结果可解释）**：知道本次接了谁、写了多少、跳过什么、为什么。
- **RECOVERY（恢复有路径）**：retry、replay、backfill 不是临时救火脚本。

### 为什么 Week03 不是「再讲一次 ETL」

| 常见误解 | Week03 真正关心 | 生产后果 |
| :--- | :--- | :--- |
| 只是能读文件 | manifest / source / batch window 是否明确 | 不知道这次到底接了谁 |
| 只是写进去 | 写入语义是否幂等、可追踪、可恢复 | 重跑后重复、缺口、污染 |
| 测试通过 | run evidence 能否解释执行事实 | 失败后无人能复盘 |
| 追求实时 | 先建立可靠性基线 | 越快地传播错误 |

### 「能采上来」 vs 「可重复采集」

| 维度 | 能采上来 | 可重复采集 |
| :--- | :--- | :--- |
| 执行方式 | 脚本跑完一次 | 同一批次可安全再执行 |
| 数据边界 | 隐含在路径或参数里 | manifest / source / batch window 显式声明 |
| 失败恢复 | 重跑全量或人工补 | retry / replay / backfill 有边界 |
| 对账能力 | 看日志猜测 | total / valid / invalid / inserted / skipped 可核对 |
| 下游影响 | 污染后才发现 | gate / report 提前留下证据 |
| 复盘能力 | 靠口头经验 | run evidence 可回放、可解释 |

## 四、Week03 的 ingest baseline 是什么

1. **输入边界明确**：manifest / source / batch window
2. **执行可以重复**：rerun 不制造额外副作用
3. **状态可以持久化**：run_id / checkpoint / cursor / watermark
4. **结果可以解释**：run evidence / report
5. **恢复有路径**：retry / replay / backfill

### 5 个必须被带起来的追踪锚点

- **manifest_id**：这次运行引用哪份输入声明。
- **batch_id**：这批数据的业务/时间边界。
- **run_id**：这一次执行的唯一身份。
- **source_fingerprint**：原始来源有没有变化。
- **trace_id**：跨模块追踪与排障链路。

### 3 个状态对象：checkpoint / cursor / watermark

- **checkpoint**：状态被落在哪里——文件、表、asset metadata，还是只有日志。
- **cursor**：下一次从哪里继续读——updated_at、offset、LSN、sequence。
- **watermark**：系统已经确认处理到哪里——用于迟到、乱序和补数判断。

### REALITY CHECK

> 如果今晚 ingest 失败，第二天你最需要知道什么？

1. 哪个 manifest 在跑？
2. 跑到哪个 source？
3. 哪些已经写入？
4. 哪些未写入？
5. 下游看到什么？
6. 恢复动作是什么？

## 五、把风险讲成可复盘的事故：四类事故

**Duplicate / Gap / Mismatch / No Traceability**，是 Week03 的排障母题。

- **DUP (Duplicate)**：重放、重试、cursor 粒度不足导致重复。
- **GAP (Gap)**：窗口错位、失败未补、source 漏读。
- **DRIFT (Mismatch / Drift)**：schema 合法但业务语义变了。
- **TRACE (No Traceability)**：无法回到 manifest / run / source。

### INCIDENT A · DUPLICATE

> 重复不是小问题：它会污染统计、检索和工具动作。重复常常来自「恢复动作不带幂等」。

1. 网络中断：执行失败
2. 团队重跑：没有 batch / idempotency key
3. 重复写入：Bronze / Silver 都增加
4. 下游消费：KPI 与检索偏高
5. 复盘困难：不知道哪次写入有效

**屏幕上看起来像什么**

- 报表数值变高一点
- RAG chunk 里出现重复内容
- 工单事件看似只是多了一条

**实际已经坏在哪**

- retry 变成重复副作用
- dedupe key / idempotency key 缺位
- run evidence 无法解释哪次写入有效

### INCIDENT B · GAP

> 缺口比失败更隐蔽：系统可能继续跑，但少了一段历史。Gap 常常来自 window / cursor / checkpoint 不一致。

1. 窗口声明不清：window_end 口径不统一
2. source 漏读：一部分记录未进入
3. checkpoint 更新：状态却前进了
4. 下游入湖：缺口被固化
5. 补数困难：不知道缺哪段

**屏幕上看起来像什么**

- 系统没报错
- 数据量只是少一点
- 下游报告仍能生成

**实际已经坏在哪**

- state 先于事实前进
- watermark 与写入结果不一致
- backfill 范围难以界定

### INCIDENT C · MISMATCH / DRIFT

> 错位/漂移：字段合法，但运行时意义已经变了。这是 Week02 语义漂移在 Week03 的运行时版本。

1. 上游改口径：updated_at 语义变化
2. contract 未更新：schema 仍可通过
3. 增量窗口错判：该读的没读
4. 报告继续生成：下游以为正常
5. 问题扩大：replay 也复现不了

**屏幕上看起来像什么**：字段还在、schema 仍过、只是「偶尔不准」。

**实际已经坏在哪**：load semantics 漂移、cursor 失去业务含义、需要 contract + manifest + state 一起查。

### INCIDENT D · NO TRACEABILITY

> 无追踪：采集成功了，但没人能解释这次到底发生了什么。这是生产排障时最伤团队信任的状态。

1. 脚本能跑：无 run_id
2. source 混合：无 manifest_id
3. 写入完成：无 source coverage
4. 下游异常：无法定位哪批
5. 只能猜：无法复盘和回放

**屏幕上看起来像什么**：日志里有 done、表里有数据、终端看起来成功。

**实际已经坏在哪**：没有 run evidence、没有 source_fingerprint、没法证明「这批数据」是什么。

## 六、Week03 排障顺序：先找锚点，再谈修复

> 不要一上来重跑。先判断边界、状态、覆盖率。

1. **manifest_id**：这次声明是什么
2. **run_id**：哪次执行出了问题
3. **checkpoint / cursor**：状态走到哪里
4. **manifest coverage**：哪些 source 被覆盖
5. **事故类别**：duplicate / gap / drift / missing trace

**判断口令**：先定位，后恢复；先边界，后命令。

## 八、SMOKE REPORT 至少要写什么

> 报告是 run evidence 的最小形态。没有报告，dry-run 只是「看起来跑过」。

- **Run Identity**：run_id / manifest_id / git_sha / command
- **Source Coverage**：哪些 source 被读取、跳过、隔离
- **Gate Result**：accept / quarantine / reject / warn + reason
- **State Snapshot**：checkpoint / cursor / watermark 当前值
- **Observed Counts**：total / valid / invalid / inserted / skipped
- **Next Action**：continue / retry / replay / backfill

## 九、行业信号：run evidence 正在被对象化

共同趋势：运行意图和运行证据正在对象化。

- **OpenLineage — Facets**：Run / Job / Dataset 上挂 metadata；适合承载 how it ran、what was used、what outcome occurred。
- **Airbyte — Incremental reality**：增量同步依赖 cursor，at-least-once 现实意味着重复不是异常。
- **Dagster — Asset materialization**：运行不只是日志，资产产出和 metadata 会成为后续观察入口。
- **Course Repo — Docker-first baseline**：统一 devbox 命令，避免「我本机能跑」的偶然成功。

### Week02 工件在 Week03 里分别变成什么

> 把「规则文件」压成「运行时对象」。Week03 不是推翻 Week02，而是消费 Week02。

- **Contract**：从字段规则变成 admission gate
- **Manifest**：从装车单变成 batch boundary
- **Gate action**：从 pass/fail 变成分流路径
- **Run evidence**：从观察记录变成恢复依据
