# 06 双源采集最小链路实践

> 本文把 Week03 五节课中的全部 **ENGINEERING HANDOFF / REPO REALITY / Recovery Drill** 汇成一条端到端的「双源采集（ticket events + documents）」最小链路实践，让你在自己机器上跑完一遍 baseline + 增量 + 资产 + 恢复演练。
>
> 这一节不引入新的概念，所有原料都来自前 5 课。它只负责把它们拼成「能跑、可解释、可演练」的一条线。

---

## 一、双源是哪两个源？

Week03 repo 内置两个 demo source，覆盖了「事件流」与「文档流」两种主流采集形态：

| 源 | 类型 | 代表 ingest | Bronze 资产 | Silver 资产 | 主键/去重 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **ticket_events_synthetic** | 工单事件 JSONL | `pipelines/ingestion/ticket_ingest.py` | `raw_ticket_events` | `ticket_fact` | bronze: `event_id` / silver: `ticket_id` |
| **doc_sources** | 文档清单 | `pipelines/ingestion/doc_ingest.py` | `raw_doc_assets` | (后续 Week04+) | `doc_id+version` |

**为什么是「双源」**：

- 事件源训练你处理「同主键多版本」「event_time vs updated_at」「迟到/乱序」。
- 文档源训练你处理「快照/批次」「内容指纹」「PII 边界」。
- 两个源共用同一套 manifest / contract / state / report 锚点 — 这正是 Week03 想强调的可靠性基线。

## 四、第 1 步：启动环境 + 契约门禁

> 任何 ingest 动作前，先确认门禁存在。这是 Week02 合同进入 Week03 运行时的第一个证据。

```bash
# 1) 启动统一开发环境（Docker-first，避免“我本机能跑”）
docker compose --env-file infra/env/.env.local \
  -f infra/docker-compose.yml up -d --build

# 2) 跑契约测试 —— 门禁是否还成立
docker compose --profile tools --env-file infra/env/.env.local \
  -f infra/docker-compose.yml run --rm devbox \
  pytest tests/contract/ -v
```

**观察点**：

- 所有 contract schema 是否仍能通过。
- 任何变更过的字段是否仍在合同允许的范围。
- 若失败：**先停手，先改 contract，不要先改数据**。

## 五、第 2 步：seed_loader dry-run（双源 manifest 准入）

> 这一步只解析、不写入。目的是让两套 source 的 manifest 都能被解释成 batch boundary。

```bash
docker compose --profile tools --env-file infra/env/.env.local \
  -f infra/docker-compose.yml run --rm devbox \
  python -m pipelines.ingestion.seed_loader \
  --manifest-dir data/seed_manifests
```

**应能看到**：

- 读取到 ticket 源 + doc 源各自的 manifest。
- 每份 manifest 解析出：`manifest_id` / `source_id` / `window_start` / `window_end` / `contract_ref`。
- 异常的 manifest 被显式拒绝并给出 reason，不被静默吞掉。

**写下来**：

- `docs/blueprints/week03/ingestion_baseline_v1.md` — 描述本次 baseline 边界。
- `reports/week03/ingest_smoke_report.md` — 最小 smoke 报告（参见第七步报告模板）。

## 六、第 3 步：ticket 源端到端

> 事件源是练「幂等 + 完整性校验 + reconcile」的最佳入口。

### 6.1 dry-run

```bash
python -m pipelines.ingestion.ticket_ingest \
  --dry-run --limit 20 --batch-id week03-smoke
```

观察 summary 中至少六个数：`total / valid / invalid / inserted / skipped / errors`。`Total ≠ Valid ≠ Inserted` 的差异，就是后续 reconcile 入口。

### 6.2 real write（带 batch 边界）

```bash
python -m pipelines.ingestion.ticket_ingest \
  --batch-id week03-smoke
```

### 6.3 验证幂等：把同一批再跑一次

```bash
# 同样的 batch-id，应该全部 skipped，inserted=0
python -m pipelines.ingestion.ticket_ingest \
  --batch-id week03-smoke
```

**期望结果**：

- Bronze 层 `raw_ticket_events`：按 `event_id` 去重，不会因为重跑而膨胀。
- Silver 层 `ticket_fact`：按 `ticket_id` upsert，重跑只更新 `updated_at`，不增加行。
- summary：第二次 `skipped` 应等于第一次 `inserted`。

> 这就是 **rerun ≠ 重复写** 的工程证据。

## 七、第 4 步：doc 源端到端（文档/快照）

```bash
# dry-run 先
python -m pipelines.ingestion.doc_ingest \
  --dry-run --limit 10 --batch-id week03-doc-smoke

# 真实写入
python -m pipelines.ingestion.doc_ingest \
  --batch-id week03-doc-smoke
```

**双源差异要记下来**：

| 维度 | ticket 源 | doc 源 |
| :--- | :--- | :--- |
| 主键 | `ticket_id` | `doc_id+version` |
| dedupe key | `event_id` | `content_hash` 或 `doc_id+version` |
| 增量方式 | event_time / updated_at | 快照/批次 |
| Silver 形态 | 事实表 upsert | 后续在 Week04 入 lake |
| 常见坑 | 状态语义漂、updated_at 不可靠 | PII 边界、二进制大字段 |

## 八、第 5 步：在 Dagster UI 观察「资产视角」

```bash
# 打开 http://localhost:3000
```

在 UI 中要看的不是绿色 ✅，而是：

- **asset key**：`seed_manifests` / `raw_ticket_events` / `raw_doc_assets` 是否稳定命名。
- **lineage**：manifest → raw → (后续 silver/lake) 链路是否完整。
- **materialization metadata**：每次产出是否带 `manifest_id / batch_id / run_id / row_count / reject_count / checksum / reason_code`。
- **partition**：能否按天 / 按 batch 切分（即使现在只声明了一种）。
- **asset checks**：是否有最小一条健康约束（如「关键字段非空率 > 阈值」）。

> 任务流回答「跑没跑」，资产流回答「这批数据能不能消费」。

## 十、第 7 步：Recovery Drill（恢复演练）

> 不要等出真事故才写 runbook。先**人为造场景**，再走完 decision → command → verify → record。

### 10.1 准备三份工件

1. `runbooks/ingestion_runbook_v1.md`
2. `docs/blueprints/week03/replay_backfill_strategy_v1.md`
3. `reports/week03/recovery_drill_report.md`

### 10.2 三个推荐演练场景

| 场景 | 推荐动作 | 演练目标 |
| :--- | :--- | :--- |
| 故意让一个 ticket source 在 manifest 中被跳过 | **scoped replay**（仅重放该 source） | 验证 replay 不会污染其他 source |
| 删除 ticket 某天的 Bronze 数据，制造历史空洞 | **backfill**（按 partition 补） | 验证补数边界精准、不重不漏 |
| 修改 ticket_contract 的某字段语义 | **backfill + asset_check** | 验证逻辑变更后历史可重算、健康约束能拦下不一致 |

### 10.3 决策必须先写，再动手

在 `replay_backfill_strategy_v1.md` 里至少写 6 项：

1. 场景描述
2. 当前症状
3. 目标动作（retry / replay / restore / backfill）
4. 为什么不是另外几个
5. 输入边界（manifest / source / window / partition）
6. 验收标准

### 10.4 决策树速查

```diagram
异常出现
   │
   ▼
当前 state / sink / run log 一致吗？
   │
   ├── 不一致 ─▶ Restore 到已知可用点 ─▶ 再判断 scoped recovery
   │
   └── 一致 ─▶ ┬─ 执行层失败  ─▶ retry / rerun
               ├─ 想验证同批  ─▶ replay
               ├─ 某分区缺失  ─▶ backfill (补)
               └─ 逻辑变了    ─▶ backfill (重算)
```

### 10.5 演练后写 recovery report

`reports/week03/recovery_drill_report.md` 至少记：

- 采用了哪个动作、为什么、引用了哪些锚点（state/manifest/run log）。
- 执行命令与时间点。
- 预期 vs 实际：恢复后 counts / reconcile / checks 是否闭合。
- 验证通过的证据（截图、数字、check 名称）。
- 尚未自动化的能力 → 列入下一步工程债。

## 十二、最终自检：你能不能用这条链路解释下面 8 句话？

如果都能脱稿讲清，本周 Week03 才算真的「跑通了」：

1. 我能说出 ticket 源和 doc 源各自的 **dedupe key 与 idempotency key**。
2. 我能解释为什么 `--batch-id` 不能在 rerun 时换掉。
3. 我能从一次 ingest report 反推出当时的 **manifest / source coverage / state**。
4. 我能解释 ticket Bronze 与 Silver 在「重复执行」时的不同表现。
5. 我能在 Dagster UI 里看着 materialization metadata 说出哪批数据可以消费。
6. 我能指出 doc 源在哪些点最容易 **quietly fail**（schema 合法但语义不对）。
7. 我能在白板上画出「retry / rerun / replay / restore / backfill」对应的恢复层级。
8. 我能引用 manifest / state / run log，写出一份不靠经验、不靠记忆的 recovery report。

> Week03 的核心不是「把数据搬进来」，而是让两条源（事件 + 文档）进入系统后**可解释、可重跑、可补数、可回放、可治理**。
> 这条最小双源链路，是后面 Week04 lakehouse、Week06 orchestration、Week08 RAG evidence 的共同底座。
