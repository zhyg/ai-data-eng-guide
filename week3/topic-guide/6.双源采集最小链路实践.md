# 导学 06：把前 5 节串起来，做一次双源演练

## 一句话理解

这一节没有新概念，就是把工单源和文档源**真的跑一遍**，看看最小采集闭环是不是能跑通、能解释、能重跑。

## 生活类比

像学车路考：前 5 节学交规、认仪表、练倒库；这一节要上路跑一圈。

- 工单源像市区路况：事件多、变化快，要防重复和乱序。
- 文档源像高速路况：按批次走，重点看版本、指纹和边界。
- 跑完再模拟一次抛锚，练一次故障恢复。

## 关键概念表格

两条源走法不同，先各自看清楚：

| 源 | 类型 | Bronze | Silver | 去重 / 主键 | 重点看什么 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| ticket_events_synthetic | 工单事件 JSONL | raw_ticket_events | ticket_fact | Bronze: event_id；Silver: ticket_id | 行数、有效 / 无效、是否重复入库 |
| doc_sources | 文档清单 | raw_doc_assets | Week04 继续入湖 | doc_id + version / content_hash | 版本、source_fingerprint、license、skip 原因 |

整条链路按 7 步走，每步都有通过标准，绿了再往下：

| 阶段 | 做什么 | 通过标准 |
| :--- | :--- | :--- |
| 1 起环境 | 用 compose 起服务，确认 FastAPI、Dagster UI 都能访问 | 服务起得来，端口对得上，不是只在自己机器上能跑 |
| 2 契约门禁 | 在 devbox 容器里跑 contract baseline 测试 | 契约和清单测试全绿，准入通过才搬数据 |
| 3 清单覆盖 | 跑 seed_loader dry-run | 每个 manifest / source / window 都能解释；pass / warn / quarantine 都有 reason_code |
| 4 工单链路 | 先用模拟器合成工单 JSONL，再跑 ticket ingest dry-run | total / valid / invalid 对得上，inserted=0；同批再跑行数不增 |
| 5 文档链路 | 跑 doc ingest dry-run | 版本、source_fingerprint、批次边界说得清；skip 的能说出原因 |
| 6 资产视角 | 打开 Dagster UI，看 seed_manifests / raw_ticket_events / raw_doc_assets | 三个 asset 连成图，视角从脚本跑成功切到资产能用 |
| 7 恢复演练 | 先写决策、再执行、最后写报告 | replay / backfill / runbook 的入口都能演练，证据可回看 |

smoke report 不能只写跑通，至少分模块写：

| 模块 | 要写什么 |
| :--- | :--- |
| Run Identity | run_id、manifest_id、batch_id、命令、git sha、生成时间 |
| Source Coverage | 读到、跳过、隔离了哪些 source |
| Gate Result | accept / warn / quarantine / reject + reason_code |
| State Snapshot | checkpoint / cursor / watermark 前后值 |
| Observed Counts | total / valid / invalid / inserted / skipped / errors |
| Reconcile | manifest coverage、Bronze / Silver 差异是否可解释 |
| Next Action | continue / retry / replay / backfill 及原因 |

跑完留下的交付物不要漏：ingestion_baseline_v1.md、checkpoint_state_v1.md、incremental_ingest_strategy_v1.md、late_arrival_decision_table.csv、asset_flow_plan_v1.md、replay_backfill_strategy_v1.md、ingestion_runbook_v1.md、双源 smoke report、recovery_drill_report.md、至少一次带 metadata 的 materialization、至少一条 asset check。

## 避坑提醒

- 跳过契约 baseline 就 ingest：第 2 步是门禁，没绿就不要进 ingest。
- 把 dry-run 当真实写入：演练里大多数环节都不落库，看的是真写会发生什么。
- 把 skip 当错误：skip 是被规则拦下，要去看 reason_code，不是 fail。
- 重跑时改 batch_id：同一批要保持同一个 batch_id，否则系统当成新批次，幂等失效。
- 只看脚本绿不绿：要切到 Dagster 资产视图，问哪个 asset 能被下游用。
- smoke report 只写跑通：要带清单、覆盖率、状态、对账和下一步，否则下次出事还是靠记忆。
- 故障演练先敲命令：先写场景、动作选择、影响范围和验收标准，再动手。

## 一句话总结

> **跑完这条链路，双源采集能解释清楚、能稳定重跑、能补数、能留下恢复证据，Week03 才算收官。**
