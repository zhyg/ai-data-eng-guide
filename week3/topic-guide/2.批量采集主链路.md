# 导学 02：批量主链路——先把「一批一批搬」做稳

## 一句话理解

先别急着实时；先做到**同一批数据跑两次，结果不变，差异能对账**。

## 生活类比

批量采集像超市每天关门后盘点：

- 一天的销售统一结算，这就是一批。
- 算错了可以按同一张清单重算，最终数字应该一致。
- 盘点小票要写清：应收多少、实收多少、退了多少、漏了多少。

批量不是落后，它是训练可靠性的第一块地基。

## 关键概念表格

先把批次边界写清楚：

| 边界 | 大白话 | 为什么重要 |
| :--- | :--- | :--- |
| manifest_id | 按哪份清单跑 | 不然不知道这次接了谁 |
| batch_id | 这一批数据的身份与业务边界 | 重跑时不能被当成新数据 |
| window_start / window_end | 这批覆盖哪段时间 | 防止多拿或漏拿 |
| source_fingerprint | 来源指纹 | 判断输入是否和上次一样 |

4 个动作不要混用：

| 动作 | 大白话 | 没做好会怎样 |
| :--- | :--- | :--- |
| 幂等写入 | 同一动作做多次，结果一样 | 重跑一次数据翻倍 |
| rerun | 同一任务失败后再执行 | 不敢恢复，只能手补 |
| 完整性校验 | 写完后数一数 | 写完了也不知道对不对 |
| reconcile | 清单、读取、写入、异常能对上 | 少了 / 多了都说不清 |

Bronze 和 Silver 的写法也不同：

| 层 | 作用 | 常用保护 |
| :--- | :--- | :--- |
| Bronze raw_ticket_events | 保存原始事件证据 | 用 event_id 去重 |
| Silver ticket_fact | 形成可用事实表 | 用 ticket_id 更新同一对象 |
| Report / State | 解释本次运行 | 用 run_id + batch_id 串起来 |
| Quarantine | 隔离坏记录 | 记录 reason_code 和来源位置 |


## 避坑提醒

- dry-run 通过就上线：试跑不等于真实数据库写入可靠。
- 重跑时换 batch_id：系统会把它当新批次，幂等容易失效。
- 只看 valid + invalid = total：还要看 manifest coverage、source count、Bronze / Silver 差异和 reason_code。
- 所有层都 append：Bronze 保证证据，Silver 保证可用事实，写入承诺不同。
- 报告只写成功：要写 inserted / updated / skipped / errors，并说明下一步。

## 一句话总结

> **同一批能安全重跑，写入能解释，结果能对账。批量主链路练的就是这三件事。**
