# 导学 04：别再问「脚本跑成功了吗」，问「这份数据能用吗」

## 一句话理解

脚本跑成功不等于数据能用；数据工程更关心**哪份资产产生了、能不能被下游放心使用**。

## 生活类比

把数据管道想象成外卖：

- **任务流视角**：骑手出发了吗？送到了吗？
- **资产流视角**：饭热不热？分量对不对？能不能吃？

用户关心的是手上的饭，不是骑手跑步姿势；下游关心的是数据资产，不是脚本是否绿色。

## 关键概念表格

任务流和资产流回答的问题不同：

| 任务流问 | 资产流问 |
| :--- | :--- |
| 哪个脚本跑了？ | 哪个资产已经产生？ |
| 哪个脚本失败？ | 哪个资产分区缺失？ |
| 退出码是不是 0？ | 这份数据能不能消费？ |
| 要重跑哪个脚本？ | 要补哪个 asset + partition？ |

4 个对象必须分开：

| 对象 | 大白话 | 常见误解 |
| :--- | :--- | :--- |
| manifest | 这次想接什么、怎么接 | 把清单当资产 |
| asset | 实际存在、可被使用的数据结果 | 把函数名当资产名 |
| job | 触发产出的执行方式 | 把 job 成功当资产成功 |
| asset check | 资产健康检查 | 只看脚本退出码 |

当前 repo 已有资产入口：

| 名字 | 含义 |
| :--- | :--- |
| seed_manifests | 清单资产入口 |
| raw_ticket_events | 工单事件 raw 资产 |
| raw_doc_assets | 文档 raw 资产 |
| pipelines/ingestion/assets.py | 资产定义位置 |
| ingest_all_job | 最小 materialization 触发方式 |

一次 materialization 的「小票」建议带：run_id、manifest_id、batch_id、asset_key、source coverage、partition_key、row_count、reject_count、skipped、reason_code、checkpoint_after、watermark_after、lineage。

## 避坑提醒

- 把 Dagster 只当调度器：这节重点是资产视角，不是工具名。
- 说 asset 是函数：asset 是表、文件、清单等持久化结果，函数只是产生方式。
- job 成功就放下游：还要看 metadata 和 asset check。
- 分区等数据大了再说：partition 是为了补得准，不只是跑得快。
- metadata 只写行数：还要写来源、批次、状态、跳过原因和上下游。

## 一句话总结

> **任务流看「跑没跑」，资产流看「能不能用」；生产里真正重要的是后者。**
