# 导学 06：把前 5 节串起来，做一次双源演练

## 一句话理解

这一节没有新概念，就是把工单源和文档源**真的跑一遍**，验证它们可解释、可重跑、可补救。

## 生活类比

像学车路考：前 5 节学交规、认仪表、练倒库；这一节要上路跑一圈。

- 工单源像市区路况：事件多、变化快，要防重复和乱序。
- 文档源像高速路况：按批次走，重点看版本、指纹和边界。
- 跑完还要模拟一次抛锚，练会故障恢复。

## 关键概念表格

双源先认清：

| 源 | 类型 | ingest 入口 | Bronze | Silver | 去重 / 主键 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `ticket_events_synthetic` | 工单事件 JSONL | `pipelines/ingestion/ticket_ingest.py` | `raw_ticket_events` | `ticket_fact` | Bronze: `event_id`；Silver: `ticket_id` |
| `doc_sources` | 文档清单 | `pipelines/ingestion/doc_ingest.py` | `raw_doc_assets` | Week04 继续入湖 | `doc_id + version` / `content_hash` |

最小流程：

| 步骤 | 做什么 | 通过标准 |
| :--- | :--- | :--- |
| 1 | 起环境，跑合同测试 | `tests/contract/` 全绿 |
| 2 | 跑 `seed_loader` dry-run | 两个源的 `manifest_id / source_id / window` 都能解释 |
| 3 | 跑工单 dry-run、真写、同批再跑 | 第二次行数不增加，`inserted=0` |
| 4 | 跑文档 dry-run、真写 | 版本、指纹、批次边界说得清 |
| 5 | 看 Dagster UI | `seed_manifests / raw_ticket_events / raw_doc_assets` 有稳定 asset key 和 metadata |
| 6 | 写 smoke report | 数量、状态、对账、下一步都写清 |
| 7 | 做恢复演练 | 先写决策，再执行，再写恢复报告 |

`seed_loader` 建议用 Docker-first 命令，避免「只在我电脑能跑」：

```bash
docker compose --profile tools --env-file infra/env/.env.local \
  -f infra/docker-compose.yml run --rm devbox \
  python -m pipelines.ingestion.seed_loader \
  --manifest-dir data/seed_manifests
```

smoke report 至少包含：

| 模块 | 要写什么 |
| :--- | :--- |
| Run Identity | `run_id`、`manifest_id(s)`、命令、git sha |
| Source Coverage | 读到、跳过、隔离了哪些 source |
| Gate Result | accept / warn / quarantine / reject + `reason_code` |
| State Snapshot | checkpoint / cursor / watermark 前后值 |
| Observed Counts | `total / valid / invalid / inserted / skipped / errors` |
| Reconcile | manifest coverage、Bronze / Silver 差异是否可解释 |
| Next Action | continue / retry / replay / backfill 及原因 |

最后交付物不要漏：`ingestion_baseline_v1.md`、`checkpoint_state_v1.md`、`incremental_ingest_strategy_v1.md`、`late_arrival_decision_table.csv`、`asset_flow_plan_v1.md`、`replay_backfill_strategy_v1.md`、`ingestion_runbook_v1.md`、双源 smoke report、`recovery_drill_report.md`、至少一次带 metadata 的 materialization、至少一条 asset check。

## 避坑提醒

- 跳过合同测试就 ingest：先确认门禁，再搬数据。
- `seed_loader` 只用本机命令：导学里用 Docker-first，和主题保持一致。
- 重跑时改 `--batch-id`：同一批要保持同一个批次身份。
- 说 Bronze / Silver 什么都不变：更准确是**行数不增加**；Silver 可能只更新审计或更新时间字段。
- smoke report 只写「跑通」：必须带清单、覆盖率、状态、对账和下一步。
- 故障演练先敲命令：先写场景、动作选择、影响范围和验收标准。

## 一句话总结

> **跑完这条链路，能证明双源采集可解释、可重跑、可补数、可留恢复证据，Week03 才算收官。**
