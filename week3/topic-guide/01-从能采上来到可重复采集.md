# 导学 01：从「采到一次」到「可重复采集」

## 一句话理解

偶尔采到数据不算可靠；**同一批数据能安全再跑，出问题还能说清哪份清单、哪次运行、处理到哪**，才叫可重复采集。

## 生活类比

把数据采集想象成每天取快递：

- **采到一次**：今天把快递拿到了。
- **可重复采集**：每天按清单取；丢件知道是哪一单；同一件不会重复签收；需要补寄时知道从哪补。
- **运行证据**：就像快递小票，写清谁取的、取了哪些、漏了哪些、下一步怎么办。

## 关键概念表格

| 概念 | 大白话 | 要回答的问题 |
| :--- | :--- | :--- |
| `manifest_id` | 本次按哪份采集清单跑 | 这次声明要接哪些源？ |
| `batch_id` | 这批数据的业务边界 | 这次处理哪一天 / 哪一段？ |
| `run_id` | 这一次执行的编号 | 出问题的是哪一次跑？ |
| `source_fingerprint` | 来源指纹 | 原始数据和上次比变没变？ |
| `checkpoint` | 状态存放位置 | 崩了以后从哪恢复？ |
| `cursor` | 下次从哪继续读 | 下一次读取起点在哪里？ |
| `watermark` | 已确认处理到哪 | 迟到、乱序、补数怎么判断？ |
| smoke report | 最小运行小票 | 跑了什么、写了多少、跳过多少、下一步是什么？ |

常见事故先认这 4 类：

| 事故 | 表现 | 常见原因 |
| :--- | :--- | :--- |
| 重复 Duplicate | 同一条写两遍，报表偏高 | 重跑没有幂等保护 |
| 缺口 Gap | 少了一段历史，系统可能不报错 | 窗口、游标、状态不一致 |
| 漂移 Drift | 字段还在，但含义变了 | 上游口径变了，合同没跟上 |
| 无追踪 Trace | 数据进来了，却说不清来源 | 没有 `manifest_id` / `run_id` / report |

最小 smoke report 至少写：`run_id`、`manifest_id`、source coverage、gate result、checkpoint / cursor / watermark、`total / valid / invalid / inserted / skipped`、next action。

## 避坑提醒

- ❌ 以为「脚本没报错 = 成功」：还要看有没有少读、重复写、静默跳过。
- ❌ 只问数据源名称：要能说出哪个 `manifest_id` 和 `source_id`。
- ❌ 把状态写在日志里：日志不是状态，`checkpoint / cursor / watermark` 要能被恢复动作读取。
- ❌ 一出事就重跑：先看清单、运行编号、状态、覆盖率，再决定 retry / replay / backfill。
- ❌ 报告只写 done：必须写清数量、跳过原因、下一步动作。

## 一句话总结

> **采到一次是运气；同一批能安全再跑、出事能解释清楚，才是可靠采集。**
