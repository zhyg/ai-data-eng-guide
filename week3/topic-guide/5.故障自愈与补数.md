# 导学 05：出事了，先选对恢复动作

## 一句话理解

系统跑通只是入门；**出事后能按边界恢复、能说明为什么这么做**，才是真正的生产能力。

## 生活类比

数据系统恢复像做菜出问题：

| 状况 | 该怎么办 | 对应动作 |
| :--- | :--- | :--- |
| 火临时灭了 | 再点一次 | retry |
| 这道菜做到一半中断 | 同一道菜重新做 | rerun |
| 想验证同一份食材按新流程是否稳定 | 同一份食材再做一遍 | replay |
| 厨房台面被污染 | 先恢复到干净状态 | restore |
| 上周少做了一道菜 | 补做那一道 | backfill |

不同问题用不同动作；一上来把整桌菜重做，通常又贵又危险。

## 关键概念表格

5 个恢复动作不要混：

| 动作 | 解决什么 | 什么时候用 | 误用风险 |
| :--- | :--- | :--- | :--- |
| retry | 执行级短暂失败 | 网络抖动、服务临时不可用 | 无脑重试，放大问题 |
| rerun | 同一 job 再执行 | 上次执行中断 | 没幂等就重复写 |
| replay | 同一批输入重放 | 验证幂等、同批重走 | 被当成新数据 |
| restore | 回到已知可用状态 | 状态或下游被污染 | 跳过它会扩散脏数据 |
| backfill | 补历史空洞或重算旧分区 | 某天缺数据、逻辑变更 | 变成全链路重跑 |

恢复要靠 4 个锚点：

| 锚点 | 回答的问题 |
| :--- | :--- |
| contract | 数据该长什么样？ |
| manifest | 要恢复哪个源、哪段窗口？ |
| state / checkpoint | 上次成功到哪，能从哪恢复？ |
| run log / report | 失败在哪层，之前做过什么？ |

Week03 要准备的恢复工件：

| 文件 | 作用 |
| :--- | :--- |
| runbooks/ingestion_runbook_v1.md | 应急手册和升级路径 |
| docs/blueprints/week03/replay_backfill_strategy_v1.md | replay / backfill 决策依据 |
| reports/week03/recovery_drill_report.md | 演练过程和结果证据 |

角色也要说清：判断者、批准者、执行者、记录者、Owner、Reviewer、Escalation、Auditor。小团队可以一人多职，但不能没人负责。

## 避坑提醒

- 出事就全量重跑：成本高、风险大，还说不清修了什么。
- 把 retry / rerun / replay 都叫重跑：动作边界不同，影响范围也不同。
- 状态已经脏了还 replay：先 restore 到可信状态，再做精准恢复。
- 恢复完不写报告：下次还是靠记忆，runbook 无法进化。
- 假装已经全自动：Week03 先讲清恢复边界；当前还没有通用 replay service、自动 checkpoint manager、一键 backfill engine。

## 一句话总结

> **先看锚点，再选动作，最后留证据。急着重跑反而扩大事故。**
