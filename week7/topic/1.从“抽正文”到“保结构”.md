## WAR STORY：财报 PDF 抽串了一栏

前几年做客户的金融 RAG，团队用 PyPDF 抽文本，Demo 跑得溜。上线第一周客户问「Q3 净利润多少」，LLM 答了一个 Q2 的数字。

- 排查 6 小时，根因是 PyPDF 把 2 栏排版的财报抽成单栏，Q3 表格的数字串到 Q2 表格旁边了。
- LLM 看不懂上下文，瞎填。

**那不是模型笨，是「文档不是字符串」。你不保结构，LLM 永远不可能答对。**

## DOCUMENT ≠ STRING：有结构 vs 没结构

同一份 PDF，有结构 vs 没结构，下游 RAG 效果差 3-5 倍。

| 维度 | 把 PDF 当字符串 | 把 PDF 当结构化资产 | 差异 |
| :--- | :--- | :--- | :--- |
| 多栏 layout | 抽成单栏 / 文字串行 | 识别 column，各栏分别处理 | 财报必坏 → 必对 |
| 表格 | 抽成几列空格分隔的乱码 | markdown table / cell 边界保留 | 看不懂 → 看得懂 |
| 标题层级 | 全是 plain text | H1 / H2 / H3 + chunk metadata | 检索精度 +30% |
| 图表 / 公式 | 直接丢 | caption + LaTeX + 引用 ref | 专业 RAG 必须 |
| 页眉页脚 | 混入正文搞乱 chunk | 识别 + 剥离 + 入 metadata | 幻觉率 -40% |
| 坐标信息 | 没有 | 每个 chunk 带 page + bbox | 可点击高亮原文 |

**老司机口诀**：PDF 处理工程化 = 拒绝「文字提取」，必须是「结构提取」，否则 RAG 答案永远飘。

## OLD VS NEW：纯文本抽取 vs IDP

## 5-STAGE PIPELINE：文档处理 5 阶段流水线

`Load → Layout → Extract → Normalize → Persist`，每阶段独立工程对象，缺一阶段就有盲区。

| 阶段 | 职责 | 内容 |
| :--- | :--- | :--- |
| **Load** | 加载 + 类型识别 | mimetype / 编码 / 扫描 vs 文本型 / 多页文档拆分 |
| **Layout** | 布局检测 | 多栏 / 表格 / 图 / 页眉页脚 / 标题层级 H1-H6 |
| **Extract** | 内容抽取 | 文字 / 表格 → markdown / 公式 → LaTeX / 图 → caption |
| **Normalize** | 清洗 + 结构化 | 去噪 / 合并断行 / chunk 划分 / metadata 标注 |
| **Persist** | 入库 + 版本 | 原文档 + chunks + bbox + version_id 一起持久化 |

**老司机说**：5 阶段是「最小不可分」，每个阶段 1 个工程对象，独立可测 / 可换工具 / 可监控。Load 错了后面全错，所以 PDF 类型识别 = 整个 pipeline 的入口，优先级最高。

## LAYOUT MATTERS：布局是语义不是好看

这 3 个东西丢了，下游 RAG 必坏：

- **多栏 column**（财报 / 论文 / 合同）：抽成单栏 → 跨栏串扰，净利率串到营收旁边。识别用 PyMuPDF `blocks_xy` 或 IDP。
- **标题层级 H1-H6**（"3.2.1 节 / Article 5"）：丢掉 → chunk 失去上下文；保留 → metadata 入检索，"找第 5 章利润分析"直接命中。识别靠字号 + 缩进 + 编号 pattern。
- **表格 cell 边界**（股价表 / 财报表）：抽成空格分隔 → LLM 不识；保 markdown 或 cell 坐标 + 表格描述双轨。识别用 LlamaParse / Camelot。

**判断**：Layout 不是 UI 问题，是「让 LLM 看得懂」的语义问题。

## PDF 5 大难点 + 类型识别

60-80% 企业知识在 PDF，而 PDF 是企业最大的工程黑洞。

**5 大难点**：Layout 复杂（多栏/表/图）；扫描版没文字层（必须 OCR）；表格/公式难保留语义；页眉页脚/水印混入；字体编码乱（老 PDF）。

**类型识别是入口**（走对工具能省 70% 成本）：

| 类型 | 文字覆盖率 | 走什么工具 |
| :--- | :--- | :--- |
| text-based | coverage > 0.8 | LlamaParse / Marker |
| scanned | coverage < 0.2 | Mistral OCR / Azure DI |
| hybrid | 介于中间 | 双管 fallback |

```python
import fitz  # PyMuPDF · PDF 类型识别 30 行代码
def detect_pdf_type(path):
    doc = fitz.open(path)
    has_text = sum(1 for p in doc if p.get_text().strip()) / len(doc)
    return "text-based" if has_text > 0.8 else "scanned" if has_text < 0.2 else "hybrid"
# text-based → LlamaParse / Marker · scanned → Mistral OCR / Azure DI · hybrid → 双管 fallback
```

## REAL ARTIFACT · Marker

MIT 协议、GitHub 19k+ stars，2024-2026 开源 PDF 处理几乎默认。

```python
# pip install marker-pdf
from marker.converters.pdf import PdfConverter
from marker.models import create_model_dict

converter = PdfConverter(
    artifact_dict=create_model_dict(),
    config={
        "extract_images": True,      # 图片单独抽
        "output_format": "markdown",
        "use_llm": True,             # LLM 兜底疑难（Claude Sonnet 4.5）
        "llm_service": "anthropic",
        "force_ocr": False,          # text-based 不必 OCR
        "paginate_output": True,     # 保留 page break，evidence 用
    },
)
rendered = converter("./financial_report_2025_Q3.pdf")
markdown_text = rendered.markdown
images = rendered.images          # {page_n: PIL.Image}
metadata = rendered.metadata      # title / authors / page_count
```

拿到的 markdown 已经处理好：多栏自动分栏识别后串接、表格转标准 markdown、公式转 `$$LaTeX$$`、标题转 `# / ## / ###`、图转 `![desc](images/img_001.png)`。

**Marker 工程要点 5 条**：`use_llm=True` 兜底复杂 layout；`extract_images=True` 图独立存后接 VLM caption；`paginate_output=True` 保 page break 给 evidence；`force_ocr=False` 先看类型再决定；输出标准 markdown 下游 chunker 直接吃，比自研抽取器节省 70% 代码量。

## REAL ARTIFACT · Docling

IBM 出品，Granite-Docling 2025.9 强化版。中文友好、公式强、完全开源，跟 Marker 互补。

```python
# pip install docling
from docling.document_converter import DocumentConverter
from docling.datamodel.base_models import InputFormat
from docling.datamodel.pipeline_options import PdfPipelineOptions

opts = PdfPipelineOptions(
    do_ocr=False,                 # 先看类型
    do_table_structure=True,      # 表格 cell 还原
    table_structure_options={"do_cell_matching": True, "mode": "accurate"},
    images_scale=2.0,             # 图像 2x，后续 VLM 看更清
    generate_picture_images=True,
)
converter = DocumentConverter(format_options={InputFormat.PDF: opts})
result = converter.convert("./contract_2025.pdf")
doc = result.document

markdown = doc.export_to_markdown()
json_doc = doc.export_to_dict()   # 含 bbox / page

for item in doc.iterate_items():  # 拿 chunk 级 metadata
    if hasattr(item, "prov"):
        for p in item.prov:
            page = p.page_no; bbox = p.bbox   # 用于 EvidenceAnchor
```

**5 个工程要点**：`do_ocr=False`（text-based 不必 OCR）；表格 cell 边界精确还原；`images_scale=2.0` 后接 VLM caption；`iterate_items()` 拿 prov 的 page_no / bbox 直接喂 EvidenceAnchor；accurate 模式比 fast 慢 30% 但精度 +5%，财务 / 法律必上。

## TABLES：表格比文本难 10 倍

表格是 PDF 里硬骨头中的硬骨头，上线必专题处理。

**3 大难点**：Layout 识别（哪些是表）、Cell 边界（合并 / 多行）、语义保留（纯 markdown 丢上下文）。5 列以上检索就掉精度。

**3 种存储策略**：

| 策略 | 适用 | 说明 |
| :--- | :--- | :--- |
| A · markdown chunk | 小表（< 20 行） | 直接当 markdown chunk |
| B · 描述 + markdown 双轨 ★ | 中表 | LLM 生成「表格描述」+ markdown，**2026 主流默认** |
| C · 转 SQL Table | 大表 | NL2SQL Agent，**财报场景必上** |

**5 个抽取工具**：PDFPlumber（开源简单）、Camelot（开源精度高）、Tabula（Java 老牌）、LlamaParse（商业首选）、AWS Textract Tables（最准最贵）。

## INDUSTRY SIGNALS 2025：IDP 4 个方向

不是「百家争鸣」，是「主干清晰 + 补丁式选型」。

- **LlamaParse + Mistral OCR**（2024-2026）：两条商业线覆盖 To B 主战场，2026 Mistral OCR 精度首次超 AWS Textract、价格只一半。
- **Granite-Docling 2025.9**：IBM Docling + Granite 3.1B 视觉模型融合，完全开源，表格抽取在 PubLayNet 上 SOTA，中文友好。
- **ColPali / ColQwen2**（2024.7-2025）：彻底跳过解析，整页 PDF 当图像检索，视觉文档 RAG 范式革新（L05 详讲）。
- **Anthropic Visual PDF**（2025.3）：Claude Sonnet 4.5 原生 PDF input，layout-aware 内建，IDP 边界正被 VLM 蚕食。

**2026 共识**：传统 IDP 主干 + VLM 兜底 + ColPali 兜兜底，这是「资产化」管道的标配。

## RECAP：本讲判断

1. 文档不是字符串——不保结构，LLM 永远答不对。
2. 有结构 vs 没结构，下游 RAG 效果差 3-5 倍。
3. 5 阶段 pipeline 是最小不可分，Load 阶段的 PDF 类型识别是整条管道入口。
4. Layout 是语义问题不是 UI 问题：多栏、标题层级、表格 cell 边界丢一个就翻车。
5. 工具选型：Marker 起步、LlamaParse 上合规、Mistral OCR 走量、VLM 兜底 5% 疑难。
6. 表格比文本难 10 倍，按表大小走 markdown / 描述双轨 / SQL Table 三种策略。
