## THREE-TRACK MODEL：音视频「三轨道」模型

Audio / Vision / Time-Stamp 三条轨独立处理 + 时间戳对齐，这是工程化处理的统一抽象。

| 轨道 | 处理 | 入 chunk 字段 |
| :--- | :--- | :--- |
| **Audio Track · 音轨** | ASR 转写（Whisper-v3-Turbo）+ Diarization（pyannote 3.1），输出说话人 + 时间戳 + 文字 | speaker_id（工业默认） |
| **Vision Track · 视频轨** | 关键帧抽取（ffmpeg / OpenCV）+ VLM Caption（Claude / Gemini），每 N 秒 1 帧 / 场景切换检测 | frame_path（可视化检索） |
| **Time-Stamp · 对齐轨** | segment_ts 把音 + 视 + 文统一，同一秒发生的内容互相关联，evidence 跳转到精确时刻 | segment_ts（同构 EvidenceAnchor） |

**判断**：三轨独立处理 + segment_ts 对齐，是 2026 音视频 RAG 的工业默认，比 monolithic 强 5x。

## AUDIO VS DOCUMENT：看似不同，本质同构

音频不是新数据类型，是同一套范式的物理形态。

| 维度 | 文档（PDF） | 音频（mp3） | 同构关系 |
| :--- | :--- | :--- | :--- |
| Parse | Marker → markdown | Whisper → 文字 + 时间戳 | 都抽语义 |
| Chunk | 按 heading / 句子 | 按 speaker / 话题 | 都按结构 |
| Evidence | page_no + bbox | segment_ts + speaker_id | 都可定位 |
| Quality | 4 维质检 / Coverage | 4 维质检 / WER + Diarization Acc | 都可量化 |
| Index | pgvector 入 | pgvector 入 + audio_url metadata | 同 store |
| Retrieve | Hybrid + Rerank | Hybrid + Rerank | 同链路 |
| 前端 | 高亮原 PDF | 跳转音频时刻 | 统一 UX |

**老司机讲**：接到「音视频 AI」项目不要慌，套 L01-L04 的范式 + 物理载体换，90% 工作量可复用。

## VIDEO PIPELINE：三轨分头处理 + 时间戳对齐

视频比音频复杂，但工程化逻辑还是一样，关键是 segment_ts 把三轨对齐起来。

| 步骤 | 工具 | 产物 |
| :--- | :--- | :--- |
| ① Demux | ffmpeg 拆音 + 视 | audio.wav / frames/*.jpg |
| ② ASR | Whisper-v3-Turbo | 转写 + 字级时间戳 |
| ③ Diarization | pyannote 3.1 | speaker A/B/C 时间段 |
| ④ KeyFrame | scene-detect 切 | 每 N 秒 1 帧 / 场景变化 |
| ⑤ VLM Caption | Claude / Gemini | 每帧 caption + OCR |
| ⑥ Align | segment_ts 合 | 三轨同 ts 入同 chunk |

**关键设计**：segment_ts 是「主键」，把音 / 视 / 文三个独立处理的结果黏合。
**实战**：1 小时视频，ASR ~10 min、KeyFrame 抽 600 帧、VLM Caption ~5 min，全 pipeline 总 < 25 min，成本 < $8。

## REAL ARTIFACT · Audio Segment

schema 和文档 EvidenceAnchor 完全同构，这就是「同一套范式」的力量。

```python
# services/audio_pipeline/models.py
from pydantic import BaseModel, Field
from typing import Literal

class AudioSegment(BaseModel):
    # === Identity（同 EvidenceAnchor）===
    doc_id:      str                  # 音频文件主键
    version_id:  str
    chunk_id:    str
    source:      str                  # "interview_2026_05.mp3"
    source_type: Literal["pdf", "audio", "video", "image", "code"] = "audio"

    # === Location · 音频用 segment_ts 替代 page+bbox ===
    segment_ts_start: float = Field(..., description="秒级开始")
    segment_ts_end:   float = Field(..., description="秒级结束")
    audio_url:        str   = Field(..., description="原音频 URL")

    # === Audio-specific ===
    speaker_id:    str = Field(..., description="说话人 A/B/C")
    speaker_label: str = Field(default="", description="可选 · 真实姓名")
    confidence_asr:         float = Field(..., ge=0.0, le=1.0)
    confidence_diarization: float = Field(..., ge=0.0, le=1.0)
    language: str = "zh"

    # === Context（同 EvidenceAnchor）===
    rerank_score:     float = Field(..., ge=0.0, le=1.0)
    retrieval_method: Literal["hybrid", "vector", "bm25"] = "hybrid"
    release_id:  str
    chunk_text:  str                  # ASR 文字
    heading_path: list[str] = Field(default_factory=list)   # 可选话题层级
```

## REAL ARTIFACT · Whisper + Diarization

OpenAI 2024.10 出 Turbo，比 v3 快 8x、精度只降 1%，生产首选。

```python
# pipelines/audio/process.py · Whisper Turbo + Diarization 完整代码
import whisper
from pyannote.audio import Pipeline

# 1. ASR · Whisper-Large-v3-Turbo（2024.10 · 比 v3 快 8x）
asr = whisper.load_model("large-v3-turbo")
result = asr.transcribe("interview.mp3", language="zh",
                        word_timestamps=True, condition_on_previous_text=False)
# result["segments"] · 含每段 + 字级时间戳

# 2. Diarization · pyannote 3.1
diar = Pipeline.from_pretrained("pyannote/speaker-diarization-3.1", use_auth_token="hf_xxx")
diarization = diar("interview.wav")   # 输出 [(start, end, speaker_label), ...]

# 3. 合并 · ASR 文字段 + speaker label
def merge(asr_segs, diar_segs):
    merged = []
    for seg in asr_segs:
        mid = (seg["start"] + seg["end"]) / 2          # 找 mid 落在哪个 diar segment 里
        spk = next((s for s in diar_segs if s.start <= mid <= s.end), None)
        merged.append({
            "start": seg["start"], "end": seg["end"],
            "text": seg["text"], "speaker": spk.label if spk else "?",
            "words": seg["words"],
        })
    return merged
```

## MULTIMODAL RAG ARCHITECTURE：3 架构

不要纠结「Unified embedding」，混用才是稳的。

| 架构 | 做法 | 优劣 | 场景 |
| :--- | :--- | :--- | :--- |
| A · 全转文本 | PDF / 图 / 音视频 → markdown / 文字，一个 vector store 存全部 | 简单（跟 L01-L04 同），但丢图细节、无 cross-modal 搜 | 早期 + 文为主 |
| **B · 多 Modal Store** ★ | 文 / 图 / 音独立 store，Query 时各 store 并行检索，结果一起喂 multimodal LLM | 灵活 + 模态间不污染 | **2026 生产推荐** ★ |
| C · Unified Embedding | CLIP / SigLIP 统一空间，图 + 文同 store，跨模态搜 | cost 高 + 效果不如专 model | 电商 / 以图搜图 |

## COLPALI · 2024-2025：把整页 PDF 当图像检索

2024.7 论文 → 2025 工业落地，视觉文档检索的范式革新，完全跳过解析。

- **原理**：PDF 不解析，整页截图当 image，VLM embedding 检索。传统是 `PDF → Marker → markdown → chunk → text embedding → 检索`；ColPali 是 `PDF 每页直接截图 → PaliGemma + Late-Interaction patch embedding → ColBERT-style MaxSim 检索`。
- **效果**：论文场景 ViDoRe benchmark，NDCG@5 大幅超过传统流水线，不必担心 IDP 解析错误，整页视觉一起检索。

| 版本 | 基础模型 | 关键能力 | 场景 | 2026 状态 |
| :--- | :--- | :--- | :--- | :--- |
| ColPali | PaliGemma-3B | 英文 PDF / 论文，ViDoRe SOTA | 英文学术 | 开源主流 ★ |
| **ColQwen2** | Qwen2-VL-2B | 中文 PDF，2025 商业落地友好 | 中文 / 多语言 ★ | 中文首选 |
| ColInternVL | InternVL-2 | 通用，学术活跃 | 研究 | 探索性 |

## ANTI-PATTERNS：多模态处理 5 个反模式

反模式 1（直接上 VLM）是 2024-2025 最高发，多花 10x 成本、效果还更差。

| 反模式 | 具体表现 | 后果 | 正确做法 |
| :--- | :--- | :--- | :--- |
| 直接上 VLM 处理 PDF | 所有 PDF 都截图给 Claude / GPT | cost 10-30x、中文 OCR 更差 | IDP 主干 + VLM 兜底 |
| 音频不 Diarization | 多人会议混成一段文字 | 客户问追溯找不到说话人 | pyannote 必上 / SaaS Speaker |
| 视频不切 chunk | 一长串 transcript 没法用 | 检索召回质量极差 | 按话题 / 场景切 chunk |
| 图 embedding 用 text | 图片描述塞进 text store | 跨模态搜效果差 | CLIP / SigLIP / Jina-CLIP |
| 不存原文件 | 只存 markdown、删原 PDF | 客户问追溯回不去原文 | 原文件 + chunk 并存 versioning |

**踩坑提醒**：真实账单——客户 10 万页 PDF，直接 Claude Vision，月账单 $40K。切到 Marker + Claude 兜底 5% 疑难，月账单降到 $3.5K、效果还好，这是 2026 多模态的 ROI 头号优化。

## RECAP：本讲判断

1. 音视频 / 图就是「另一种非结构化数据」，套 L01-L04 范式、换物理载体，90% 工作量可复用。
2. 三轨道（Audio / Vision / Time-Stamp）独立处理 + segment_ts 对齐，比 monolithic 强 5x。
3. AudioSegment 与 EvidenceAnchor 同构——音频的 segment_ts 就是文档的 page+bbox。
4. Vision LLM 选型：中文走 Qwen2.5-VL / Gemini，英文走 Claude / GPT-5，视频走 Gemini 2.5。
5. Multimodal RAG 推荐架构 B（多 Modal Store + Hybrid），别强行 Unified embedding。
6. 头号烧钱坑是直接上 VLM 处理 PDF（贵 100-300 倍），IDP 主干 + VLM 5% 兜底才对。
7. 下一讲：把整周的 Parse / Chunk / Evidence / Quality / Multimodal 收口成一座非结构化数据工厂。
