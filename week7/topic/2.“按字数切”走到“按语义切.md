# 2. 从「按字数切」走到「按语义切」

## 核心目标与背景

- **定调**：切片决定 RAG 的召回上限。切片不是「文字处理」，是「语义对齐」——这是 RAG 工程链路里 ROI 最高的一个动作。
- **演进**：从「按字数 / token 暴力切」升级到「按结构 / 语义切」，并叠加 2024 两大游戏规则改变者：Late Chunking 与 Contextual Retrieval。
- **主要任务**：掌握 4 种切片策略、粒度调参方法论、6 种特殊内容处理、代码 AST 切片与代码 Embedding 选型。

## 本课聊 5 件事

1. 4 种主流切片策略 + Granularity 调参方法论。
2. Late Chunking（Jina 2024.09）+ Contextual Retrieval（Anthropic 2024.09）。
3. 6 种特殊内容必须单独处理（表 / 代码 / 公式 / 引用 / 列表 / 对话）。
4. 代码 AST Chunking，TreeSitter，不能按行切。
5. Voyage code-3 / BGE-Code，代码 Embedding 选型。

## WAR STORY：合同被「拦腰切断」

接过的项目，团队上来就 `text.split`（每 512 字 chunk_size，overlap 50），Demo 跑通。

上线后客户问「合同第 5 条第 3 款怎么规定」——检索回来的 chunk 把「第 5 条」切到上一段尾巴，「第 3 款」切到下一段开头，LLM 看到的是断裂的语义碎片。

**那不是 chunk 算法笨，是「按字数切」根本不顾结构，RAG 召回上限被切片就锁死了。**

## WHY CHUNKING MATTERS：切片决定召回上限

一个绕不开的简单逻辑：正确答案没被切到任何 chunk → ANN 检索召不回 → Reranker 排不到 → LLM 看不到 → RAG 必错。

> Reranker 是「从候选里选最好」，候选里没正确答案，神仙也无解。

反过来，切片把上下文切碎，正确答案在 chunk 里但和上下文断裂，LLM 看到孤立信息，信心低，出现幻觉或拒答。

| 切错的症状 | 根因 | 怎么修 |
| :--- | :--- | :--- |
| 答案漂 · 上下文断裂 | 按字符 / 按 token 暴力切 | 按结构（heading / sentence）切 |
| 财务数字串到错的表格 | 表格被 chunk 拦腰切 | 表格独立 chunk + 描述双轨 |
| 代码理解错 · 函数被切 | 按行切代码 | TreeSitter AST 按函数切 |
| 多轮对话理解错 | 一问一答 chunk 分开 | 保留 turn 元信息或合并 |

## 4 STRATEGIES：4 种主流切片策略

没有银弹，但有默认起点——结构感知是 90% 场景的最优解。

| 策略 | 做法 | 场景 |
| :--- | :--- | :--- |
| ① Fixed-size | 每 N 字符 / token 切 + overlap 50-100，简单通用不顾结构 | 纯散文 / 邮件（默认起点但低质量） |
| ② **Structure-aware** ★ | 按 heading / section 切，保留 H1/H2 层级 metadata | PDF / Markdown / HTML，IDP 输出后直接吃，**90% 场景最佳起点** |
| ③ Semantic | LLM-based 按主题切 / embedding 相邻度切，语义连贯但 cost 高 | 论文 / 长报告，5% query 走 Agentic |
| ④ **Late Chunking** ★ | Jina 2024.09 创新，先 embed 整段再 chunk，上下文不丢 | 长文档，**2026 检索 SOTA** |

**判断**：Structure-aware 起步 + Late Chunking 长文档 + Semantic 高价值场景，Fixed 只在 demo。

## GRANULARITY TRADE-OFF：切片粒度的权衡

默认从 300-600 token 起步，按 retrieval@k 评测反推，不要凭感觉调。

| 粒度 | chunk_size (token) | 优势 | 劣势 | 场景 |
| :--- | :--- | :--- | :--- | :--- |
| 超细 | 50-150 | 检索精度高、命中精确 | LLM 看到孤立信息易幻觉 | 事实问答 / FAQ |
| **细** ★ | 300-600 | 默认起点、平衡 | 复杂查询上下文可能不够 | 通用 RAG ★ |
| 中 | 600-1200 | 上下文充足、推理类好 | 检索精度略低 | 论文 / 长报告 |
| 粗 | 1200-2000 | 完整段落、推理理解强 | 召不回精确事实 | 法律 / 合同 |
| 超粗 | > 2000 | 一整页、不损上下文 | 检索召回质量差 | LongContext + RAG 混合 |

**调参方法**：先 300-600 默认，Recall@5 拉到 80% 后再换粒度对比，不要凭感觉。

## LATE CHUNKING：Jina AI 2024.09 游戏规则改变者

颠倒「先切再 embed」→「先 embed 再切」，上下文不丢。

- **原理**：传统先切 chunk → 每个 chunk 单独 embed → 每 chunk 失去整段上下文。Late Chunking 整段先过 encoder 拿 token-level embedding → 再按 chunk 边界对 token-embedding 池化（mean pool）。
- **效果**：长文档检索 nDCG@10 平均 +9%（Jina paper），对「中间引用前文」类 chunk 修复显著。

```python
# Late Chunking · Jina v3 / Embed v4 实现
from transformers import AutoModel
model = AutoModel.from_pretrained("jinaai/jina-embeddings-v3", trust_remote_code=True)

# 1. 整段过 encoder 拿 token-level embedding
inputs = tokenizer(long_text, return_tensors="pt", truncation=True, max_length=8192)
outputs = model(**inputs, output_hidden_states=True)
token_emb = outputs.last_hidden_state   # [1, seq_len, dim]

# 2. 按 chunk 边界（预先用 structure-aware splitter 算）对 token-emb mean pool
chunk_embs = []
for (start, end) in chunk_boundaries:
    chunk_embs.append(token_emb[0, start:end, :].mean(dim=0))
```

**适用**：长文档（> 2k tokens）、chunk 间互相引用强、论文 / 法律 / 技术文档。Jina v3 / Voyage v3 原生支持，OpenAI embedding 无法。**ROI 高**：不动切片机制，只换 embed 顺序，nDCG +9%。

## CONTEXTUAL RETRIEVAL：Anthropic 35-49% 精度提升

不动切片机制，给每个 chunk 自动加「全文摘要前缀」。

```python
# Anthropic Contextual Retrieval · 2024.09 paper
import anthropic
client = anthropic.Anthropic()

CONTEXTUALIZE_PROMPT = """
<document>{doc}</document>
下面是上述文档中的一个 chunk：
<chunk>{chunk}</chunk>
请用一句话总结这个 chunk 在文档整体中的上下文 / 位置 / 所属节，以便检索时帮模型理解。
只输出简短描述，不要复述 chunk 内容。
"""

def contextualize_chunk(doc: str, chunk: str) -> str:
    resp = client.messages.create(
        model="claude-haiku-4-5-20251001",   # Haiku 跑省钱
        max_tokens=128,
        system=[{"type": "text", "text": CONTEXTUALIZE_PROMPT,
                 "cache_control": {"type": "ephemeral"}}],   # 全文走 cache
        messages=[{"role": "user", "content": CONTEXTUALIZE_PROMPT.format(doc=doc, chunk=chunk)}],
    )
    return resp.content[0].text

# 入库时把 context_prefix 拼到 chunk 前面再 embed
context = contextualize_chunk(full_doc, chunk)
augmented = f"{context}\n\n{chunk}"
embedding = embedder.embed(augmented)
```

- **效果数据**：Recall@20 错误率 -35%；配合 Contextual BM25 错误率 -49%。
- **关键 Prompt Caching**：整篇文档进 cache（read 价 0.1x），一份长文档跑 200 chunk，cache 命中后 cost 极低，整本 1000 chunk 书 $5 以内。
- **何时不上**：chunk 已自带 H1/H2 上下文、短文档（< 5k token）。

## SPECIAL CONTENT：6 种特殊内容必须单独处理

通用切片器在 Demo 阶段够用，生产里至少要 6 个分支。漏一个，上线后扛 30% bad case。

| 类型 | 错误做法 | 正确做法 | 工具 |
| :--- | :--- | :--- | :--- |
| 表格 | 混进文本流 cut | 独立 chunk + 描述双轨 | LlamaParse / Camelot |
| 代码 | 按行切 | AST 按函数切 | TreeSitter ★ |
| 公式 | 抽成乱码 | 保 LaTeX / MathML | Mathpix / Marker |
| 列表 / 枚举 | 断在 item 中间 | 整列表当原子 chunk | Marker 自带 |
| 对话 / FAQ | 问答分开 | turn-aware 保 Q+A 一组 | 自定义 |
| 引用 / 脚注 | 混入正文 | 入 metadata，不混正文 | GROBID / Marker |

## CODE AST CHUNKING：代码不能按行切

代码必须按 AST 结构切。

```python
# pip install tree-sitter tree-sitter-languages
import tree_sitter_languages
from tree_sitter import Parser

parser = Parser()
parser.set_language(tree_sitter_languages.get_language("python"))  # Java/Go/TS/Rust 同接口

def chunk_by_ast(code: str, max_size: int = 1500) -> list[dict]:
    tree = parser.parse(code.encode())
    chunks = []
    def walk(node, parent_ctx=""):
        if node.type in ("function_definition", "class_definition"):  # 函数 / 方法独立 chunk
            content = code[node.start_byte:node.end_byte]
            name = next((c.text.decode() for c in node.children if c.type == "identifier"), "?")
            if len(content) <= max_size:
                chunks.append({
                    "type": node.type, "name": name,
                    "content": parent_ctx + content,   # 含 import 上下文
                    "start_line": node.start_point[0],
                    "end_line": node.end_point[0],
                })
            else:  # 大类拆方法
                for c in node.children:
                    walk(c, parent_ctx=content[:200] + "...\n\n")
        for c in node.children:
            walk(c, parent_ctx)
    walk(tree.root_node)
    return chunks
```

**4 种切法**：按函数（每函数一 chunk）、按类（大类拆方法）、按文件（小文件整切）、保 imports + class 上下文作 anchor。**TreeSitter 优势**：50+ 语言一统、比 ast 模块通用、GitHub Code Search 同款引擎。**2026 共识**：代码 chunking 必须 AST，单 chunk 上限 1500-2000 token，函数 > max_size 时拆方法，永远保 imports + 注释。

## REAL ARTIFACT：结构感知切片实现

Marker 输出 markdown → 按 heading + 长度双约束切。

```python
# pipelines/chunker/structure_aware.py
from typing import Iterator
from langchain_text_splitters import MarkdownHeaderTextSplitter

def structure_aware_chunks(md: str, chunk_size: int = 500, overlap: int = 50) -> Iterator[dict]:
    # Phase 1 · 按 heading 切大段
    header_splitter = MarkdownHeaderTextSplitter(
        headers_to_split_on=[("#", "H1"), ("##", "H2"), ("###", "H3")],
    )
    sections = header_splitter.split_text(md)
    for sec in sections:
        text = sec.page_content
        meta = sec.metadata               # {"H1": "...", "H2": "..."}
        # Phase 2 · 大段内按句子 + 长度切小 chunk
        if len(text) <= chunk_size:
            yield {"content": text, "metadata": meta}
        else:
            sents = sentence_split(text)   # 用 PySBD / spaCy
            buf, buf_len = [], 0
            for sent in sents:
                if buf_len + len(sent) > chunk_size and buf:
                    yield {"content": " ".join(buf), "metadata": meta}
                    buf = buf[-overlap:]   # 滑窗 overlap
                    buf_len = sum(len(s) for s in buf)
                buf.append(sent); buf_len += len(sent)
            if buf:
                yield {"content": " ".join(buf), "metadata": meta}
```

**4 个工程要点**：2 phase 切（先 heading 后句子）；heading 入 metadata = chunk 上下文；overlap 50 token 滑窗防句子被切碎；sentence_split 用 PySBD（比 nltk 准、中英双语）；chunk metadata 含 H1/H2/H3 + source / page / bbox，后续 EvidenceAnchor 直接用。

## CHUNKING ECOSYSTEM 2025：4 个值得关注的实现

切片是「检索质量 + 业务结构 + token 经济」三方权衡。

- **LangChain Splitters**：Markdown / HTML / Recursive Char / Code，90% Python 项目用它起步，跟 LangGraph 天然集成。
- **LlamaIndex NodeParser**：Sentence / Semantic / Hierarchy / Window 4 种，Agentic RAG 友好。
- **Late Chunking · Jina**（2024.09）：embedding 顺序颠倒，长文档 nDCG +9%，不动切片机制 ROI 高。
- **Contextual Retrieval · Anthropic**（2024.09）：chunk 自动加全文摘要 prefix，Recall 错误率 -35-49%，Haiku 4.5 + Prompt Caching = 整本书 $5 以内。

**2026 推荐组合**：Structure-aware 起步 + Late Chunking 长文档 + Contextual Retrieval 高价值场景。

## HOW TO TUNE：看评测指标调切片，不凭感觉

| 指标 | 作用 | 调参方向 | 阈值 |
| :--- | :--- | :--- | :--- |
| Recall@5 | 前 5 个含正答比例 | 太低 → 加 chunk_size / 加 overlap | > 0.80 |
| Recall@20 | 召回兜底 | 太低 → 换 embedding / 加 hybrid BM25 | > 0.92 |
| MRR | 正答平均排名 | 低 → 上 reranker | > 0.65 |
| Faithfulness | 答案是否扎根 chunk | 低 → 增大 chunk size / 加 ctx | > 0.85 |
| Context Precision | 上下文相关度 | 低 → 减 chunk size / 加 rerank | > 0.80 |
| Latency | 检索 + 生成总耗时 | 高 → 减 top-k / 加 cache | P95 < 3s |

**调参方法论**：先 default → eval 跑 200 sample → 看哪个指标失守 → 反推调哪个参数，不要瞎试。
