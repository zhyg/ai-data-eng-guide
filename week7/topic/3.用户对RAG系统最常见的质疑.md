# 3. 客户对 RAG 系统最常见的两个质疑

## 核心目标与背景

- **定调**：客户对 RAG 最常见的两个质疑是「这答案哪来的」和「让我看原文」。能不能答得上来，是生产 RAG 的及格线。
- **演进**：从「只给答案」升级到「答案 + 可定位 + 可解释 + 可审计」的证据链（Evidence Chain）。
- **主要任务**：掌握 5 层证据等级、EvidenceAnchor schema、Anthropic Citations API、5 层端到端无损传递与审计治理入口。

## 本课聊 5 件事

1. 5 levels of evidence，你的 RAG 在哪一档。
2. EvidenceAnchor schema，Identity / Location / Context 必含 3 组字段。
3. Anthropic Citations API 2025.1 GA，字符级定位、幻觉率 -50%。
4. 5 层端到端无损传递，90% 团队在 Index → Retrieve 这步丢字段。
5. 审计 + Governance，Week 14 治理周的入口。

## WAR STORY：金融客户评审现场卡 15 秒

去年带团队给一家头部券商做 RAG Demo，PMF 都过了，客户技术评审会上突然问：「这个答案是来自哪份研报？能不能给我跳到那一页？」

- 团队当场卡 15 秒，因为 chunk 入库时把 page_no / bbox 都丢了，只剩纯文字，根本回不到原文。
- 那一刻评审没过，6 个月的功夫归零。

**根因是「证据链」从一开始就没设计。**

## 4 / 5 LEVELS OF EVIDENCE：你的 RAG 在哪一档

生产级及格线 = L3 段落 + 坐标。低于 L3，客户对系统建立不起长期信任。

| 等级 | 能力 | 颗粒度 | 场景 | 客户接受度 |
| :--- | :--- | :--- | :--- | :--- |
| L0 | 只给答案、无 source | 无 | Demo only | 低 · 必拒 |
| L1 | 给文档名（source field） | 文档级 | 内部使用 | 低 |
| L2 | 页码 + section 标题 | 段落级 | 内部知识库 | 中 |
| **L3** ★ | 页码 + bbox + 段落引用 | 坐标级 | To B 生产线 | 高 · **及格线** |
| **L4** ★★ | 字符级 + Citations API | token 级 | 金融 / 法律 / 医疗 | 极高 · 合规必备 |

**判断**：2025 起 To B RAG 必须 L3 起步，金融 / 法律 / 医疗必须 L4 + Anthropic Citations。

## EVIDENCEANCHOR FIELDS：必含 3 组字段

Identity 能回原资料、Location 能定位、Context 能解释。

| 组 | 字段 | 缺失后果 |
| :--- | :--- | :--- |
| **Identity · 身份** | doc_id（原文档主键）、version_id（避免文档更新丢链）、chunk_id（本 chunk 唯一 id）、source（如 "财报2025_Q3.pdf"） | 客户问追溯不到 |
| **Location · 定位** | page_no（页码）、bbox（[x0,y0,x1,y1] 坐标）、span_start / span_end（字符级偏移） | 前端没法高亮 |
| **Context · 解释** | heading_path（H1 > H2 > H3）、rerank_score、retrieval_method（hybrid / vector）、release_id（哪次发布命中） | 审计断链 |

**踩坑提醒**：见过的项目，team 为了 ANN 性能把 metadata 全砍掉，上线后 4 个月发现客户全在投诉「答案不可信」。

## REAL ARTIFACT · EvidenceAnchor

前端拿到这个结构，一键高亮原 PDF + 显示置信度热力图。

```python
# services/rag_api/models.py · Pydantic Schema
from pydantic import BaseModel, Field
from typing import Literal

class EvidenceAnchor(BaseModel):
    # === Identity · 必填 ===
    doc_id:      str = Field(..., description="文档主键")
    version_id:  str = Field(..., description="文档版本，避免更新断链")
    chunk_id:    str = Field(..., description="chunk 唯一 id")
    source:      str = Field(..., description='e.g. "财报2025_Q3.pdf"')
    source_type: Literal["pdf", "html", "audio", "video", "code"] = "pdf"

    # === Location · 必填 ===
    page_no:    int = Field(..., ge=1, description="页码 1-based")
    bbox:       list[float] = Field(..., min_items=4, max_items=4)   # [x0,y0,x1,y1]
    span_start: int = Field(..., description="字符级起始")
    span_end:   int = Field(..., description="字符级结束")

    # === Context · 必填 ===
    heading_path:     list[str] = Field(default_factory=list)   # ["H1", "H2", "H3"]
    rerank_score:     float = Field(..., ge=0.0, le=1.0)
    retrieval_method: Literal["hybrid", "vector", "bm25", "graph"] = "hybrid"
    release_id:       str = Field(..., description="哪次 release 命中")

    # === 内容 ===
    chunk_text: str
    confidence: float = Field(..., ge=0.0, le=1.0)
```

## ANTHROPIC CITATIONS API：模型自己输出引用

模型自己输出引用片段，不再靠 prompt hack 让 LLM 说出来源。

```python
# Anthropic Citations API · 2025.1 GA
import anthropic
client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-5-20251022",
    max_tokens=1024,
    messages=[{
        "role": "user",
        "content": [
            # 关键 · documents 块 · citations.enabled=True
            {"type": "document",
             "source": {"type": "text", "media_type": "text/plain", "data": chunk_1.text},
             "title": chunk_1.source,
             "context": chunk_1.heading_path,
             "citations": {"enabled": True}},
            {"type": "document",
             "source": {"type": "text", "data": chunk_2.text},
             "title": chunk_2.source,
             "citations": {"enabled": True}},
            {"type": "text", "text": "请基于上述文档回答..."},
        ],
    }],
)

# 答案里每一句话自动带 cited_text 段
for block in response.content:
    if block.type == "text":
        for citation in block.citations or []:
            print(citation.cited_text, citation.document_index)
```

- **关键能力**：模型字符级输出 citation（cited_text + document_index），不需要 prompt hack。
- **生产数据**（Anthropic 公开）：幻觉率 -50%、答案准确率 +18%、不增加延迟。
- **配套**：prompt cache 缓存 system，document 每篇 entry 单独标，`max_tokens` 不必加、开销持平。
- **2026 共识**：To B 必上 Citations API，OpenAI 也跟进（Structured Outputs + tool_use）。

## END-TO-END FLOW：5 层端到端无损传递

任何一层丢字段，下游全断。90% 团队在 Index → Retrieve 这步丢字段，为优化向量库性能把业务 metadata 丢了。

| 阶段 | 动作 | 必传字段 |
| :--- | :--- | :--- |
| **Parse** | 抽取 + 坐标 | doc_id / page_no / bbox / span 必入 |
| **Chunk** | 切片 + 元数据 | heading_path / version / chunk_id 入 metadata |
| **Index** | 向量化 + ANN | metadata 跟 vector 绑存，别砍 |
| **Retrieve** | 混检 + 排序 | rerank_score / retrieval_method 标 |
| **Generate** | LLM + 引用 | release_id / cited_text 回写 evidence |

**断链高发地**：Index 阶段为了 ANN 性能砍 metadata，上线后客户问追溯只能空白响应。
**2026 共识**：metadata 必跟 vector 同库存，pgvector / Milvus 2.5+ 都支持 hybrid metadata，不要为省 10% 内存丢业务字段。

## ANTI-PATTERNS：证据链 4 个反模式

存了不用 / 用了不对 / 评测不验证，这是三大主流坑。

| 反模式 | 具体表现 | 后果 | 正确做法 |
| :--- | :--- | :--- | :--- |
| 没存 bbox | chunk 只有文本 + page | 前端不能高亮原文 | Parse 阶段必入 bbox |
| 存了不传 | API 只给答案不给 evidence | 客户问追溯空白 | RAGAnswer 必带 evidence[] |
| 传了不对齐 | 答案 cite 第 1 chunk 但实际看的第 3 | evidence 假，比没有更危险 | Citations API 强对齐 |
| 评测不验证 | Eval 只看答案不看 citation | 上线后 30% citation 是错的 | Ragas citation_accuracy 必测 |

**踩坑提醒**：那个金融评审的结局——团队回去花 3 个月重做 evidence pipeline，Parse 阶段重新跑全量 PDF + 重建索引。客户后来签了，但 6 个月信任损耗，这种代价比技术成本高 10 倍。**从 day 1 就要设计 evidence。**

## INDUSTRY SIGNALS 2025：证据链已是 To B 合规基线

2025 起证据链不是「锦上添花」，是 AI 产品对 B 端的合规基线。

- **Anthropic Citations API**（2025.1 GA）：字符级 cited_text、幻觉率 -50%、模型原生输出引用，Claude Sonnet/Opus/Haiku 4.5 全线支持。
- **OpenAI Structured Outputs**（2024.8 + 2025 强化）：Pydantic schema 直传、citation 字段强约束，GPT-4o / GPT-5 都支持。
- **Perplexity Sonar**（2024-2025）：答案带 footnote 形式引用，C 端 Citations 体验的 reference 实现。
- **EU AI Act 合规**（2024.8 生效）：高风险 AI 强制 traceability，Citations 是合规基础，2026 中欧合规审查必看。

**判断**：L3 段落级是 To B 及格线，L4 字符级 + Citations API 是金融 / 法律 / 医疗硬要求。

## AUDIT + GOVERNANCE：证据链支撑全链路审计

证据链做对了，合规审计从「开会扯皮」变成「点几下出报告」。这是 Week 14 治理的入口。

```sql
-- governance/audit_query.py · 客户问「3 个月前那次回答凭啥」
SELECT
  answer.text,
  evidence.doc_id, evidence.version_id, evidence.page_no, evidence.bbox,
  evidence.chunk_text,
  answer.release_id,                                          -- 哪次 release 产出
  rel.prompt_version, rel.model_version, rel.index_snapshot   -- 原子绑定回查
FROM rag_answer_log answer
JOIN evidence ON answer.id = evidence.answer_id
JOIN releases rel ON answer.release_id = rel.release_id
WHERE answer.user_id = ? AND answer.created_at BETWEEN '2026-02-01' AND '2026-02-15'
ORDER BY answer.created_at;
-- 一条 SQL 拉出完整链路：审计员从 answer 反推到 doc 哪页 / 哪个 chunk / 哪版 prompt / 哪版索引
```

**审计场景 4 类**，日常都用证据链支撑：

- **客户投诉**「答案错了」→ 反查 chunk + version 看哪一步出问题 → 入 bad case。
- **监管检查**「3 个月前你怎么答的」→ release_id 锁定 + evidence 还原。
- **内部 review**「为什么改了答案」→ 对比新旧 release 的 prompt/model/index 差异。
- **合规年审**：EU AI Act 要 traceability，这一条 SQL 出 90% 报告。

## RECAP：本讲判断

1. 客户两大质疑「答案哪来的」「让我看原文」，答不上来就是 Demo。
2. 证据链分 5 档，To B 生产及格线是 L3（页码 + bbox + 段落引用）。
3. EvidenceAnchor 必含 Identity / Location / Context 三组字段，缺一组就断链。
4. Anthropic Citations API 让模型字符级输出引用，幻觉率 -50%、不增加延迟。
5. 证据链要 5 层端到端无损传递，断链高发地在 Index 阶段砍 metadata。
6. 证据链是全链路审计与合规（EU AI Act）的地基，必须 day 1 设计。
