# 6. 非结构化数据工程实践：把非结构化资产变成可治理的数据资产

## 核心目标

本节是第七周的代码实践课，目标是把企业中的非结构化资产转化为可治理的数据资产。需要解决四个问题：

1. 资产来源及其验证
2. 不同模态的解析能力
3. 如何追溯原文页面、时间戳或对象
4. 确保处理后的数据能安全用于第八周的检索、重排等用途

## 整体处理流程

整个非结构化数据资产的处理流程分为六个阶段，结构清晰：

```
资产清单 → 验证脚本 → 源文档加载 → IDP 智能解析 → 切块 + 证据锚点 → 质量门禁 → 入库
```

- **资产清单（Manifest）**：登记所有待处理的非结构化资产，包括 PDF、图像、音频、视频、文本
- **验证脚本**：在入口处校验资产的完整性和可信度
- **源文档加载**：读取本地文件或输入路径，转换为结构化文档，计算源指纹
- **IDP 智能解析器**：根据资产类型自动路由到不同的解析方式
- **切块与证据锚点**：将解析结果切分为数据段，并为每个段生成可追溯的锚点
- **运行包裹（Parcel Run）**：将以上步骤串联为一个完整的运行流程

## 资产清单的关键字段

资产清单是整条链路的起点，每个资产记录包含以下关键字段：

| 字段 | 作用 |
| :--- | :--- |
| **source_id** | 后续 track 和证据锚点追溯到具体资产的唯一标识 |
| **asset_type** | 决定解析适配器（Adapter）的路由方式 |
| **checksum_256** | 确保处理的原始文件和声明的文件一致，防止篡改 |
| **PI 状态** | 决定数据是否能进入索引的全链路 |
| **契约引用** | 关联契约测试，确保输出满足下游约束 |

音视频资产还会额外携带 transcript_object_path（转录产物路径），图片类资产携带 OCR_text_path。视频中还包含关键帧的 OCR 路径。在企业实践中，通常不会直接使用原始文件，而是接收上游 ASR、OCR 或 VLM 产出的半成品。

## 源加载器与质量门禁

源加载器的职责是将本地或清单中的输入路径转换为标准路径。它只确认文件是否存在、读取内容并计算指纹，不执行复杂的业务逻辑。文件不存在时，根据配置产生警告或强制中断。

**质量门禁**根据校验结果阻止脏数据进入后续流程。如果输入不可信，在入口处就暴露问题。文档、跟踪、锚点等所有环节的源头都需要通过这道关卡。

## 多模态解析引擎

解析引擎根据资产类型自动路由到不同的解析方式：

| 资产类型 | 解析方式 |
| :--- | :--- |
| **PDF** | 优先使用标记方法（Marker），其次使用其他解析技术 |
| **图像** | 通过 OCR 处理 |
| **音频** | 转录流程 |
| **视频** | 视频转录 + 关键帧提取 |

解析后输出带有回溯信息和存储记录的数据段，再通过结构化感知的方法进行切分。

## 证据锚点：本周核心

证据锚点是第七周内容的核心。它确保模型引用资料时，系统能够识别资料来源的具体位置和时间戳。在生产环境中，这是必须遵守的规范。

不同资产类型的锚点设置方式：

| 资产类型 | 锚点位置 |
| :--- | :--- |
| **PDF** | 页面配置对象（page_no + bbox） |
| **图像** | 特定对象中的定位信息 |
| **音视频** | 时间戳和关键帧（segment_ts + speaker_id） |

锚点记录的元信息指纹包括页码、时间戳、cabinet 标识和 release_id 等关键信息。这些字段在后续的证据引用中发挥作用，必须在这一阶段固定下来。

## 同构思维：结构化与非结构化同等重要

第二周到第六周完成了结构化数据的处理，第七周将非结构化数据处理压缩到一周完成。这并不意味着非结构化数据不重要，而是因为前六周打下的基础（约 80%）可以直接复用。

处理的核心差异在于：

| 维度 | 结构化数据 | 非结构化数据 |
| :--- | :--- | :--- |
| 解析 | Schema 已知，直接读取 | 需要 IDP/OCR/ASR 抽取结构 |
| 切分 | 按行/按表 | 按章节/按话题/按场景 |
| 追溯 | 行号/主键 | 页码/时间戳/bbox |
| 质检 | 字段级校验 | 覆盖率/WER/锚点完整性 |

链路的底层范式一致：契约验证、质量门禁、入库流程都可以复用。接到新模态项目时，先问"它的解析、切分、证据、质检分别是什么"，大部分工作量可复用。

## 实验要点

实验课演示了完整的端到端流程：

1. **环境准备**：确保环境配置正确，启动后台服务
2. **契约测试**：运行契约测试验证输出满足约束，如缺少锚点或页面配置的问题应在契约阶段暴露
3. **Parcel Run**：设置解析方式（自动化）、切块策略（章节感知）、release_id 等参数，串联整个流程
4. **报告生成**：生成质量检测报告，包括切分计数、摘要文件、第八周的门禁质检记录
5. **入库验证**：将数据写入数据库，通过查询验证证据锚点的完整性
6. **回归测试**：全量运行测试确保流程无误

关键注意事项：
- 去掉 dry-run 参数后会导致真实数据入库，操作时需谨慎
- 先查看效果再决定是否入库，预防脏数据的成本远低于事后清理
- 链路可灵活复用，支持自定义输入和本地文件，不依赖固定路径

## 通向 Week 8

经过本周处理后的产出，即带有证据锚点、通过质量门禁、绑定了 release_id 的 chunk 与索引，就是下周 Week 8 RAG 服务化的直接输入。工厂建好，服务化才有可信、可追溯的原料。

## 本讲小结

1. 非结构化数据工程的核心任务是把 PDF、图像、音视频、文本转化为可治理的数据资产。
2. 处理流程：资产清单 -> 源加载验证 -> 多模态解析 -> 结构化切块 -> 证据锚点 -> 质量门禁 -> 入库。
3. 证据锚点是本周核心，确保模型引用时能定位到原文的具体页面、时间戳或对象。
4. 质量门禁在入口处阻止脏数据，预防成本远低于事后清理。
5. 同构思维让结构化和非结构化数据共用底层范式，新模态项目大部分工作量可复用。
6. 链路设计灵活，支持自定义输入，不依赖固定路径。
