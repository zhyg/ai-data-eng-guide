# 4. 切片质检：50-200 份样本 + 反例库 + CI 门禁

## 核心目标与背景

- **定调**：切片质检是专业团队的标准操作，不是「上线后再说」。RAG 质检不是「数据质检的延伸」，是「语义质检的新建」。
- **演进**：从 BI 时代的 not_null / unique 升级到 RAG 时代的完整性 / 噪声 / 证据 / 连贯 4 维语义质检 + CI 门禁。
- **主要任务**：掌握 4 大质检维度、抽样策略、6 个量化指标 + CI 门禁、Drift Detection、PDF 增量更新。

## 本课聊 5 件事

1. 质检 4 大维度：完整性 / 噪声 / 证据 / 连贯，每维都有指标 + 阈值。
2. 70% 分层 + 20% 困难 + 10% 对抗，抽样策略的最优组合。
3. 6 个量化指标 + CI 门禁，`gate_decision = block` 时阻断索引构建。
4. Drift Detection 3 监控点，上线后情况会变，早发现早修。
5. PDF 增量更新策略：diff → byte range → chunk 级更新。

## WAR STORY：「先上线试试」停服 36 小时

前年做过一个客户案例，团队抢工期，切片没做质检直接上线，上线 3 天客户反馈「答案乱说话」。

- 排查后发现：Marker 抽 PDF 时 5% 的页面 layout 解析错（多栏识别失败），下游 chunk 噪声严重，幻觉率 18%。
- 回滚 + 重新切 + 重建索引，停服 36 小时，外加客户信任损耗。

**如果上线前跑 100 sample 抽样，这种事故根本不会发生。**

## QUALITY AS GATE：质检是 CI 门禁的工程流程

质检不是「看几个 chunk」。BI 时代的 not_null / unique 在这里不够用。

| 维度 | BI 时代质检 | RAG 时代质检 | 差异 |
| :--- | :--- | :--- | :--- |
| 对象 | 表格 / 行 / 列 | chunk / 文档 / 多模态 | 颗粒不同 |
| 指标 | not_null / unique / range | 完整性 / 噪声 / 证据 / 连贯 | 语义化 |
| 工具 | GreatExpectations / Soda | Ragas / DeepEval / Phoenix | 专用 |
| 评测者 | 人工 + 规则 | LLM-as-Judge + 规则 | 半自动化 |
| 抽样 | 全表跑 / 抽样跑 | 分层 + 困难 + 对抗 | 更聪明 |
| 门禁 | block 写入 | gate_decision = block 索引 | 同思路 |
| 频率 | 每次写 | 每次切 + 每天 drift | 加频 |

**判断**：RAG 质检不是「数据质检的延伸」，是「语义质检的新建」，工具 + 思路都要换。

## 4 QUALITY DIMENSIONS：切片质检 4 大维度

少一个就有盲区，每个都有指标 + 阈值 + 不达标根因。

| 维度 | 关注点 | 指标 | 不达标根因 | 工具 |
| :--- | :--- | :--- | :--- | :--- |
| ① **完整性** | chunk 覆盖率、关键 entity 召回率 | coverage > 95% | 切片粒度太大 / split 错 | 自研 set diff |
| ② **噪声** | 页眉页脚 / 水印 / nav 混入、空 chunk / 重复 chunk | noise_ratio < 3% | IDP layout 还原弱 | 规则 + LLM 抽检 |
| ③ **证据** | EvidenceAnchor 必字段完整（bbox / page_no / version_id 不空） | evidence_complete = 100% | Parse 阶段砍字段 | Pydantic 强校验 |
| ④ **连贯** | chunk 句子结构完整、不切在句中 / 段中 | sentence_break_rate < 1% | splitter 算法过粗 | PySBD / LLM 抽检 |

## SAMPLING STRATEGIES：纯随机是错误起点

生产级 = 70% 分层 + 20% 困难 + 10% 对抗，这是平衡覆盖和聚焦的最优组合。

| 抽样类型 | 占比 | 怎么选 | 目的 |
| :--- | :--- | :--- | :--- |
| 分层抽样 | 70% | 按 doc_type（财报/合同/手册）+ 长度 + 章节均匀采样 | 保证覆盖代表性 |
| 困难样本 | 20% | 历史 bad case + LLM-judge 低分 + 复杂 layout 页 | 聚焦关键风险 |
| 对抗样本 | 10% | 边界 case + 极长 / 极短 / 极复杂表格 | 验证 robustness |
| 纯随机（错误） | 0% | 从全集随机抽 100 个 | 掩盖关键问题 |
| **抽样规模** | 50-500 | 小项目 50 / 生产 200 / 关键场景 500 | 准确性 + 成本平衡 |

**ROI 数据**（3 个 RAG 项目对比）：纯随机 100 sample 漏检率 25%；70/20/10 分层 200 sample 漏检率 4%。多花的 100 样本标注成本 $50，避免上线事故损失 $50K，ROI > 1000x。

## REGRESSION COMPARISON：版本对比，策略迭代不再靠赌

没有版本对比，改切片策略每次都是「拍脑袋」。专业团队 4 种比法配合用。

| 比法 | 做法 | 阈值 / 工具 | 场景 |
| :--- | :--- | :--- | :--- |
| A · 指标对比 | 新旧版本同一 eval set，Recall@5 / MRR / Faith 对比 | 不能比旧版退步 3% · Ragas + pandas | PR merge 强制 |
| B · 答案 diff | 同一 query 跑新旧版，答案文字 diff + LLM-as-judge | Phoenix / 自研 | prompt 大改 |
| C · Bad case 回归 | 上线后积累的 bad case，每次大改都跑 | 老 bad case 都能修 · 自研 jsonl | 迭代必跑 |
| D · 流量 A/B | 5% canary 对比指标，CSAT / Citation Acc 监控 | 退化 → 自动回滚 · LaunchDarkly | 上线必走 |

**判断**：A 跑 CI、B 评 prompt、C 防回归、D 上线兜底，少一个都有盲区。

## REAL ARTIFACT · Quality Report

CI 阻断的核心：`gate_decision = block` 就停止索引构建。

```python
# pipelines/quality/report.py · Pydantic Quality Report
from pydantic import BaseModel, Field
from typing import Literal

class QualityReport(BaseModel):
    run_id:      str
    doc_set_id:  str
    sample_size: int = Field(..., ge=50, le=2000)
    sampling:    dict = Field(..., description='{stratified: 0.7, hard: 0.2, adversarial: 0.1}')

    # 4 大维度量化指标
    completeness_coverage:  float = Field(..., ge=0.0, le=1.0)   # 阈值 > 0.95
    noise_ratio:            float = Field(..., ge=0.0, le=1.0)   # 阈值 < 0.03
    evidence_complete_rate: float = Field(..., ge=0.0, le=1.0)   # 阈值 = 1.0
    sentence_break_rate:    float = Field(..., ge=0.0, le=1.0)   # 阈值 < 0.01

    # 检索质量
    recall_at_5:            float
    recall_at_20:           float
    mrr:                    float
    faithfulness_llm_judge: float   # LLM-as-Judge

    # 门禁决策 · CI 拿这个字段做决定
    gate_decision: Literal["pass", "warn", "block"]
    block_reasons: list[str] = Field(default_factory=list)

    # Drift 监控
    drift_metrics:      dict = Field(default_factory=dict)
    embedding_drift_kl: float = Field(default=0.0)
```

## 6 QUANTITATIVE METRICS：切片层的「生产准入条件」

跟 SLA 一样必须签，少一个就有盲区。

| 指标 | 定义 | 阈值（生产） | 不达标根因 | 修复方向 |
| :--- | :--- | :--- | :--- | :--- |
| Coverage | 内容入库覆盖率 | > 95% | split 漏 / 大段没切 | 检查 chunker 边界 |
| Noise Ratio | 噪声 chunk 占比 | < 3% | IDP layout 还原弱 | 换 Marker / 加 VLM 兜底 |
| Evidence Complete | metadata 完整率 | = 100% | Parse 阶段砍字段 | 强 Pydantic 校验 |
| Sentence Break | 句子被切碎率 | < 1% | splitter 算法过粗 | 用 PySBD 切 |
| Recall@5 | 检索召回@5 | > 80% | chunk_size 不对 / embedding 弱 | 调粒度 / 换 embedder |
| Faithfulness | 答案扎根 chunk 度 | > 85% | chunk 上下文不够 | 加 chunk_size / Late Chunking |

**踩坑提醒**：6 个指标必须 day 1 就建，少一个生产线总有死角，CI 强制阻断不达标的 PR。

不用造轮子，「切片质量」和「模型质量」被同等对待。

## QUALITY GATE FLOW：从生成到上线 5 个卡点

质检接进 CI/CD，切片质检不过，索引构建不允许启动。

| 卡点 | 动作 | 产物 |
| :--- | :--- | :--- |
| ① 切片完成 | 产物入暂存 | chunks.jsonl 暂存目录 |
| ② 抽样 + 标注 | 70/20/10 抽样 | 200 sample，人工 30 min |
| ③ Ragas 跑评测 | 6 指标 + LLM-Judge | quality_report.json，gate_decision |
| ④ CI 门禁判断 | block / pass / warn | gate=block → PR 阻断 |
| ⑤ 索引构建 | 通过后启动 | index_v2 ← release 绑定 |

**老司机说**：5 个卡点是「不可分」，漏一个就会有「不达标切片」混进索引。那个 36 小时事故，根因就是漏了步骤③（没跑评测），直接从切片到索引。后来把这 5 步硬编码到 GitHub Actions pre-merge gate，从此再没翻车。

## DRIFT DETECTION：上线后情况会变，3 个监控点

上线 1-3 个月效果一定会漂，早发现早修。

| 监控点 | 看什么 | 统计 | 阈值 | 修法 |
| :--- | :--- | :--- | :--- | :--- |
| ① Chunk 数量漂 | 新文档进库 vs 老文档比例 | 新增 chunk 数周环比 | > 50% 突变 → 告警 | 全量重切 + Ragas 复评 |
| ② Embedding 分布漂 | 新数据 vs 老索引向量分布 | KL divergence / MMD | > 0.15 → 告警 | re-embed 增量数据 |
| ③ Answer 质量漂 | Faithfulness / CSAT 周环比 | Faith < 0.85 → 告警 | 模型升级 / prompt 退步 | 回滚 release_id |

**工具**：Alibi / Evidently AI / Phoenix / Ragas + Slack hook。
**踩坑提醒**：没有 Drift Detection 的 RAG 上线 = 闭着眼开车，3 月内必出大事故，这是 2026 To B 硬要求。

## INCREMENTAL UPDATE：PDF 增量更新

文档改一行不能全量重切，chunk 级精准更新省 90% 算力。

```python
# pipelines/incremental/update.py · PDF 增量更新
import hashlib
from pathlib import Path

def incremental_update(new_pdf: Path, old_version_id: str):
    """文档改了 → diff → 受影响 chunk → 只重切 / 重 embed 这些"""
    old_chunks = chunk_store.load(old_version_id)
    new_md = marker_parse(new_pdf)                  # 重新 parse
    new_chunks_full = structure_aware_chunks(new_md)

    # 1. 算每个 chunk 的 content hash
    def chunk_hash(c): return hashlib.md5(c["content"].encode()).hexdigest()
    old_hashes = {chunk_hash(c): c for c in old_chunks}
    new_hashes = {chunk_hash(c): c for c in new_chunks_full}

    # 2. diff · 找新增 / 删除 / 不变
    added   = [new_hashes[h] for h in (set(new_hashes) - set(old_hashes))]
    removed = [old_hashes[h] for h in (set(old_hashes) - set(new_hashes))]
    kept    = set(new_hashes) & set(old_hashes)

    # 3. 只对增量做 embed + index
    new_embeds = embedder.embed_batch([c["content"] for c in added])
    index.insert(added, new_embeds)
    index.delete([c["chunk_id"] for c in removed])

    # 4. 记录 release_id 绑定
    new_version_id = f"v{int(old_version_id.lstrip('v'))+1}"
    return {"added": len(added), "removed": len(removed),
            "kept": len(kept), "version_id": new_version_id}
```

- **增量更新 ROI**：全量重切 1000 页 PDF（embedding $5、耗时 15 min）vs 增量只重切 50 chunk（$0.5、30 sec），节省 90% 算力 + 96% 时间。
- **关键设计**：content hash 是主键（md5 / sha256），hash 不变 → chunk 不变，不依赖易碎的 byte range。
- **版本绑定**：version_id 跟 release.yaml 绑，客户问追溯有据可查。
