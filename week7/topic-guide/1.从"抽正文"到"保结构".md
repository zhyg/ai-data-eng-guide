# 导学 01：为什么 PDF 抽出来的文字总是乱的？

## 一句话理解

PDF 处理的核心是保住文档的结构（多栏、标题、表格），而不只是把文字抽出来。

## 用生活打比方

这一周我们用"整理散乱档案归入图书馆"来串起所有概念。

假设你接手了一批企业的旧档案，要把它们整理好放进图书馆，让以后任何人都能快速找到需要的信息。

第一步是"登记入档"。如果你只是把每份文件的文字全部抄到一张纸上，一股脑塞进柜子，那章节没了、表格散了、两栏内容串到一行里了。以后有人来查"第三章利润表的净利润是多少"，翻出来的抄本里利润数字跑到了营收旁边，根本没法用。

正确的做法是按原件的格式登记：标题是标题、表格是表格、两栏分开记录、页码也标上。这样以后查阅时，信息的位置和结构都还在。

PDF 处理就是这个道理：你给 AI 的原料如果结构全丢了，AI 再聪明也答不对。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **智能文档处理（IDP）** | 一套能识别文档布局、保留结构的处理工具，比如能分清两栏、识别表格、提取标题层级，和单纯"抽文字"完全不同 |
| **5 阶段流水线（Load-Layout-Extract-Normalize-Persist）** | 文档处理的标准流程：加载文件、识别布局、抽取内容、清洗整理、入库保存，每步独立，少一步就有盲区 |
| **PDF 类型识别** | PDF 分三类：有文字层的（直接读）、扫描件（需要 OCR）、混合型（两种都有）。先判断类型再选工具，走对了省 70% 成本 |
| **布局检测（Layout Detection）** | 识别文档里的多栏排版、标题层级、表格边界、页眉页脚，这些信息对 AI 理解文档内容至关重要 |
| **坐标信息（bbox）** | 每段内容在原 PDF 上的位置坐标，有了它才能在前端高亮显示"答案来自这一页这个位置" |
| **表格双轨存储** | 表格同时保存 markdown 格式和一段文字描述，小表直接存 markdown，大表转成数据库表格用 SQL 查 |

## 避坑提醒

- **以为用 PyPDF 抽出文字就够了**：PyPDF 只做文字提取，不识别布局。两栏财报会被抽成单栏，表格数字会串到错误的列里。IDP 工具（比如 Marker、LlamaParse）才能保住结构。
- **以为表格和正文可以一起处理**：表格混进文本流后，AI 看到的是一堆空格分隔的数字，完全理解不了行列关系。表格必须单独处理，5 列以上的大表检索精度会明显下降。
- **以为页眉页脚无所谓**：页眉页脚混入正文后会被切进内容块，AI 可能把"第 42 页"当成答案的一部分，导致幻觉率上升约 40%。
- **以为所有 PDF 用同一个工具就行**：文字型 PDF 用 Marker 就够，扫描件必须走 OCR 工具（比如 Mistral OCR），混合型需要两条路线兜底。不做类型识别就处理，要么浪费钱，要么结果错。

## 一句话总结

> **给 AI 喂没有结构的文字，就像给人看打乱顺序的拼图碎片，看再久也拼不出原图。**
