# 导学 04：上线前不做质检，出了事再补来得及吗？

## 一句话理解

切片质检是上线前的强制检查，4 个维度、6 个指标、CI 门禁阻断不合格的内容入库。

## 用生活打比方

继续我们的图书馆档案整理。前面三步做完了：文件按格式登记、按章节拆分、每块都贴好了来源标签。现在要正式上架入库了。

但在上架之前，你得做一轮质检。就像超市上货前要检查保质期、包装完整性、标签是否贴对。如果直接把没检查的货往货架上摆，过期的、破损的、标签贴错的全混进去了，顾客买到问题商品导致退货投诉。

档案入库也是一样：有的块内容不完整、有的块混进了页眉页脚的噪声、有的块上的来源标签是空的、有的块句子被切到一半。这些问题必须在上架前拦住，等上线后客户发现再改，停服一天半都是轻的。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **4 维质检** | 检查切片质量的 4 个维度：完整性（有没有漏掉内容）、噪声（有没有混进页眉页脚等垃圾）、证据（来源标签是不是填全了）、连贯（句子有没有被切到一半） |
| **CI 门禁（gate_decision）** | 把质检接到自动化流程里，检查不通过就阻止入库。就像工厂的质检流水线，不合格的产品自动拦下来，不让出厂 |
| **分层抽样（70/20/10）** | 质检不是随机抽几个看看：70% 按文档类型均匀抽、20% 挑历史上出过问题的困难样本、10% 挑极端情况。纯随机抽 100 个漏检率 25%，这个组合 200 个漏检率只有 4% |
| **漂移检测（Drift Detection）** | 上线后持续监控 3 个指标：新增内容块数量有没有突变、向量分布有没有偏移、答案质量有没有下降。上线 1-3 个月效果一定会变，不监控就是闭眼开车 |
| **增量更新** | 文档改了一行不用全部重新处理。用内容指纹（hash）判断哪些块变了，只重新处理变化的部分，省 90% 算力和 96% 时间 |
| **LLM-as-Judge** | 用 AI 来辅助评估切片质量，比如判断答案是否扎根于切片内容。配合人工规则一起用，半自动化 |

## 避坑提醒

- **以为上线后再慢慢查质量就行**：如果跳过质检直接上线，布局解析错误导致的切片噪声很可能几天内就被客户发现。回滚加重建索引的停服时间往往以天计。上线前跑 100 个样本抽检就能拦住大部分问题。
- **以为随机抽几个看看就是质检**：纯随机抽样会掩盖关键问题。极端情况（超长表格、复杂布局）和历史 bad case 必须专门覆盖，分层抽样（70/20/10）能大幅降低漏检率。
- **以为质检只做一次就够了**：文档会更新，模型会升级，prompt 会调整，上线后效果一定会漂移。没有持续监控的 RAG 系统，大概率出事故。
- **以为 BI 时代的数据质检方法能直接搬过来**：BI 质检看的是"字段是不是空的、值在不在范围内"，RAG 质检看的是"内容完整吗、噪声多吗、证据链断没断、句子切碎没有"。对象不同、指标不同、工具不同，需要重新建设。

## 一句话总结

> **质检入 CI 门禁，不合格的切片禁止入库，这 5 个卡点少一个就会有问题块混进索引。**
