# 导学 02：为什么搜出来的答案总是"差一点"？

## 一句话理解

切片方式决定了 AI 能不能找到正确答案，按字数切就像随机撕书页，按结构切才能保证每块内容语义完整。

## 用生活打比方

继续我们的图书馆档案整理。上节课把文档按原始格式登记好了，这节课要解决另一个问题：文件太长，一份合同几百页，怎么让查阅的人快速定位到需要的那一段？

如果你把合同每 500 字剪一刀，不管剪到哪里就断开，结果"第五条"的开头在上一段末尾，"第三款"的内容在下一段开头。查阅的人拿到的是断裂的碎片，根本读不懂。

正确的做法是按章节、按条款拆分：第一章放一个文件夹，第二章放另一个，每个条款保持完整。这样查阅的人说"找第五条第三款"，直接拉出对应的文件夹就行。

切片就是给 AI 做这个拆分工作。拆得好，AI 能精准找到答案；拆得烂，正确答案被切碎分散在两个块里，再聪明的 AI 也拼不回来。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **结构感知切片（Structure-aware Chunking）** | 按文档的标题、章节、段落来切分，每块保持语义完整。比如遇到"第三章"就从这里开始新的一块，90% 场景的最佳起点 |
| **切片粒度（Chunk Size）** | 每块内容的大小，一般 300-600 个 token 起步。太小了 AI 看到的信息太零碎，太大了检索精度下降 |
| **Late Chunking** | Jina 在 2024 年提出的新方法：先让 AI 读完整篇文章生成理解，再按结构切分。好处是每一块都"记得"全文的上下文，长文档检索精度能提升约 9% |
| **上下文检索（Contextual Retrieval）** | Anthropic 提出的方法：给每个切片自动加上一句话总结，说明"这块内容在全文中的位置和角色"，检索错误率降低 35-49% |
| **AST 切片** | 专门给代码用的切法：按函数、按类来切分，而不是按行数。用 TreeSitter 工具能支持 50 多种编程语言 |
| **召回率（Recall@5）** | 检索前 5 个结果里包含正确答案的比例。这是评估切片质量最直接的指标，生产环境要求大于 80% |

## 避坑提醒

- **以为按字数切就够用了**：按字数切（比如每 512 字一块）完全不顾文档结构，合同条款、表格、代码函数都可能被拦腰切断。Demo 阶段看不出问题，上线后 bad case 集中爆发。
- **以为切片粒度凭感觉调就行**：粒度应该靠评测指标反推，先用 300-600 token 的默认值跑一轮评测，看 Recall@5 是否达到 80%，再根据结果决定调大还是调小。
- **以为表格和代码可以跟正文一起切**：表格、代码、公式、对话这些特殊内容必须单独处理。表格混进文本流会被切碎，代码按行切会把一个完整函数拆成几块，AI 都理解不了。
- **以为切片只影响检索速度**：切片直接决定检索质量的上限。正确答案如果没有完整地落在某个切片里，后续的排序和生成再好也救不回来。

## 一句话总结

> **切片质量决定了 AI 检索的天花板，按结构切是起点，按评测指标调是方法。**
