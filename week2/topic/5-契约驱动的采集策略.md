# 05 · 契约驱动的采集策略

> Week 02 · Lesson 05 · Manifest —— 接 pipeline
> 契约驱动的采集：Manifest 是声明，Gate 是放行，Evidence 是追溯

前 4 节课我们立了风险、盘了对象、建了 metadata、写了 contract。今天最后一讲 —— 把所有规则都接进一次真实的 ingest 流水线。

这一讲讲清楚三个最后会进 Week3 工程的对象：**Manifest（这一次到底接什么）、Gate（现在能不能放行）、Evidence（以后能不能追溯）。**

---

## 这节课先解决什么

### 为什么它重要
- 为什么 Source Manifest 不能写成「文件路径列表」。
- 为什么 dry-run 不是「跑了没报错」就算成功。
- 为什么全量 vs 增量的判断要写在 Manifest 里、而不是 Cron 里。
- 为什么 Run Evidence 是 AI 系统未来 13 周一切回溯的根。

### 学完你能做到
- 为一类输入对象写出可被 ingest pipeline 直接消费的 Source Manifest。
- 区分 full / incremental 两类 manifest，并定义合理的增量窗口。
- 把 Contract → Manifest → Gate → Evidence 接成一条可追溯链。
- 在 OmniSupport 仓库里跑通 `seed_loader` dry-run，并产出 run evidence 报告。

---

## 核心论点：声明式 ingest

**Contract 定义「什么算合格」；Manifest 声明「这次接哪一批」；Gate 判断「现在能不能放行」。**

很多团队的 ingest 是「写一个 Python 脚本扫目录」。生产级的 ingest 是「读一份 Manifest，按声明执行」。

> 这两种做法的差别不是写法 —— 是 ingest 出问题之后能不能追溯、能不能重放、能不能审计。声明式 ingest 的核心：每一次入湖都有一份「声明」，事后能精确知道这一次接了什么、从哪儿接、为什么放行。

---

## Source Manifest 真实片段

它声明的是「一次 ingest 行为」：

```json
// data/seed_manifests/ticket_full_v1.json
{
  "manifest_id": "tic.northstar.20260518.full.v1",
  "contract_ref": "tic.northstar_workspace.support_ticket",
  "contract_version": "v3.0.0",
  "ingest_mode": "full",
  "source": {
    "type": "object_store",
    "path": "s3://omnisupport-seed/tickets/2026-05-18/",
    "filter": "*.parquet"
  },
  "incremental_window": null,
  "expected_rows": 12480,
  "expected_pii_fields": ["customer_name", "customer_email"],
  "gate_policy": {
    "on_schema_violation": "reject",
    "on_quality_below_sla": "quarantine",
    "on_pii_unmasked": "reject"
  },
  "trace": {
    "request_id": "ing-req-20260518-001",
    "actor": "scheduled.dagster.ingest_tickets",
    "approval_chain": ["support-platform-bot"]
  }
}
```

> **讲师解读** —— 看清楚 manifest 在声明什么：
> - `contract_ref`：这一批数据必须满足哪份契约。
> - `ingest_mode` + `incremental_window`：全量还是增量，窗口边界。
> - `expected_rows` / `expected_pii_fields`：预期值，便于 dry-run 校验。
> - `gate_policy`：违反契约时的具体动作。
> - `trace`：谁发起、谁批准 —— 后续审计的根。
>
> Manifest 是「这一次接什么」的事先声明，不是文件列表。

---

## Manifest vs 文件列表 —— 看似相似，差一个时代

| 维度 | 文件列表（错） | Source Manifest（对） |
| --- | --- | --- |
| 本质 | "我有什么文件" | "我要接哪一批 + 接的规则" |
| 可追溯性 | 事后只能看日志 | 每个 `manifest_id` 对应一次 ingest 行为 |
| 重放能力 | 难（环境变了找不到原文件） | 强（manifest 不变 → 结果可复现） |
| 契约绑定 | 没有 | `contract_ref` 显式绑定，违约自动拦截 |
| 增量管理 | 靠脚本 / Cron | 声明 `incremental_window`，pipeline 自动算 |
| 审计 | 靠日志 | 声明 `trace.actor` / `approval_chain` |

> 这张对比是 Week2 和 Week3 之间的工程鸿沟 —— 能跨过去，才能谈 Lakehouse、Iceberg、版本化。

## 一次 ingest 全流程 —— 5 个阶段，每个阶段都可拦截

```diagram
╭──────────╮   ╭─────────────╮   ╭──────────────╮   ╭─────────────╮   ╭──────────╮
│ Manifest │──▶│ Schema Check│──▶│ Quality Gate │──▶│ Gate Decide │──▶│ Evidence │
│ 读取声明 │   │  对照       │   │ completeness │   │ accept/warn/│   │ run log  │
│ 校验自身 │   │  contract   │   │ freshness/   │   │ quarantine/ │   │ trace    │
│ Schema   │   │ JSON Schema │   │ uniqueness   │   │ reject      │   │ lineage  │
╰──────────╯   ╰─────────────╯   ╰──────────────╯   ╰─────────────╯   ╰──────────╯
```

> 每个箭头都是一道工程门 —— 前面任何一道失败，pipeline 立即停止并产生 evidence。这就是「声明式 ingest」该有的样子。

---

## 增量窗口 —— Manifest 里最容易写错的一段

Cron 跑全量是 BI 时代的做法。AI 时代 ingest 都是增量为主 —— 而增量的核心难点不是技术（CDC / Kafka 都成熟），是「增量窗口」的语义。一个错误的 `incremental_window` 能让 KPI 周报全错，但你完全发现不了。

### 增量窗口的 3 个真实陷阱

| 陷阱 | 具体表现 | 为什么坑 | 正确做法 |
| --- | --- | --- | --- |
| 用 wall-clock 时间 | `window = [now-1h, now]` | 时区差、机器时钟漂移、数据上报延迟全是雷 | 用 `event_time + watermark`，不用 wall-clock |
| 窗口边界不闭合 | `window = (00:00, 24:00)` | `23:59:59.999` 这条记录可能丢 | 用 `[start, end)` 半开区间 + 显式时区 |
| 没有 grace period | 上游延迟 2 分钟到达就漏掉 | AI 训练集和 KPI 会有微小但稳定的偏差 | 声明 `watermark_lateness`（如 5 分钟） |

> 增量窗口要写进 Manifest，让 pipeline 自动判断 —— 而不是工程师每次手算。

## 一次 dry-run 应该产出什么 —— OmniSupport 实战示例

```bash
$ python -m pipelines.ingestion.seed_loader \
    --manifest data/seed_manifests/ticket_full_v1.json \
    --dry-run

─── DRY RUN REPORT ─────────────────────────────────
Manifest:       tic.northstar.20260518.full.v1
Contract:       tic.northstar_workspace.support_ticket v3.0.0
Mode:           full   |   Files: 47   |   Rows: 12,480
Schema Check:   ✓ all rows valid against contract.schema
Quality Check:  ⚠ freshness 18 min  (SLA: 15 min)
PII Check:      ✓ customer_phone redacted (P3)
                ✓ customer_name / email masked (P2)
Gate Decision:  QUARANTINE
└── 触发原因: quality.freshness_minutes (18 > 15)
└── 处置建议: 通知 support-platform owner; 48h 内 review
Evidence:       ingest_run_id = ing-run-20260518-001
                trace_id = ing-req-20260518-001
                output = .runs/ingest/2026-05-18/001/
─────────────────────────────────────────────────
```

> **讲师解读**：这就是 dry-run 应该输出的形态 —— 不是「跑完没报错」这么简单。它要告诉你：哪一份 manifest、对照哪份 contract、校验过哪些层、最终决策是什么、证据落在哪里。注意 Gate Decision 是 QUARANTINE（隔离）—— 因为 freshness 18 分钟超过了 SLA 的 15 分钟。这种情况不是简单 reject，而是隔离 + 通知 owner —— 保业务连续性，同时不放过问题。

## Run Evidence 是 AI 系统 13 周内一切回溯的根

| Evidence 要包含什么 | 对应字段 | 后续 13 周谁会用它 |
| --- | --- | --- |
| 这一次 ingest 接了什么 | `manifest_ref` / `contract_ref` | Week3 ingest / Week14 治理 |
| 一共接了多少行 | `row_count` / `file_count` / `size` | Week3 完整性验收 |
| 内容指纹 | `content_hash` / `source_fingerprint` | Week4 时间旅行 / Week14 血缘 |
| 决策依据 | `gate_decision` / `violations` | Week11 回归 / Week12 Trace |
| 执行者 | `actor` / `approval_chain` | Week14 合规审计 |
| 输出位置 | `output_path` / `artifact_uri` | Week4 ingest 落地 / Week6 编排 |

> Run Evidence 看起来是日志，其实是 AI 系统的「黑匣子」—— 出问题时它是唯一能拿出来的证据。

---

## 行业的统一方向

| 方向 | 形态 | 与 manifest + evidence 的关系 |
| --- | --- | --- |
| **Iceberg** | Snapshot + Time Travel | 快照机制本质就是「manifest + evidence」—— 每个快照都对应一份 manifest 文件（Week4 展开）。 |
| **OpenLineage** | Run Events | Run Event 模型和 manifest + evidence 同构，便于直接集成（Week14 治理用）。 |
| **lakeFS** | Data as Code | 把 manifest 和 evidence 升级成 Git-like 的 branch / commit（Week14 把版本治理做成原子操作）。 |
| **Dagster** | Software-Defined Assets | 把 ingest 升级成 asset，每个 asset 自带 manifest + evidence（Week6 专门讲）。 |

> Week2 不是孤立的 —— 它是后续 13 周所有工程动作的「输入控制面」。这条链路接好，整门课的工程基线就立住了。
