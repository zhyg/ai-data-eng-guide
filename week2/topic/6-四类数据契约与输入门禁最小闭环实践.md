# 06 · 四类数据契约与输入门禁最小闭环实践

> Week 02 · 实践课 —— 把理论接进代码
> 用一条可运行的控制链，验证前 5 节课立下的所有判断

这一节是第二周的实践课。通过代码的方式，把理论课里没完全吃透的内容再巩固一遍。要先建立一个整体认识：**第二周的产出不是几个孤立的 Python 文件或几份 JSON，而是一条系统性的输入控制面 —— 从资产盘点到契约管理，完整地定义了输入边界。**

## 整体结构：一条从盘点到追溯的控制链

实践工程不是散落的脚本，而是一条首尾相接的控制链。每一层都有明确的输入和产出，最终把「合格的输入」交付给第三周的采集层。

```diagram
╭───────────╮   ╭──────────╮   ╭──────────╮   ╭────────╮   ╭──────────╮
│ Inventory │──▶│ Contract │──▶│ Manifest │──▶│  Gate  │──▶│ Evidence │
│ 资产盘点  │   │ 四类契约 │   │ 声明层   │   │ 放行层 │   │ 证据产物 │
│ 三张输入表│   │ 统一骨架 │   │ 批次声明 │   │ 三态决策│  │ 报告/清单│
╰───────────╯   ╰──────────╯   ╰──────────╯   ╰────────╯   ╰──────────╯
                                                              │
                                                              ▼
                                                   交付给 Week3 采集层
```

> 这条链路控制着「输入的资产、元数据、PII 信息」三件事 —— 它就是后续 13 周所有工程动作的输入控制面。

---

## 第一层 · 资产盘点：三张输入表

资产盘点由三张输入表共同构成，它们一起控制输入的资产、元数据与 PII 信息：

- **资源清单（inventory_v1.csv）**：输入地图。每一行不仅表示一个数据源，还告诉系统它来自哪个产品线的来源系统、来源类型和数据模态，以及数据负责人是谁。
- **最小化元数据**：保证每个资产在运行时能被 RAG、引用、权限、审计、回放稳定消费。
- **PII 矩阵**：把敏感度分级映射到具体动作。

### 资源清单里每一行在声明什么

| 字段类别 | 作用 |
| --- | --- |
| 身份证（asset_id） | 每个数据资源唯一的身份标识 |
| 来源信息 | 来源系统、来源类型、数据模态，说明数据从哪来、是什么 |
| 数据负责人 | 明确出问题先找谁 |
| 访问边界 | 标明数据的访问权限，如公开或受限 |
| pii_level 等级 | 数据的安全等级，决定对外使用时的处理方式 |
| 证据粒度 | 说明记录级别，或「文档 + 申请」等细节 |
| contract reference | 标明这一行绑定了哪份数据契约 |
| 采集方式 | 这批数据该怎么接 —— 工单可能全量，音频可能增量，视频可能需要回填 |

> 资源清单里的 manifest 路径会对应到运行时真正读取的说明文件 —— 它不仅指向数据源，还关联了 PII 与契约信息。

## 最小元数据：数据进入主链的「快递面单」

对 AI 数据工程来说，最小元数据极其重要 —— 它会在后续的 RAG 检索、证据引用、权限过滤、审计、回放中被反复使用。如果缺了 `source_id`、`schema_version`、`ingest_batch_id`、`pii_level` 这些字段，即使把结构化数据存进了向量库，也无法证明它的来源和可用性。

| 字段类别 | 代表字段 | 在干什么 |
| --- | --- | --- |
| 身份链路 | `source_id` / `schema_version` / `ingest_batch_id` | 把资产、manifest、契约、运行清单串联起来 |
| 业务路由 | 产品线 / 责任归属 / 工具路由字段 | 支持产品线过滤、责任归属、工具路由 |
| 合规边界 | `license_tag` / `pii_level` / `quality_gate` | 决定数据能否分发、索引、脱敏或需人工复核 |
| 证据抓手 | `source_fingerprint` / `source_url` | 后续 citation 与 audit 过程中的证据回指 |
| 增量回放 | `created_at` / `updated_at` / `ingest_at` | 增量采集、replay、backfill 的基础 |

> 元数据相当于快递包裹的面单 —— 包含来源证明、状态检查和处理要求，确保数据能放心地进入 AI 数据工程的主链。

---

## PII 矩阵：为 AI 而非 BI 设计

PII 矩阵的设计原理是面向 AI 而不是 BI：它做分级分类，并在特定场景下执行相应动作。

以 ticket 工单为例，`customer_id`、`org_id`、`description` 等字段的安全等级各不相同 —— 在不同的点位上对应不同的动作：检索时默认 action 允许过滤，生成时的 action 则必须做掩码输出。这些规则明确地为 AI 提供了可执行的决策依据。

## 第二层 · 四类资产契约：统一骨架 + 字段扩展

在 `contracts` 路径下放着所有契约定义，覆盖四类资产：结构化票据（ticket）、文档（document）、音频（audio）、视频（video）。它们基于统一的契约骨架，再按不同资产类型做字段扩展。

| 资产类型 | 形态 | 契约重心 | 证据锚点 | 最易翻车点 |
| --- | --- | --- | --- | --- |
| **Ticket** | 结构化（工单 / 对话流） | 状态机枚举、时间语义、角色字段 | `ticket_id` + `created_at` | 状态枚举语义漂移（90% 事故） |
| **Document** | 非结构化（PDF / 手册） | 版本、页码、布局、License | `page_no` / `bbox` / `doc_version` | 回答指向已废弃版本、引用断链 |
| **Audio** | 半结构化（通话 / 转写） | 角色分离、时间戳、置信度、PII 脱敏 | `call_id` + `start_ts`/`end_ts` | PII 进索引追不回、缺降级信号 |
| **Video** | 多模态（录屏 / 操作视频） | 三轨拆分（音轨/OCR/caption）、`frame_ts` | `frame_ts`（同构 `page_no`） | 端到端理解成本高、License 越界 |

每份契约都包含必需字段和约束条件。以 ticket 为例：字段类型可能是 string，并带有详细描述；字段要遵循业务规则（如用正则表达式校验）；枚举值（如 `status`）类型虽是 string，但取值必须落在预定义范围内。

> 若要更新或删除枚举值，往往需要升级版本号、引入一份新的契约文件 —— 这正是 Lesson 4 里 additive / breaking / coordinated 三类变更的工程体现。契约不只含 schema，还含语义信息；视频、音频的脱敏处理等风险，也都属于业务语义的一部分。

## 第三层 · 声明层 Manifest：这次到底接哪一批

要采集的数据批次、以及它绑定的资产和契约，都在声明层里定义。Manifest 包含 `manifest_id` 和 `batch_id`，说明每一次导入接的是什么数据、对照哪份契约、有哪些缺失项和风险，并把这些声明保留下来。

构建 Manifest 时有两件事要分清：**先验证清单模板本身是否正确，再检查清单里的流程是否合理。** 清单模板必须包含 `manifest_id`、`schema_evolution`、`batch_id`、`modality` 等字段 —— 这是组成一个 family 的最小字段集。

> 第二周的核心实验样本（runtime manifest）用的是：结构化工单数据 + 工单契约，以及增量的 CDC 和 PII 扫描数据，被拆成三个接口用于演示。

---

## 第四层 · Gate 放行：三态决策闭环

Gate 层把契约和元数据真正接进运行时，按固定顺序逐道拦截：

```diagram
╭──────────╮  ╭─────────────╮  ╭──────────────╮  ╭───────────╮  ╭──────────────╮
│ 加载     │─▶│ 校验 JSON   │─▶│ 检查契约 +   │─▶│ 应用 PII  │─▶│ 质量评估     │
│ manifest │  │ schema      │  │ 必要元数据   │  │ 策略      │  │ → 三态决策   │
╰──────────╯  ╰─────────────╯  ╰──────────────╯  ╰───────────╯  ╰──────────────╯
```

最终根据结果做出三态决策：**接受（accept）/ 警告（warn）/ 隔离（quarantine）**。这一周只做 dry-run，不直接跑生产采集 —— 加载并校验 manifest，对每个资产做 gate 判断，打印报告并生成证据文件。

> 在生产上线前先做集成测试或单元测试，而不是直接处理线上数据 —— 这是「声明式 ingest」与「写脚本扫目录」的根本区别。

---

## 第五层 · Evidence：可追溯的报告产物

运行结束后会生成评测报告与清单类证据，便于后续评审或治理：

- **run report**：包含 `run_id`、`batch_id`、生成时间、是否 dry-run、manifest 路径等信息。
- **quarantine list**：被隔离的资产及其原因。
- **accepted asset list**：通过放行的资产清单。

报告会把通过、警告、隔离三类资产分别列出，说明每个资产的处理方式与原因，并汇总三类数量、最终状态和推荐行动。以结构化数据形式呈现，便于 AI 评测时直接查看。

> 例如出现 warn 时，报告会给出具体原因 —— 可能是元数据状态字段缺失，或 SHA256 校验和的问题。这些证据是后续 citation、audit、治理的根。

## 测试门禁：把契约变成可回归的代码

回归测试是整个输入门禁的一部分。引入四类契约文件后，按四类契约路径分别编写测试，作为 `pytest` 的运行基础。测试围绕三件事展开：

1. **四类有效记录测试**：确保契约不会误伤正常输入。
2. **四类无效记录测试**：确保所有坏输入都能被拦截。
3. **manifest 正确性测试**：证明质量门禁能实现三态闭环。

> 实战仓库里 `contracts` 下共有 66 个测试用例，覆盖 `records_pass` 等字段验证与各类失败案例。它们不局限于第二周 —— 这正是把契约从「文档」变成「门禁」的关键。

## 实操流程：从起环境到 dry-run

实践环节为了节省本机资源，只启动必要的进程：

1. **复用第一周的本地配置**：先确认配置文件在位、内容正确，使用本地配置而非默认设置。
2. **用 Docker 起后台**：只拉起后台数据库 Postgres 和对象存储 MinIO，在控制台确认两个进程都成功运行。
3. **构建 devbox**：用于运行临时测试脚本（这一步可能需要一些时间）。
4. **跑 Lesson 4 的契约测试**：以单元测试为主，用 `pytest` 验证四类契约。
5. **跑 Lesson 5 的 manifest dry-run**：通过 `seed_loader` 加载数据，对 `seed_manifests` / `manifest_week02` 做清单检测，指定 `batch_id`，并把结果记录为证据，落在 `docs/blueprints/week02` 目录下（如 `week02_dry_run` 报告）。

> 为减少重复输入命令，可以封装一个简化函数，或直接用 docker compose。本周不直接将采集面与第三周内容结合，整体效果要到第三周才能完整看到。

---

## 破坏性测试：验证门禁真的能拦

光看「正常通过」不足以证明门禁有效 —— 还要做反模式测试，故意制造坏输入。例如选一个 `rejects` 类测试方法，把 manifest 里的一个增量字段弹出，重跑后可以看到 manifest 确实拒绝了该字段。

> 这种破坏性测试和 Lesson 1 那个 `pending → in_progress` 事故的回归测试是同一个思路 —— 只有「坏数据能被拦下来」，门禁才算真正成立。

---

## 本课应该带走的判断

- **它是一条链，不是一堆文件**：从资产盘点到契约、声明、放行、证据，是一个完整的输入控制面。
- **元数据是面单**：缺了身份链路与合规边界字段，数据进了向量库也无法证明来源和可用性。
- **PII 矩阵为 AI 而设**：分级 + 场景化动作（检索过滤 / 生成掩码），给 AI 明确的决策依据。
- **Gate 是三态闭环**：accept / warn / quarantine，配合可追溯的 Evidence 才闭合。
- **门禁靠测试落地**：有效记录不误伤、无效记录全拦截、manifest 正确性 —— 破坏性测试是验收标准。
