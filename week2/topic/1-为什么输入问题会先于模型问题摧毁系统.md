# 01 · 为什么输入问题会先于模型问题摧毁系统

> Week 02 · Lesson 01 · Risk —— 立判断
> 输入控制面：为什么 AI 系统先坏在输入

做过 RAG 上线的同学多半有这样的经历 —— 模型没换、Prompt 没动、服务也在跑，但是回答开始慢慢变歪。这一讲不讲数据清洗，而是讲架构师在系统进入生产之前，必须先建立的一个判断：**生产里最危险的不是系统挂掉，而是系统还在跑 —— 并且高自信地错。**

---

## 这节课先解决什么

### 为什么它重要
- 为什么「模型变好了，回答反而变差了」—— 根因常在输入侧。
- 为什么 Demo 阶段从来不出问题的字段，上了生产就开始漂移。
- 为什么「提示词工程」压不住的问题，往往是契约 / 元数据问题。
- 为什么传统的「数据质量监控」不能直接当 AI 的输入控制面。

### 学完你能做到
- 用「事实底线 / 证据底线 / 边界底线」三层结构解释一个 AI 系统的输入风险。
- 在拿到一个真实场景时，能判断「先改模型」还是「先改输入」。
- 能跟产品 / 业务侧把「输入风险」翻译成可执行的工程动作。
- 在 OmniSupport 项目仓库里定位 Week2 的 5 个工程入口（contract / manifest / tests / loader / CI）。

---

## 核心论点

### 失效模式：系统还在跑，并且稳定地输出错误
生产里更常见的失效模式不是「系统挂掉」，而是「系统还在跑，并且稳定地输出错误」。

服务挂了好处理 —— 挂了会报警，会有人上线，会触发应急流程。真正吃苦头的，是接口能返回、RAG 能召回、模型能生成、工具还在执行；但事实变歪、引用断链、边界失守的同时 —— 系统看上去一切正常。

这门课接下来 5 节，做的是同一件事：**把「输入」这个看似最普通的环节，立成 AI 系统的第一道控制面。**

### Demo vs 生产：同样是「答错」，根因完全不同

| 维度 | Demo 阶段 | 生产环境 |
| --- | --- | --- |
| 最先看到的现象 | 输出不够聪明、格式不稳 | 今天答得对，下周开始稳定地答错 |
| 团队的第一反应 | 调 Prompt / 换模型 / 加 reranker | （应该先查）上游 schema / 元数据 / PII 边界 |
| 最常忽略的维度 | 上游 schema / 枚举漂移 | 证据链 / 运行时 gate / 权限边界 |
| 真正先该修的层 | 可以从下游表达层先修 | 永远是输入控制面 |

> Demo 修症状不会翻车；生产修症状会越修越歪 —— 因为根因还在源头继续生产新的错。

## 三条可执行底线

把「数据质量不好」拆成三条可执行的底线：

### 事实底线 · FACT
- 枚举 / 主键 / 时间窗 / 增量窗口。
- 字段没报错 ≠ 字段含义没变。
- 上游一改语义，KPI 和 route 全都漂。
- 对应工程对象：**Schema + Semantics**。

### 证据底线 · EVIDENCE
- `doc_version` / `page_no` / `bbox` / `section_path`。
- 正文还在 ≠ 能回到来源。
- 证据断了，引用 / 审计 / bad case 复盘全断。
- 对应工程对象：**Metadata + Provenance**。

### 边界底线 · BOUNDARY
- PII 分级 / tenant / role / access policy。
- 等到输出再过审 = 已经晚了。
- 检索 / 工具越权，往往源于输入侧没分级。
- 对应工程对象：**PII Policy + Access Boundary**。

## 把风险讲成可复盘的事故

讲架构课最怕只讲概念。下面用两个真实生产里高频发生的事故复盘 —— 一类叫「语义漂移」，一类叫「证据断链」。看完你会理解：为什么 Week2 必须先存在，再谈检索、生成、Agent。

### 事故 A · 语义漂移（SEMANTIC DRIFT）
**上游把 `pending` 改成了 `in_progress`，5 天后 KPI 全错。**

1. **上游改了状态枚举，没人通知下游**：工单系统把 `pending` 改成 `in_progress` 表示「已分配但未开始」—— 这是产品侧的细分需求，但没走任何变更评审。
2. **API 仍然返回，JSON 结构完全没坏**：类型 / 字段 / 格式校验全部通过，监控指标完全正常，0 告警。
3. **旧 Data Contract 没更新，下游照旧消费**：RAG 把 `in_progress` 当成「待处理工单」召回，KPI 把它算成「未闭环」—— 结果对，逻辑全错。
4. **一周后业务发现：响应时长指标「突然变好了」**：因为 `in_progress` 工单被错误地算进了「已开始处理」—— TTFR、AHT 全部失真。
5. **一句话总结：字段没坏，但运行时意义变了**：这就叫语义漂移。Schema 验证回答的是「像不像」，Data Contract 回答的是「意思变没变」。

### 事故 B · 证据断链（PROVENANCE BREAK）
**没有 provenance 的「正确答案」，在生产里通常不算合格答案。**

| 切片携带的字段 | 回答能力 | 引用能力 | 审计能力 |
| --- | --- | --- | --- |
| 只有正文 `text` | 可能还能答 | 几乎为零 | 几乎为零 |
| `text` + `page_no` | 基本可引用 | 较稳 | 较稳 |
| `+ section_path + bbox + doc_version` | 可回指、可比对 | 稳 | 稳 |

> 客户回退一个 bad case 给你的时候，他给的从来不是字符串 —— 是「你昨天那个回答指错页了」。没有 provenance，你连「哪一页」都答不上。

## 行业共识 + 工程入口

这不是个人观点，而是行业方向：Anthropic / OpenAI / Azure / NIST 这四家做的是同一件事 —— 把输入边界、结构边界、来源边界前置成工程对象。

| 来源 | 主张 | 含义 |
| --- | --- | --- |
| **Anthropic** | Simple, composable patterns | 最成功的 Agent 实现不靠复杂黑盒，靠可组合的简单工作流 —— 前提是每一步输入都能被验证。 |
| **OpenAI** | Structured Outputs / JSON Schema | 系统边界是可验证接口，不是文本约定。JSON Schema 从「输出格式」演进成「边界契约」。 |
| **Azure AI Search** | Multimodal retrieval, layout-aware | 顺序、布局、annotation 必须和正文一起进索引 —— 输入对象是带结构的，不是字符串。 |
| **NIST** | Risk management upfront | 生成式 AI 风险管理白皮书强调：风险必须在输入端识别，而不是等系统跑起来再被动响应。 |

> 系统越要可靠，越不能把输入当成「先喂进去再说」的黑盒。

---

## 风险传播：输入问题不是局部 bug

输入问题会同时穿透到 RAG、Tool、Audit 三层：

| 输入侧的问题 | RAG 层怎么坏 | Tool 层怎么坏 | Audit 层怎么坏 |
| --- | --- | --- | --- |
| Schema / 枚举漂移 | 召回 + 解释口径变偏 | 路由依据失真、误调用 | 事后难定位出错批次 |
| Metadata 缺失 | 命中有了但引用不稳 | 工具拿不到必要上下文 | 回退原页 / 原段失败 |
| PII / Access 边界缺失 | 检索越权 | 工具越权 | 合规追责一起失效 |

> Week2 这 5 节课不是治理课，是入口工程化课 —— 根因不修，下游永远在修症状。
