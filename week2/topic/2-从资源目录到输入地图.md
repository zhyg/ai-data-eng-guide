# 02 · 从资源目录到输入地图

> Week 02 · Lesson 02 · Inventory —— 盘对象
> 输入对象建模：从「我们有哪些数据」到「哪些数据值得进入 AI 系统」

很多公司做 AI 之前会发一份 Excel ——「请各业务线提交一下你们的数据资源」。这种做法在 2018 年的 BI 时代没问题，但拿到 AI 时代就会翻车。

这一讲讲清楚：**AI 系统的输入不是「资源列表」，是带身份、带 Owner、带证据、带边界、带消费目标的对象集合。**

---

## 这节课先解决什么

### 为什么它重要
- 为什么「数据资产清单」做完之后，AI 团队还是不敢用。
- 为什么不同业务线交上来的字段名都不一样，谁也对不齐。
- 为什么文档和音视频这类「非结构化」资产，最容易在盘点环节被低估。
- 为什么盘点不是写 Excel，而是写「输入对象 = 类 + 字段 + 责任人 + 消费方」。

### 学完你能做到
- 在拿到一个新业务时，能用「五要素法」快速产出对象级的资产清单。
- 区分四类输入资产（Ticket / Document / Audio / Video）的建模差异。
- 为每一个对象写清「谁消费它 / 它支撑哪些动作 / 它最容易坏在哪」。
- 在 OmniSupport 仓库里读懂四类 source manifest 的字段结构，并能扩展第五类。

---

## 核心论点：Inventory 是输入对象建模层

Asset Inventory 不是资源台账 —— 它是 AI 系统的输入对象建模层。

做台账的人想的是「我们有多少东西」。做 Inventory 的人想的是「这些东西在系统里以什么身份存在、谁负责、被谁消费、出错时谁先发现」。

> 这两个视角差一句话，但落到代码里差一整套架构 —— 前者最后通常变成 Excel；后者最后变成可被 ingest pipeline 直接消费的 manifest。

---

## 盘点环节最常见的三种翻车方式

| 错误做法 | 为什么看起来对 | 上线后翻车在哪 |
| --- | --- | --- |
| 按系统名盘点（CRM / 工单 / 客服 IM） | 工程师习惯，开发好对接 | 同一个系统里的不同对象语义不同，无法精细治理 |
| 按文件类型盘点（PDF / 表格 / 音视频） | 看起来很完备 | 丢掉 owner、生命周期、合规边界等关键属性 |
| 让业务线自己写 Excel 交上来 | 推动快，工作量小 | 字段命名不一致、口径冲突、PII 边界没人定 |

> Inventory 的对象单位是「业务实体 + 工程身份」，不是系统、不是文件、更不是 Excel 行。

---

## 五要素法：一个合格的输入对象必须能回答的五个问题

### 身份 + Owner（IDENTITY）
- `asset_id`：稳定且唯一。
- `asset_type`：tic / doc / audio / video。
- `owner_team`：谁出问题先找谁。
- `lifecycle`：rolling / snapshot / event。

### 证据 + 来源（EVIDENCE）
- `source_fingerprint`：可追溯。
- `source_system`：从哪来。
- `license_tag`：能不能用。
- `last_seen_at`：最近一次更新。

### 边界 + 消费（BOUNDARY）
- `pii_level`：分级，不是布尔。
- `tenant_scope` / `role_scope`：谁能看。
- `consumed_by`：RAG / Tool / KPI / Audit。
- `criticality`：宕了多久业务受影响。

## 四类输入对象的工程建模

OmniSupport 的真实业务里，输入对象最后会被压成四类：Ticket / Document / Audio / Video。重点不是字段表，而是「这一类对象最容易坏在哪、坏了之后下游哪一层会先翻车」。

### Ticket · 结构化资产（工单 / 对话流）
结构化资产看起来最简单 —— 但 Week2 的语义漂移事故，90% 出在它身上。

1. **最危险的字段**：`status` / `priority` / `channel` 这类枚举。上游一改语义，下游 KPI、route、检索口径会一起偏，schema 验证完全发现不了。
2. **容易被忽略的字段**：`created_at` / `closed_at` 的时间语义。是「工单进入系统时间」还是「人工首次响应时间」？不同团队答案完全不同。
3. **AI 视角额外需要**：`comment_thread_id`、`agent_role`、`customer_role`。这些字段决定了 RAG 召回时能不能区分「客户提问」和「客服回答」—— 直接影响生成质量。
4. **Owner 责任边界**：状态机的所有权属于工单系统，不属于 AI 团队。所以 contract 必须能在上游变更时自动报警，而不是事后人工核对。

### Document · 非结构化资产（PDF / 手册 / 帮助中心）
Document 最容易在 Demo 阶段看起来「能跑」，但生产里全靠 metadata 的设计。

1. **正文不是核心 —— 元数据才是**：RAG Demo 用纯文本能跑；生产里 `page_no` / `bbox` / `section_path` / `doc_version` 决定了引用是否可用。
2. **版本是关键属性**：同一篇帮助文档可能有 V1 / V2 / V3 三个版本同时存在，没有 `doc_version` 字段会导致回答指向已废弃版本。
3. **Layout 必须保留**：表格、列表、代码块如果只抽纯文本，在 RAG 阶段语义会断（Week7 还会专门讲）。
4. **License 字段不可省**：内部文档 / 公开手册 / 第三方文档的边界，决定了输出可不可以引用。

### Audio · 半结构化资产（客服通话 / 转写）
音频是最容易被 PII 卡死的资产 —— 也是最容易被「先转文本再说」忽悠掉关键字段的。

1. **转写不是终点，speaker diarization 才是起点**：同一段通话里客户和客服语义完全不同 —— 必须区分 `speaker_role`。
2. **`start_ts` / `end_ts` 是证据链的根**：没有时间戳，bad case 复盘时根本回不到原音频的某一段。
3. **PII 必须在转写阶段处理，不能留到下游**：电话号、地址、银行卡号 —— 一旦进入索引层就追不回了。
4. **`confidence` 字段必须保留**：ASR 的低置信度片段是后续 LLM 误回答的高发区 —— 必须给下游一个降级信号。

### Video · 多模态资产（教学录屏 / 操作视频）
视频是「最贵也最容易做错」的资产 —— 但课程里用最实用的方式建模。

1. **视频拆三轨**：`transcript`（音轨）+ `frame OCR`（视觉文字）+ `key frame caption`（图像描述）。三轨各有 `segment_ts`，能在检索时合并也能单独消费。
2. **不要做端到端视频理解**：在课程项目里成本太高、效果不稳 —— 先做「音轨转写 + 关键帧 OCR/caption」的组合就足够支撑 RAG。
3. **`frame_ts` 是 video 的 `page_no`**：Document 用 `page_no` 回指原页，Video 用 `frame_ts` 回指原画面 —— 证据链同构。
4. **License 比文档更严格**：第三方教学视频的二次分发要逐项核验 —— 课程项目优先用规范化资产，不直接分发原片。

## 真实交付物：`asset_inventory.yaml`

```yaml
# data/inventory/asset_inventory.yaml
assets:
  - asset_id: tic.northstar_workspace.support_ticket
    asset_type: ticket
    owner_team: support-platform
    source_system: northstar.workspace.tickets
    lifecycle: event
    pii_level: P2
    consumed_by: [rag, kpi_mart, tool.ticket_query]
    criticality: P1
    notes: |
      上游 status 枚举由 support-platform 维护，
      Schema 变更必须在 ADR 中声明，且 contract test
      会先于 ingest 失败。
  - asset_id: doc.northstar_workspace.help_center_v3
    asset_type: document
    owner_team: docs
    source_system: northstar.docs.help_center
    lifecycle: rolling
    pii_level: P0
    consumed_by: [rag, evidence_anchor]
    criticality: P2
```

> **讲师解读**：这个文件是 Week2 实战环节的第一个产出。这些字段不是给人看的，是给 ingest pipeline 在 Week3 直接消费的。`asset_inventory.yaml` 改一个字段，Week3 之后的 5 节实验都会受影响。所以盘点这件事的本质是：你今天在 YAML 里写下的每一个字段，都是在向自己和团队做一个工程承诺。

---

## 下游耦合：每个字段背后都对应一个真实的运行时动作

| Inventory 字段 | 哪个下游服务消费它 | 出错时表现为 |
| --- | --- | --- |
| `pii_level` | PII redaction + access boundary check | 客户能查到不该看到的内容 |
| `consumed_by` | ingest scheduling + impact analysis | 变更上线时下游评估缺失 |
| `criticality` | SLO + Runbook + on-call routing | 出问题没人响应、响应不及时 |
| `lifecycle` | incremental window + replay strategy | 增量 / 全量切换错误，数据缺失或重复 |
| `owner_team` | audit / bad-case routing | 出 case 不知道找谁，MTTR 飙升 |

> 盘点字段 ≠ 写文档。每一个字段都是后面 13 周里某一个服务的输入 —— 这就是「Inventory 是输入对象建模」的真实含义。
