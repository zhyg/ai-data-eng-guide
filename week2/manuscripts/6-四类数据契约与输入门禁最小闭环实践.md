我们要采集的数据批次及其绑定的资产和契约在声明层中定义。Manifests 包括 Manifest ID 和 batch ID，说明每次导入的数据、契约、缺失项及风险，并将其保留下来。 Gate 层首先加载 manifest 并验证 JSON schema，接着检查契约和必要的元数据，应用 PII 策略，进行质量评估。根据结果做出接受、警告或隔离的决策。 运行结束后会生成评测报告或财务表，如 run report、quarantine list 和 accepted asset list，这些证据便于后续评审或治理。最终，数据将被传递到第三周的采集层作为输入控制面。
在第一课中，我们提到了一个资产。它实际上是一个输入地图，每一行不仅表示一个数据源，还告诉系统产品线的来源系统、来源类型和数据模态。此外，还包括数据负责人信息，明确了数据的来源、类型及其负责人。 每个数据资源都有唯一的asset_id作为其身份证。访问边界字段显示了数据的访问权限，如公开或受限。pII_level等级则表明了数据的安全等级，决定了数据对外使用的处理方式。证据粒度字段说明了记录级别或文档加申请等细节。contract reference字段标明了数据契约的绑定情况。
这批数据应如何采集，例如通过工单还是全量采集。音频可能是增量的，视频则可能需要回填。Manifest路径对应到运行时的manifest，这是实验运行时读取的说明文件。该文件不仅表明数据源，还关联了PII或合同信息。 对于AI数据工程来说，最小元数据非常重要。它在后续的RAG检索、证据引用、权限过滤、审计和回放中都会被使用。如果缺少source ID、schema version、ingest batch ID、PII level等字段，即使将结构化数据存入向量库，也无法证明其来源和可用性。 身份链路字段如source_id、schema_version和ingest_batch_id，用于将资产、manifest、合同和运行清单串连起来。业务路由字段支持产品线过滤、责任归属和工具路由。合规边界字段如license_tag和pii_level，确保数据使用的合法性和安全性。
quality_gate 决定了数据能否分发、索引、脱敏或需要人工复核，这些都是合规边界的字段。证据抓手包括 source_fingerprint 和 source_url 等字段，这些字段用于后续的 sitation 和 audit 过程中的证据回指。 还有一些字段与增量采集或回放有关，例如 created_at、updated_at 和 ingest_at。这些字段是进行增量采集、replay 或 backfill 的基础。元数据相当于快递包裹的面单，包括来源证明、状态检查和处理要求，确保数据能够放心地进入 AI 数据工程的主链。 设计时的规则原理和 PII 矩阵主要用于 AI 而不是 BI。它包括分级分类，并在特定场景下执行相应操作。例如，ticket 描述可以展示其构成。
