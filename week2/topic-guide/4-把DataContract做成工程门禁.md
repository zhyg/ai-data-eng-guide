# 导学 04：数据合同写完没人用，问题出在哪？

## 一句话理解

数据合同（Data Contract）是 Schema 的生产级超集，它要能在 CI 里自动跑，不合格的数据当场拦下来，写在文档里没人读就等于没有。

## 用生活打比方

继续我们的海关口岸。这一节要把通关标准写成正式协议，再接到闸机上。

协议规定的不只是单据格式，还有货物含义对不对、质量达不达标、有没有按时到、出问题找谁。协议要变成闸机上的自动检查，货不达标当场不放行。闸机有四档：直接放行、放行但告警、隔离等人工复核、整批退回。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **数据合同（Data Contract）** | 比 Schema 管得更宽的一份约定。Schema 只管格式对不对，合同还管含义、质量、时效、兼容性、负责人。能被机器读取和自动校验。 |
| **六层结构** | 一份能扛生产的合同要覆盖六层：结构（Schema）、语义（Semantics）、质量（Quality）、时效（SLA）、兼容性（Compatibility）、负责人（Owner）。 |
| **语义层（Semantics）** | 把字段的业务含义钉死在合同里。比如明确写下"in_progress = 已分配且客服已首次响应"，防止上游悄悄改含义，下游还在按老意思用。 |
| **兼容性三类变更（Additive / Breaking / Coordinated）** | 新增可选字段是向后兼容的，可以直接上线；删字段或改类型是不兼容的，必须协调下游；改业务口径（结构没变但含义变了）最容易被忽视，必须先发评审再同步上线。 |
| **门禁动作（Gate Action）** | 数据违约时的四种处置：accept（正常放行）、warn（放行但告警）、quarantine（隔离问题数据）、reject（整批拒绝入库）。 |
| **合同测试（Contract Test）** | 把合同写成 pytest 测试用例，接进 CI 流水线。任何一条没过，代码合不进去，合同就从文档变成了真正的拦截器。 |

## 避坑提醒

- **以为写份 JSON Schema 就是数据合同**：Schema 只管结构正不正确。Schema 通过了，下游照样可能用到含义错误、质量不达标的数据。合同还要覆盖语义、质量、时效、兼容性、负责人。
- **以为合同写进 Wiki 就算落地**：写进文档没人翻，等于没有。要放进代码仓库（比如 contracts/data/*.json），让 CI 每次自动校验。
- **以为最危险的变更是删字段**：生产里 80% 的事故来自第三类变更，结构没变但业务口径悄悄改了。所有人以为是小改动，其实必须协调上下游同步上线。
- **以为所有变更都得走重协调流程**：过度保守会让团队不敢改任何东西。单纯新增可选字段是向后兼容的，应该能快速上线，只有真正的 breaking 变更才需要重协调。

## 一句话总结

> **数据合同要能在 CI 里自动拦截不合格的数据，过不了测试就上不了线，这才叫门禁；写在文档里没人看，那只是一张告示。**
