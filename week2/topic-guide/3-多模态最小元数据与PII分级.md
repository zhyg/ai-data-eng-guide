# 导学 03：元数据凭什么被四个服务同时依赖？

## 一句话理解

元数据是四个服务每次请求都要读的接口，可填可不填的备注撑不住生产。敏感信息要分等级、配动作，用「是/否」一刀切会出问题。

## 用生活打比方

继续我们的海关口岸。这一节看档案上最关键的那几栏：报关单。

报关单是好几个窗口共用的依据：查验窗口靠它决定开不开箱，征税窗口靠它算税率，安检窗口靠它拦违禁品。少填一栏，对应窗口当场就卡住。报关单上还要标清货物的敏感级别，普通日用品直接放行，管制品走特殊通道，违禁品直接扣押，每一级处理方式都不一样。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **元数据是运行时接口（Metadata as Interface）** | 元数据被检索、引用、审计、权限四个服务每次请求都读，像报关单被多个窗口共用。设计它要像设计一个被调用的接口，不能随手记一笔了事。 |
| **共享核心字段（Shared Core Fields）** | 所有类型数据都必须有的 5 个字段：唯一编号（asset_id）、内容指纹（source_fingerprint）、版本号（source_version）、入库批次号（ingest_batch_id）、许可类型（license_tag）。少一个，下游某个服务就有盲区。 |
| **模态专属字段（Modality-Specific Fields）** | 核心字段之外，每类数据有自己的扩展。比如文档要页码和定位框，音频要说话人角色和时间窗，视频要关键帧时间和字幕外键。 |
| **PII 分级（PII Levels P0–P3）** | 把敏感数据分四级：P0 公开（比如产品文档）、P1 内部（比如工单标题）、P2 敏感（比如客户姓名）、P3 高敏（比如电话、银行卡号）。等级决定后续每个环节的处理方式。 |
| **动作矩阵（Action Matrix）** | 每一级敏感数据在存储、检索、输出三个环节各配一个明确动作，形成一张机器可读的策略表。合规审查看的就是这张表。 |
| **Mask 与 Redact** | 两个容易混淆的动作。Mask 是展示给前端时用星号替换字符，保留长度和位置。Redact 是数据进索引前永久删除整个字段或片段。两者发生在不同位置。 |

## 避坑提醒

- **以为元数据是备注，能填就填**：它是四个服务在线上每次请求都要读的依据。填不全，检索过滤不了、引用回指不了、审计定位不了，某个环节当场就卡住。
- **以为 PII 标个是或否就合规了**：合规审查会问你对敏感数据具体做了什么。回答"标了 is_pii = true"一定被打回，要有每一级在每个环节的具体动作。
- **以为 Mask 和 Redact 效果一样**：Mask 发生在展示层，给授权用户看打码版本；Redact 发生在入库前，数据永久去掉。搞混了，要么敏感数据进了索引，要么授权用户也看不到该看的内容。
- **以为字段越多元数据越完整**：大厂的元数据越做越简单。先定最小集 5 个核心加 5 个模态扩展，剩下的按需加。每个保留的字段都必须有明确的下游使用方。

## 一句话总结

> **元数据要像 API 一样设计，每个字段都有明确的调用方；敏感信息要按等级配动作，一张矩阵表比一个布尔值管用得多。**
