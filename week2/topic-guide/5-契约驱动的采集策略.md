# 导学 05：采集数据，凭什么要先写一份声明？

## 一句话理解

生产级的采集要先写一份声明，说清楚「这次接哪一批、按哪份合同、违约怎么办」，再按声明执行、放行、留证。

## 用生活打比方

继续我们的海关口岸。一批货要入境，靠的是一张报关清单。

清单写明：这批是哪些货、对照哪份标准、预计多少件、不达标时退运还是扣押、谁申报谁批准。闸机按清单逐道检查，做出放行、告警、隔离或退运的决定，整个过程留下通关记录，以后追溯就靠它。

## 关键概念

| 概念 | 大白话解释 |
| :--- | :--- |
| **声明式采集（Declarative Ingest）** | 先写一份声明再按它执行。好处是出问题后能追溯、能重放、能审计。 |
| **采集清单（Manifest）** | 声明「这一次接哪一批」的文件。写明对照哪份合同、全量还是增量、预计多少行、违约动作、谁发起谁批准。它是事先声明，不是文件路径列表。 |
| **放行闸（Gate）** | 按读清单、查结构、查质量、定决策的顺序逐道拦截。前面任何一道失败就停下并留证。 |
| **运行证据（Evidence）** | 一次采集的通关记录：接了什么、多少行、内容指纹、判断依据、谁执行、结果存在哪。系统出事时，它是唯一能拿出来的黑匣子。 |
| **增量窗口（Incremental Window）** | 增量采集时「这次接哪一段时间」的边界，写在清单里让系统自动判断。写错了 KPI 会悄悄偏，还很难发现。 |
| **试跑（dry-run）** | 不真正入库，只把整条流程走一遍并出报告。成功的标准是报告说清了对照哪份合同、查了哪些层、最终决策是什么、证据存在哪。 |

## 避坑提醒

- **以为采集清单就是一份文件列表**：文件列表只说"我有什么文件"，清单说的是"我要接哪一批、按什么规则"，还显式绑定合同，违约自动拦截。两者在追溯、重放、审计能力上差了一个量级。
- **以为增量窗口用机器时间就行**：机器钟会被时区差、时钟漂移、上报延迟坑到。要用事件时间加水位线，用半开区间 `[start, end)`，再留一段迟到容忍时间（比如 5 分钟）。
- **以为试跑没报错就算通过**：试跑的产出是一份完整报告，要写清对照哪份合同、查了哪几层、决策是什么、证据落在哪里。只看"跑完没报错"等于没测。
- **以为全量和增量的切换靠定时任务控制**：全量还是增量、窗口边界多大，这些信息要写在清单里，让流水线自动判断，不能靠工程师每次手动设定时任务来切换。

## 一句话总结

> **采集先声明再执行：清单说清这次接什么，闸机决定能不能放行，证据保证以后能追溯。**
